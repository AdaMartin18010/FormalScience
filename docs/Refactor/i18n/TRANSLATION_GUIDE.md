## 翻译原则

### 1. 准确性原则

- **忠实原文** - 准确传达原文的含义，不增删内容
- **专业准确** - 使用学科领域内的标准术语
- **上下文理解** - 结合上下文理解专业术语的含义

### 2. 可读性原则

- **流畅自然** - 翻译应符合目标语言的表达习惯
- **简洁明了** - 避免冗长复杂的句式
- **逻辑清晰** - 保持原文的逻辑结构和层次

### 3. 一致性原则

- **术语一致** - 同一术语在全文中保持统一译法
- **风格一致** - 保持翻译风格的一致性
- **格式一致** - 保持文档格式和结构的一致性

### 4. 文化适应性原则

- **本地化处理** - 考虑目标语言文化的表达习惯
- **示例适配** - 适当调整示例以符合目标读者
- **避免直译** - 避免生硬的字面翻译

## 术语规范

### 术语来源

1. **优先使用** [terms.json](./terms.json) 中定义的标准术语
2. **参考权威** - 参考学科权威词典和文献
3. **社区共识** - 遵循开源社区的通用译法

### 术语类型

| 类型 | 处理方式 | 示例 |
|------|---------|------|
| 专有名词 | 保留英文或音译 | Kubernetes → Kubernetes |
| 技术术语 | 优先使用标准译法 | Load Balancing → 负载均衡 |
| 数学术语 | 严格遵循学科标准 | Theorem → 定理 |
| 新造词 | 创造新译法并记录 | - |

### 术语格式

- **首次出现** - 中文译法 (英文原词)
  - 示例: 负载均衡 (Load Balancing)
- **后续使用** - 直接使用中文译法
- **代码保留** - 代码中的术语保持不变

## 格式要求

### 文件规范

```
文件名规则:
- 使用小写字母
- 单词间使用连字符 (-)
- 语言代码使用标准格式

示例:
- quick-start.md
- api-reference.md
- architecture-design.md
```

### 文档结构

```markdown
# 标题 (与原文保持一致)

## 简介

## 主要内容

### 子章节

## 总结

---

*译者注: (如有需要)*
*Last updated: YYYY-MM-DD*
```

### Markdown 格式

#### 保留不变的元素

- 代码块及其内容
- 链接 URL
- 图片路径
- HTML 标签
- frontmatter 中的 key 名称

#### 需要翻译的元素

- 标题和正文
- 链接文本
- 图片 alt 文本
- 表格内容
- 列表项

### 代码示例

````markdown

def fibonacci(n):
    if n <= 1:
        return n
    return fibonacci(n-1) + fibonacci(n-2)
```
````

## 审核流程

### 自审检查清单

提交 PR 前，请完成以下自检：

- [ ] 术语使用符合 [terms.json](./terms.json) 规范
- [ ] 文档格式符合要求
- [ ] 代码示例运行正常
- [ ] 链接可以正常访问
- [ ] 无明显的语法错误
- [ ] 风格与项目保持一致

### PR 审核流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  提交 PR    │ → │  自动检查   │ → │  人工审核   │ → │  合并发布   │
│             │    │  (CI/CD)    │    │  (Reviewer) │    │             │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
                           │                  │
                           ▼                  ▼
                    ┌─────────────┐    ┌─────────────┐
                    │  格式检查   │    │  内容审核   │
                    │  链接检查   │    │  术语审核   │
                    └─────────────┘    └─────────────┘
```

### 审核标准

| 检查项 | 权重 | 说明 |
|--------|------|------|
| 准确性 | 40% | 翻译是否准确传达原意 |
| 术语规范 | 25% | 术语使用是否符合规范 |
| 可读性 | 20% | 表达是否流畅自然 |
| 格式规范 | 15% | 格式是否符合要求 |

### 审核反馈

审核意见将分为以下级别：

- **阻塞 (Blocking)** - 必须修改的问题
- **建议 (Suggestion)** - 建议改进但不强制
- **问题 (Question)** - 需要译者确认的问题
- **赞赏 (Praise)** - 翻译质量优秀的部分

## 常见问题

### Q1: 遇到无法确定的术语怎么办？

**A:** 
1. 查阅 [terms.json](./terms.json) 术语库
2. 参考权威词典和文献
3. 在社区讨论区发起讨论
4. 临时保留英文原文，添加译者注

### Q2: 长句如何翻译？

**A:**
- 分析句子结构，理清逻辑关系
- 适当拆分长句，提高可读性
- 保持原文的逻辑层次
- 避免逐字翻译导致的生硬

### Q3: 如何处理代码中的注释？

**A:**
- 代码注释建议翻译为目标语言
- 在注释后保留英文原文（可选）
```python
# 计算平均值 (Calculate average)
mean = sum(values) / len(values)
```

### Q4: 如何更新已翻译的文档？

**A:**
1. 对比原文的更新历史
2. 识别变更内容
3. 同步更新翻译
4. 更新版本号和日期

### Q5: 多人协作时如何避免冲突？

**A:**
1. 在任务看板领取任务
2. 及时同步工作进度
3. 小粒度提交 PR
4. 使用工具检测冲突
