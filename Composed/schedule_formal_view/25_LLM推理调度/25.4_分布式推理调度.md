## 1 概述

### 1.1 核心洞察

大型语言模型（如GPT-4、LLaMA-70B+）的参数量远超单GPU内存容量，必须采用分布式推理。
分布式LLM推理面临独特挑战：需要在保持低延迟的同时，高效协调多GPU间的通信和计算。

### 1.2 分布式推理挑战

| 挑战 | 描述 | 影响 |
|------|------|------|
| **模型切分** | 如何将模型分布到多GPU | 通信开销 |
| **KV-Cache同步** | 分布式KV-Cache管理 | 内存效率 |
| **通信瓶颈** | GPU间数据传输 | 延迟增加 |
| **负载均衡** | 各GPU负载不均 | 资源浪费 |
| **故障恢复** | 节点故障处理 | 可靠性 |

## 3 张量并行

### 3.1 张量并行原理

张量并行（Tensor Parallelism, TP）将单层的计算分布到多个GPU上，适合低延迟场景。

```mermaid
graph TB
    subgraph "单GPU计算"
        A1[输入 X]
        W1[权重 W]
        O1[输出 Y = XW]
        A1 --> W1 --> O1
    end

    subgraph "2-way张量并行"
        A2[输入 X]
        W2a[W₁ on GPU0]
        W2b[W₂ on GPU1]
        O2a[Y₁ = XW₁]
        O2b[Y₂ = XW₂]
        AR[AllReduce]
        O2[Y = Y₁ + Y₂]

        A2 --> W2a --> O2a
        A2 --> W2b --> O2b
        O2a --> AR
        O2b --> AR
        AR --> O2
    end
```

### 3.2 注意力层张量并行

```text
多头注意力张量并行:

原始: Attention = Concat(head₁, ..., head_h) × W_O
      head_i = Softmax(Q_i × K_i^T / √d) × V_i

张量并行 (TP=2):
  GPU0: head₁, head₂, ..., head_{h/2}
  GPU1: head_{h/2+1}, ..., head_h

  计算流程:
  1. [并行] 各GPU计算负责的头
  2. [通信] AllReduce合并输出投影结果
```

```python
class TensorParallelAttention:
    """张量并行注意力层"""

    def __init__(self, hidden_size: int, num_heads: int, tp_size: int, rank: int):
        self.tp_size = tp_size
        self.rank = rank
        self.num_heads_per_rank = num_heads // tp_size

        # 每个rank只持有部分头的参数
        self.q_proj = Linear(hidden_size, hidden_size // tp_size)
        self.k_proj = Linear(hidden_size, hidden_size // tp_size)
        self.v_proj = Linear(hidden_size, hidden_size // tp_size)
        self.o_proj = Linear(hidden_size // tp_size, hidden_size)

    def forward(self, x: Tensor, kv_cache: Optional[KVCache]) -> Tensor:
        # 1. 计算本地Q, K, V
        q = self.q_proj(x)
        k = self.k_proj(x)
        v = self.v_proj(x)

        # 2. 更新KV-Cache
        if kv_cache is not None:
            k, v = kv_cache.update(k, v)

        # 3. 计算注意力（只涉及本地头）
        attn_output = scaled_dot_product_attention(q, k, v)

        # 4. 输出投影
        output = self.o_proj(attn_output)

        # 5. AllReduce合并所有rank的结果
        output = all_reduce(output)

        return output
```

### 3.3 MLP层张量并行

```text
MLP张量并行 (TP=2):

原始: Y = GeLU(X × W₁) × W₂

列并行 + 行并行:
  W₁ 按列切分: W₁ = [W₁ᴬ | W₁ᴮ]
  W₂ 按行切分: W₂ = [W₂ᴬ]
                    [W₂ᴮ]

GPU0: Y₀ = GeLU(X × W₁ᴬ) × W₂ᴬ
GPU1: Y₁ = GeLU(X × W₁ᴮ) × W₂ᴮ

AllReduce: Y = Y₀ + Y₁
```

### 3.4 通信分析

```text
张量并行通信开销:

每层通信量:
  - 注意力层: 2 × batch × seq_len × hidden_size × dtype_size
  - MLP层: 2 × batch × seq_len × hidden_size × dtype_size

总计每层: 4 × B × S × H × sizeof(dtype)

示例 (LLaMA-70B, B=1, S=2048, H=8192, FP16):
  每层通信: 4 × 1 × 2048 × 8192 × 2 = 128MB
  总通信 (80层): 80 × 128MB = 10GB

通信时间 (NVLink 300GB/s):
  T_comm = 10GB / 300GB/s ≈ 33ms
```

## 4 流水线并行

### 4.1 流水线并行原理

流水线并行（Pipeline Parallelism, PP）将模型按层切分，不同stage处理不同的层。

```mermaid
graph LR
    subgraph "Stage 0 (GPU 0)"
        L1[Layers 0-19]
    end

    subgraph "Stage 1 (GPU 1)"
        L2[Layers 20-39]
    end

    subgraph "Stage 2 (GPU 2)"
        L3[Layers 40-59]
    end

    subgraph "Stage 3 (GPU 3)"
        L4[Layers 60-79]
    end

    L1 -->|激活值| L2 -->|激活值| L3 -->|激活值| L4
```

### 4.2 流水线气泡问题

```text
朴素流水线 (4 stages, 4 microbatches):

时间 →
      Stage0: [MB0][MB1][MB2][MB3][   ][   ][   ]
      Stage1: [   ][MB0][MB1][MB2][MB3][   ][   ]
      Stage2: [   ][   ][MB0][MB1][MB2][MB3][   ]
      Stage3: [   ][   ][   ][MB0][MB1][MB2][MB3]
              └─────气泡开销─────┘

气泡比例 = (p-1) / (m+p-1)
  其中 p=stages数, m=microbatch数

示例: p=4, m=4 → 气泡 = 3/7 = 42.9%
```

### 4.3 1F1B调度

```text
1F1B (One Forward One Backward) 调度:

时间 →
      Stage0: [F0][F1][F2][F3][B3][B2][B1][B0]
      Stage1: [  ][F0][F1][F2][B2][B3][B1][B0][B]
      Stage2: [  ][  ][F0][F1][B1][B2][B3][B0][B][B]
      Stage3: [  ][  ][  ][F0][B0][B1][B2][B3][B][B][B]

推理简化版 (仅前向):
      Stage0: [F0][F1][F2][F3]...
      Stage1: [  ][F0][F1][F2][F3]...
      Stage2: [  ][  ][F0][F1][F2][F3]...
      Stage3: [  ][  ][  ][F0][F1][F2][F3]...

稳态后无气泡!
```

### 4.4 流水线并行实现

```python
class PipelineStage:
    """流水线并行Stage"""

    def __init__(self, layers: List[Layer], stage_id: int, num_stages: int):
        self.layers = layers
        self.stage_id = stage_id
        self.num_stages = num_stages
        self.is_first = (stage_id == 0)
        self.is_last = (stage_id == num_stages - 1)

    def forward(self, hidden_states: Optional[Tensor] = None) -> Tensor:
        # 从上一stage接收激活值
        if not self.is_first:
            hidden_states = recv_tensor(src=self.stage_id - 1)

        # 计算本stage的层
        for layer in self.layers:
            hidden_states = layer(hidden_states)

        # 发送到下一stage
        if not self.is_last:
            send_tensor(hidden_states, dst=self.stage_id + 1)
            return None
        else:
            return hidden_states


class PipelineScheduler:
    """流水线调度器"""

    def __init__(self, num_stages: int, num_microbatches: int):
        self.num_stages = num_stages
        self.num_microbatches = num_microbatches

    def generate_schedule(self) -> List[Tuple[int, int]]:
        """生成1F1B调度"""
        schedule = []

        # Warm-up: 填充流水线
        for mb in range(min(self.num_stages, self.num_microbatches)):
            for stage in range(self.num_stages):
                if mb >= stage:
                    schedule.append(('forward', stage, mb - stage))

        # Steady state: 1F1B
        for mb in range(self.num_stages, self.num_microbatches):
            for stage in range(self.num_stages):
                schedule.append(('forward', stage, mb - stage))

        return schedule
```

### 5.2 配置优化

```python
def optimize_parallel_config(
    model_size: int,
    num_layers: int,
    hidden_size: int,
    num_gpus: int,
    gpu_memory: int,
    intra_node_bandwidth: float,  # NVLink
    inter_node_bandwidth: float,  # InfiniBand
) -> Tuple[int, int]:
    """
    优化并行配置
    Returns: (tp_size, pp_size)
    """
    best_config = None
    best_latency = float('inf')

    for tp in range(1, num_gpus + 1):
        if num_gpus % tp != 0:
            continue
        pp = num_gpus // tp

        if num_layers % pp != 0:
            continue

        # 估算内存
        model_memory_per_gpu = model_size / tp / pp
        kv_cache_per_gpu = estimate_kv_cache(hidden_size, num_layers // pp)
        total_memory = model_memory_per_gpu + kv_cache_per_gpu

        if total_memory > gpu_memory:
            continue

        # 估算延迟
        tp_comm = 4 * hidden_size * 2 / intra_node_bandwidth  # AllReduce
        pp_comm = hidden_size * 2 / inter_node_bandwidth  # P2P
        compute_time = estimate_compute(model_size, tp, pp)

        total_latency = compute_time + (num_layers // pp) * tp_comm + (pp - 1) * pp_comm

        if total_latency < best_latency:
            best_latency = total_latency
            best_config = (tp, pp)

    return best_config
```

## 7 形式化模型

### 7.1 分布式推理形式化

```text
定义: 分布式LLM推理系统

结构 DistributedLLM:
  Model = List<Layer>
  Partition: Model → Map<DeviceId, List<Layer>>

并行配置:
  Config = {
    tp_size: Nat,      // 张量并行度
    pp_size: Nat,      // 流水线并行度
    ep_size: Nat,      // 专家并行度
  }

约束:
  tp_size × pp_size × ep_size = num_devices
  ∀ stage ∈ [0, pp_size): |layers(stage)| = num_layers / pp_size
```

### 7.2 延迟模型

```text
端到端推理延迟:

T_total = T_prefill + T_decode × output_length

预填充延迟 (TP=t, PP=p):
  T_prefill = T_compute(prefill) / t + (p-1) × T_bubble + T_comm_tp × num_layers

解码延迟:
  T_decode = T_compute(decode) / t + T_comm_tp × num_layers / p + T_comm_pp

其中:
  T_comm_tp = 2 × H × dtype_size / BW_nvlink  (AllReduce)
  T_comm_pp = B × S × H × dtype_size / BW_ib  (P2P)
  T_bubble = T_compute(decode) × (p-1) / m    (流水线气泡)
```

### 7.3 最优配置定理

```text
定理: 给定硬件拓扑，存在最优并行配置

给定:
  - n 个GPU，分布在 k 个节点
  - 节点内带宽 BW_intra
  - 节点间带宽 BW_inter
  - BW_intra >> BW_inter

最优配置:
  tp_size = gpus_per_node  (利用节点内高带宽)
  pp_size = num_nodes      (最小化节点间通信)

证明思路:
  张量并行通信量大但可用NVLink
  流水线并行通信量小适合跨节点
```

```python
class OverlappedTensorParallel:
    """计算-通信重叠的张量并行"""

    def forward(self, x: Tensor) -> Tensor:
        # 分割输入用于流水
        chunks = x.chunk(num_chunks, dim=0)
        outputs = []

        # 异步通信句柄
        handles = []

        for i, chunk in enumerate(chunks):
            # 计算当前块
            local_out = self.compute(chunk)

            # 异步启动AllReduce
            handle = all_reduce_async(local_out)
            handles.append(handle)

            # 等待上一个块的通信完成
            if i > 0:
                handles[i-1].wait()
                outputs.append(handles[i-1].result())

        # 等待最后一个
        handles[-1].wait()
        outputs.append(handles[-1].result())

        return torch.cat(outputs, dim=0)
```

```python
class DistributedKVCache:
    """分布式KV-Cache管理"""

    def __init__(self, tp_size: int, pp_size: int):
        self.tp_size = tp_size
        self.pp_size = pp_size
        # 每个stage维护自己层的KV-Cache
        self.local_cache = LocalKVCache()

    def update(self, layer_id: int, k: Tensor, v: Tensor):
        """更新KV-Cache"""
        # K,V已经是张量并行切分后的
        self.local_cache.update(layer_id, k, v)

    def migrate(self, request_id: int, target_device: int):
        """跨设备迁移KV-Cache"""
        cache_data = self.local_cache.get(request_id)
        # 异步传输
        send_async(cache_data, target_device)
```

### 8.3 故障恢复

```text

   - 从检查点恢复
```

### 8.4 系统级分布式调度实现

#### 8.4.1 vLLM 分布式调度架构

vLLM的分布式调度采用 **Centralized Scheduler + Distributed Workers** 架构：

```python
class DistributedvLLMScheduler:
    """
    vLLM 分布式调度器
    - 中央调度器运行在Driver进程
    - Workers通过Ray/ZMQ分布到多GPU
    """

    def __init__(self, num_workers, tp_size, pp_size):
        self.num_workers = num_workers
        self.tp_size = tp_size  # 张量并行度
        self.pp_size = pp_size  # 流水线并行度
        self.workers = [Worker.remote(i) for i in range(num_workers)]

        # 调度器只维护元数据，实际KV-Cache由workers持有
        self.block_manager = DistributedBlockManager(
            num_gpu_blocks_per_worker=1024,
            block_size=16
        )

    def schedule(self) -> DistributedSchedulerOutputs:
        # 1. 本地调度决策（与单节点vLLM相同逻辑）
        local_outputs = self.local_scheduler.schedule()

        # 2. 构建分布式执行计划
        # 对于TP：同一stage的workers执行相同输入
        # 对于PP：按stage顺序传递激活值
        execute_model_req = ExecuteModelRequest(
            seq_group_metadata_list=local_outputs.scheduled_seq_groups,
            num_seq_groups=len(local_outputs.scheduled_seq_groups),
            num_batched_tokens=local_outputs.num_batched_tokens,
        )

        # 3. 广播到所有workers（Driver-Worker通信）
        if self.pp_size > 1:
            # 流水线并行：需要调度micro-batches
            return self._schedule_pipeline_parallel(execute_model_req)
        else:
            # 仅张量并行
            ray.get([w.execute_model.remote(execute_model_req)
                     for w in self.workers])
            return local_outputs

    def _schedule_pipeline_parallel(self, req):
        """流水线并行调度：将请求拆分为micro-batches"""
        seq_groups = req.seq_group_metadata_list
        num_micro_batches = min(len(seq_groups), self.pp_size * 2)

        micro_batches = []
        batch_size = max(1, len(seq_groups) // num_micro_batches)
        for i in range(0, len(seq_groups), batch_size):
            micro_batches.append(seq_groups[i:i + batch_size])

        # 使用1F1B调度填充流水线
        schedule = []
        for mb_idx, mb in enumerate(micro_batches):
            for stage in range(self.pp_size):
                worker_id = stage * self.tp_size
                schedule.append((mb_idx, stage, worker_id, mb))

        return schedule
```

**vLLM分布式通信特点**：

- **张量并行（TP）**：每个Worker持有模型的一部分，通过NCCL AllReduce通信。对于80层的LLaMA-70B，每层2次AllReduce，总通信量约 **10GB/请求**
- **流水线并行（PP）**：通过PyTorch Distributed的 `send/recv` 在stage间传递激活值，通信量约 **2 × batch_size × seq_len × hidden_size × sizeof(dtype)**
- **KV-Cache分布**：在张量并行组内，每个worker只存储本地头的KV-Cache，无需额外同步；在流水线并行组内，KV-Cache按层分布到对应stage

#### 8.4.2 SGLang 分布式调度

SGLang的分布式调度器（`SGLang Runtime`）将前缀缓存和结构化生成逻辑扩展到了多节点：

```python
class DistributedSGLangRuntime:
    """
    SGLang 分布式运行时
    特点：
    1. 全局RadixTree管理跨节点前缀共享
    2. 支持TP/DP（数据并行）混合
    """

    def __init__(self, num_nodes, gpus_per_node):
        self.num_nodes = num_nodes
        self.gpus_per_node = gpus_per_node
        # 每个节点维护自己的KV池和本地RadixTree
        self.node_managers = [
            NodeKVManager(node_id=i)
            for i in range(num_nodes)
        ]
        # 全局前缀路由表
        self.prefix_router = GlobalPrefixRouter()

    def dispatch_request(self, request: Request) -> int:
        """
        基于前缀匹配将请求路由到最优节点
        返回: 目标node_id
        """
        input_ids = request.input_ids
        best_node = -1
        best_match_len = 0

        for node_id, manager in enumerate(self.node_managers):
            match_len = manager.radix_tree.match_prefix_length(input_ids)
            if match_len > best_match_len:
                best_match_len = match_len
                best_node = node_id

        # 如果没有匹配或目标节点过载，选择负载最轻的节点
        if best_node < 0 or self.node_managers[best_node].is_overloaded():
            best_node = min(range(self.num_nodes),
                          key=lambda i: self.node_managers[i].get_load())

        return best_node

    def schedule(self):
        """节点级调度后聚合执行"""
        node_batches = [[] for _ in range(self.num_nodes)]

        for req in self.pending_requests:
            node_id = self.dispatch_request(req)
            node_batches[node_id].append(req)

        # 各节点并行执行本地调度
        futures = []
        for node_id, batch in enumerate(node_batches):
            if batch:
                future = self.node_managers[node_id].execute_batch.remote(batch)
                futures.append(future)

        return ray.get(futures)
```

**SGLang分布式前缀共享效果**：
在 multi-turn conversation trace 上，跨节点全局前缀匹配可以将首Token延迟（TTFT）降低 **35-55%**。当系统提示词长度达到 2K tokens 时，缓存命中带来的收益尤为明显。

#### 8.4.3 TensorRT-LLM 分布式调度

TensorRT-LLM的分布式调度完全集成在其C++引擎中，Python层只负责请求队列管理：

```python
class TensorRTLLMDistributedScheduler:
    """
    TensorRT-LLM 分布式调度包装器
    底层调度由C++ Executor完成
    """

    def __init__(self, engine_dir, tp_size, pp_size):
        from tensorrt_llm import Executor
        self.executor = Executor(
            model_path=engine_dir,
            tensor_parallelism=tp_size,
            pipeline_parallelism=pp_size,
            # 启用in-flight batching
            max_batch_size=128,
            max_num_tokens=8192,
            kv_cache_free_gpu_memory_fraction=0.9,
        )

    def enqueue(self, request: Request) -> int:
        """将请求提交给C++调度器"""
        req_id = self.executor.enqueue_request(
            input_tokens=request.input_ids,
            max_tokens=request.max_output_len,
            sampling_config=request.sampling_params,
            # 可选：指定优先级和SLA
            priority=request.priority,
            end_id=request.eos_token_id,
        )
        return req_id

    def await_responses(self, req_ids: List[int]) -> List[Response]:
        """阻塞等待C++引擎返回结果"""
        return self.executor.await_responses(req_ids)

    def get_kv_cache_stats(self) -> KVCacheStats:
        """获取KV Cache使用统计"""
        return self.executor.get_kv_cache_stats()
```

**TensorRT-LLM分布式优化**：

- **通信重叠（Communication Overlap）**：在TP场景下，AllReduce与下一层的部分计算重叠，可将通信开销隐藏 **40-60%**
- **专家并行（MoE）**：支持EP（Expert Parallelism），通过All-to-All在expert间路由token，并自动负载均衡
- **量化感知调度**：FP8/INT4 KV-Cache不仅节省内存，还减少了TP通信量（因为通信数据量减半）

### 8.5 分布式推理性能基准（Llama-3-70B）

在 **2×H100 NVL**（通过NVLink-C2C连接）和 **8×A100 80GB** 两种配置下，Llama-3-70B的分布式推理性能：

| 配置 | 并行策略 | 峰值吞吐 (tok/s) | TTFT (ms) | TBT (ms) | 每token成本 |
|------|---------|-----------------|----------|---------|------------|
| **2×H100 NVL** | TP=2 | 6,200 | 210 | 18 | 基准 |
| | TP=2, PP=1 (IFB) | 8,500 | 195 | 16 | -27% |
| **8×A100 DGX** | TP=8 | 3,800 | 380 | 42 | 基准 |
| | TP=4, PP=2 | 4,200 | 420 | 48 | -10% |
| | TP=2, PP=4 | 4,500 | 510 | 58 | -16% |
| **跨节点 16×A100** | TP=8, PP=2 | 7,800 | 620 | 72 | 基准 |
| | TP=4, PP=4 | 8,400 | 680 | 78 | -7% |

_注：跨节点使用InfiniBand HDR (200Gbps)。TP跨节点时通信开销显著增加，因此大尺度部署通常采用较小的TP group（如TP=4或8），剩余维度使用PP。_
