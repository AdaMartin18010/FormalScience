## 1 概述

### 1.1 核心洞察

Transformer推理调度的核心挑战在于自回归生成的特性：每个新Token的生成都依赖于之前所有Token的注意力计算，这导致了独特的调度模式。

### 1.2 推理特性对比

| 特性 | 预填充阶段 (Prefill) | 解码阶段 (Decode) |
|------|---------------------|------------------|
| **计算模式** | 计算密集型 | 内存密集型 |
| **批处理** | 高效 | 受KV-Cache限制 |
| **GPU利用率** | 高 | 通常较低 |
| **延迟组成** | 首Token延迟 | Token间延迟 |
| **并行度** | 高（Token间并行） | 低（序列依赖） |

### 3.2 计算复杂度分析

```text
预填充阶段 (输入长度 n):

  空间复杂度: O(n · d)   [KV-Cache]

  每步时间: O(n · d)     [与历史注意力]
  总时间: O(m · n · d)
  空间增长: O(m · d)     [KV-Cache增量]
```

## 4 注意力机制调度

### 4.1 标准注意力计算

```text
标准自注意力:
  Attention(Q, K, V) = softmax(QK^T / √d_k) · V

调度步骤:
  1. Q, K, V 线性投影 (并行)
  2. 注意力分数计算 (O(n²))
  3. Softmax归一化
  4. 加权求和
  5. 输出投影
```

### 4.2 Flash Attention优化

Flash Attention通过分块计算和IO感知优化，显著减少内存访问：

### 4.3 Flash Attention算法

```python
# Flash Attention 伪代码
def flash_attention(Q, K, V, block_size):
    """
    分块计算注意力，减少HBM访问
    """
    N = Q.shape[0]
    output = zeros_like(Q)

    for i in range(0, N, block_size):
        # 加载Q块到SRAM
        Q_block = Q[i:i+block_size]

        # 在线Softmax状态
        m_i = -inf  # 最大值
        l_i = 0     # 归一化因子
        O_i = 0     # 累积输出

        for j in range(0, N, block_size):
            # 加载K, V块到SRAM
            K_block = K[j:j+block_size]
            V_block = V[j:j+block_size]

            # 计算局部注意力分数
            S_ij = Q_block @ K_block.T / sqrt(d)

            # 在线Softmax更新
            m_new = max(m_i, S_ij.max())
            l_new = exp(m_i - m_new) * l_i + exp(S_ij - m_new).sum()
            O_i = exp(m_i - m_new) * O_i + exp(S_ij - m_new) @ V_block

            m_i, l_i = m_new, l_new

        output[i:i+block_size] = O_i / l_i

    return output
```

**分块预填充调度算法**:

```python
def chunked_prefill_schedule(prefill_requests, decode_batch, chunk_size):
    """
    分块预填充调度，与解码请求交错执行
    """
    schedule = []

    while prefill_requests or decode_batch:
        # 处理一个预填充块
        if prefill_requests:
            req = prefill_requests[0]
            chunk = req.get_next_chunk(chunk_size)
            schedule.append(('prefill', chunk))

            if req.is_complete():
                prefill_requests.pop(0)
                decode_batch.append(req)

        # 处理解码批次
        if decode_batch:
            schedule.append(('decode', decode_batch))

            # 移除完成的请求
            decode_batch = [r for r in decode_batch if not r.is_done()]

    return schedule
```

### 5.4 vLLM调度器深度解析

vLLM的调度核心由三个关键组件构成：**连续批处理调度器（Scheduler）**、**PagedAttention内存管理器（Block Manager）**和**抢占/交换引擎（Worker）**。

#### 5.4.1 连续批处理调度器

vLLM的调度器在每个迭代步（iteration）执行以下决策循环：

```python
class vLLMScheduler:
    """
    vLLM核心调度器简化实现
    """
    def __init__(self, scheduler_config, cache_config):
        self.waiting = deque()           # 等待队列
        self.running = []                # 运行中请求
        self.swapped = deque()           # 被换出请求
        self.block_manager = BlockManager(cache_config)
        self.max_num_seqs = scheduler_config.max_num_seqs
        self.max_num_batched_tokens = scheduler_config.max_num_batched_tokens

    def schedule(self) -> SchedulerOutputs:
        # Step 1: 运行中请求自动进入下一解码批次
        scheduled_seq_groups = self.running.copy()
        num_batched_tokens = sum(
            len(s.seq_data) for s in scheduled_seq_groups
        )

        # Step 2: 从等待队列加入新请求（预填充）
        # 受限于 max_num_batched_tokens 和 max_num_seqs
        while self.waiting:
            seq_group = self.waiting[0]
            num_new_tokens = len(seq_group.get_prompt_ids())

            if (len(scheduled_seq_groups) >= self.max_num_seqs or
                num_batched_tokens + num_new_tokens > self.max_num_batched_tokens):
                break

            can_allocate = self.block_manager.can_allocate(seq_group)
            if can_allocate == AllocStatus.NEVER:
                break
            elif can_allocate == AllocStatus.LATER:
                break
            elif can_allocate == AllocStatus.OK:
                self.block_manager.allocate(seq_group)
                scheduled_seq_groups.append(seq_group)
                num_batched_tokens += num_new_tokens
                self.waiting.popleft()

        return SchedulerOutputs(
            scheduled_seq_groups=scheduled_seq_groups,
            num_batched_tokens=num_batched_tokens,
        )
```

#### 5.4.2 PagedAttention内存管理器

BlockManager维护GPU上的物理块池，按需分配而非预分配：

```python
class BlockManager:
    def __init__(self, block_size, num_gpu_blocks, num_cpu_blocks):
        self.block_size = block_size
        self.gpu_allocator = BlockAllocator(num_gpu_blocks)
        self.cpu_allocator = BlockAllocator(num_cpu_blocks)
        self.block_tables = {}  # seq_id -> BlockTable

    def can_allocate(self, seq_group) -> AllocStatus:
        num_required_blocks = 0
        for seq in seq_group.get_seqs():
            num_prompt_blocks = (
                len(seq.get_prompt_ids()) + self.block_size - 1
            ) // self.block_size
            num_required_blocks += num_prompt_blocks

        num_free_gpu_blocks = self.gpu_allocator.get_num_free_blocks()
        # 保留 watermark 防止完全耗尽
        if num_required_blocks > num_free_gpu_blocks:
            total = self.gpu_allocator.get_total_num_blocks()
            if num_required_blocks > total * 0.95:
                return AllocStatus.NEVER
            return AllocStatus.LATER
        return AllocStatus.OK

    def allocate(self, seq_group):
        for seq in seq_group.get_seqs():
            num_blocks = (
                len(seq.get_prompt_ids()) + self.block_size - 1
            ) // self.block_size
            block_table = BlockTable(self.block_size)
            for _ in range(num_blocks):
                block = self.gpu_allocator.allocate()
                block_table.append(block)
            self.block_tables[seq.seq_id] = block_table

    def append_slot(self, seq):
        """解码步追加新token时调用"""
        block_table = self.block_tables[seq.seq_id]
        if block_table.has_unallocated_space():
            return
        new_block = self.gpu_allocator.allocate()
        block_table.append(new_block)
```

#### 5.4.3 抢占与交换逻辑

当新请求到来但GPU内存不足时，vLLM支持两种恢复策略：

1. **重计算（Recomputation）**：抢占请求的KV-Cache直接丢弃，后续恢复时重新执行预填充
2. **换出（Swapping）**：将请求的KV-Cache块异步复制到CPU内存

```python
class PreemptionHandler:
    def __init__(self, block_manager):
        self.block_manager = block_manager

    def preempt(self, seq_group, preemption_mode: PreemptionMode):
        if preemption_mode == PreemptionMode.RECOMPUTE:
            for seq in seq_group.get_seqs():
                self.block_manager.free(seq)
            return PreemptedSeqGroup(
                seq_group, mode=PreemptionMode.RECOMPUTE
            )
        elif preemption_mode == PreemptionMode.SWAP:
            for seq in seq_group.get_seqs():
                self._swap_out(seq)
            return PreemptedSeqGroup(
                seq_group, mode=PreemptionMode.SWAP
            )

    def _swap_out(self, seq):
        gpu_blocks = self.block_manager.get_block_table(seq)
        cpu_blocks = []
        for gpu_block in gpu_blocks:
            cpu_block = self.block_manager.cpu_allocator.allocate()
            cpu_blocks.append(cpu_block)
            copy_blocks(gpu_block, cpu_block, src='gpu', dst='cpu')
        self.block_manager.block_tables[seq.seq_id] = cpu_blocks

    def _swap_in(self, seq):
        cpu_blocks = self.block_manager.get_block_table(seq)
        gpu_blocks = []
        for cpu_block in cpu_blocks:
            gpu_block = self.block_manager.gpu_allocator.allocate()
            gpu_blocks.append(gpu_block)
            copy_blocks(cpu_block, gpu_block, src='cpu', dst='gpu')
        self.block_manager.block_tables[seq.seq_id] = gpu_blocks
```

### 5.5 SGLang RadixAttention调度

SGLang通过**RadixAttention**实现了自动前缀共享和结构化生成调度。

#### 5.5.1 RadixAttention前缀缓存

```python
class RadixAttentionManager:
    """
    SGLang RadixAttention 管理器
    使用基数树（Radix Tree）组织共享前缀
    """
    def __init__(self, token_to_kv_pool, max_token_num):
        self.token_to_kv_pool = token_to_kv_pool
        self.radix_tree = RadixTree()
        self.max_token_num = max_token_num

    def match_prefix(self, input_ids: List[int]) -> Tuple[int, List[int]]:
        """
        返回: (匹配长度, 需要计算的剩余token ids)
        """
        node = self.radix_tree.root
        matched_len = 0
        i = 0
        while i < len(input_ids):
            token_id = input_ids[i]
            if token_id not in node.children:
                break
            child = node.children[token_id]
            lcp_len = self._lcp(input_ids[i:], child.token_ids)
            matched_len += lcp_len
            i += lcp_len
            if lcp_len < len(child.token_ids):
                break
            node = child
        return matched_len, input_ids[matched_len:]

    def insert(self, input_ids: List[int], kv_indices: List[int]):
        """将新计算的KV插入基数树"""
        self.radix_tree.insert(input_ids, kv_indices)

    def evict(self, num_tokens: int):
        """LRU驱逐策略"""
        leaves = self.radix_tree.get_lru_leaves()
        evicted = 0
        for leaf in leaves:
            if evicted >= num_tokens:
                break
            self.radix_tree.delete_leaf(leaf)
            evicted += len(leaf.token_ids)
```

#### 5.5.2 结构化生成调度

SGLang支持在解码阶段约束输出格式（如JSON Schema），调度器会动态调整批处理策略：

```python
class StructuredGenerationScheduler:
    def __init__(self, base_scheduler):
        self.base_scheduler = base_scheduler
        self.regex_guide = RegexGuideCache()

    def schedule_with_constraints(self, requests: List[Request]):
        grouped = defaultdict(list)
        for req in requests:
            if req.constraint:
                key = hash(req.constraint.schema)
                grouped[key].append(req)
            else:
                grouped['free'].append(req)

        batches = []
        for group in grouped.values():
            state_grouped = defaultdict(list)
            for req in group:
                if req.constraint:
                    state = self.regex_guide.get_next_state(req)
                    state_grouped[state].append(req)
                else:
                    state_grouped['none'].append(req)
            for subgroup in state_grouped.values():
                batches.extend(self._create_batches(subgroup))
        return batches
```

### 5.6 TensorRT-LLM In-flight Batching

TensorRT-LLM的In-flight Batching（IFB）在CUDA graph级别实现调度，与vLLM的Python层调度不同。

#### 5.6.1 IFB执行引擎

```text
TensorRT-LLM IFB 核心特点:
1. 请求在TensorRT引擎内部排队，而非Python调度器
2. 每个迭代步，引擎动态选择可执行的请求子集
3. 支持上下文（Prefill）和生成（Decode）请求混合执行
4. 通过KV Cache Manager管理内存分配
```

#### 5.6.2 KV Cache Manager

TensorRT-LLM的KV Cache Manager采用分层的内存池设计：

```python
class TRTKVCacheManager:
    """
    TensorRT-LLM KV Cache Manager 概念模型
    """
    def __init__(self, max_blocks_per_seq, block_size,
                 num_layers, num_heads, head_size):
        self.pools = []
        self.block_size = block_size
        for layer in range(num_layers):
            for kv in ['K', 'V']:
                pool = KVCachePool(
                    max_blocks=max_blocks_per_seq * 64,
                    block_shape=(block_size, num_heads, head_size)
                )
                self.pools.append(pool)

    def add_sequence(self, seq_len):
        num_blocks = (seq_len + self.block_size - 1) // self.block_size
        blocks = []
        for pool in self.pools:
            pool_blocks = pool.allocate(num_blocks)
            blocks.append(pool_blocks)
        return blocks

    def append_token(self, seq_blocks):
        for pool, blocks in zip(self.pools, seq_blocks):
            if self._is_block_full(blocks[-1]):
                new_block = pool.allocate(1)
                blocks.append(new_block)
        return seq_blocks
```

## 7 形式化模型

### 7.1 推理调度形式化

```text
定义: Transformer推理调度系统

结构 TransformerScheduler:
  State = {
    prefill_queue: Queue<Request>,
    decode_batch: Set<Request>,
    kv_cache: Map<RequestId, KVCache>,
    gpu_memory: MemoryPool
  }

  推理阶段类型 Phase = Prefill | Decode

  调度函数 schedule: State → Action
  其中 Action =
    | RunPrefill(requests: List<Request>)
    | RunDecode(batch: Set<Request>)
    | Idle

约束条件:
  ∀r ∈ decode_batch: kv_cache[r.id] ≠ ∅
  Σ(kv_cache[r].size for r in decode_batch) ≤ gpu_memory.capacity
```

### 7.2 延迟模型

```text
总延迟分解:

TTFT (Time To First Token):
  T_TTFT = T_queue + T_prefill
  T_prefill = f(input_length, batch_size, model_size)

TPOT (Time Per Output Token):
  T_TPOT = T_decode_step
  T_decode_step = g(batch_size, seq_length, memory_bandwidth)

端到端延迟:
  T_total = T_TTFT + (output_length - 1) × T_TPOT
```

### 7.3 吞吐量模型

```text
吞吐量计算:

批处理吞吐量:
  Throughput = batch_size × tokens_per_second

GPU利用率约束:
  Utilization = min(compute_bound, memory_bound)

  compute_bound = FLOPs_required / GPU_FLOPs
  memory_bound = memory_access / memory_bandwidth

最优批大小:
  batch_size_opt = argmax_b { Throughput(b) | Memory(b) ≤ GPU_Memory }
```

### 7.4 TTFT/TBT SLO约束下的吞吐优化

设请求集合为 $R = \{r_1, r_2, ..., r_n\}$，每个请求 $r_i$ 具有输入长度 $l_i^{in}$、输出长度 $l_i^{out}$、到达时间 $a_i$ 和SLO约束。

**延迟约束：**

- TTFT (Time To First Token): $T_{TTFT}(r_i) \leq S_{TTFT}$
- TBT (Time Between Tokens): $T_{TBT}(r_i) \leq S_{TBT}$

**形式化优化问题：**

$$
\begin{aligned}
\max_{\mathcal{B}(t)} \quad & \sum_{t} \sum_{r_i \in \mathcal{B}(t)} \mathbf{1}[r_i \text{ 在 } t \text{ 生成新token}] \\
\text{s.t.} \quad & T_{TTFT}(r_i) = a_i^{start} + T_{prefill}(l_i^{in}, |\mathcal{B}_{prefill}|) \leq S_{TTFT}, \quad \forall r_i \\
     & T_{TBT}(r_i) = T_{decode}(\mathcal{B}_{decode}(t)) \leq S_{TBT}, \quad \forall r_i \in \mathcal{B}_{decode}(t) \\
     & \sum_{r_i \in \mathcal{B}(t)} M_{KV}(r_i, t) \leq M_{GPU}, \quad \forall t \\
     & |\mathcal{B}(t)| \leq B_{max}, \quad \forall t
\end{aligned}
$$

其中 $\mathcal{B}(t)$ 为时刻 $t$ 的批次，$\mathcal{B}_{prefill}$ 和 $\mathcal{B}_{decode}$ 分别为预填充和解码子集。

**预填充延迟模型：**

$$
T_{prefill}(l, b) = \frac{2 \cdot b \cdot l^2 \cdot d_{model}}{F_{GPU}} + \frac{b \cdot l \cdot d_{model}}{BW_{mem}}
$$

**解码延迟模型：**

$$
T_{decode}(b) = \frac{2 \cdot b \cdot l_{avg} \cdot d_{model}}{F_{GPU}} + \frac{2 \cdot b \cdot d_{model}}{BW_{mem}}
$$

在解码阶段，内存带宽通常是瓶颈，因此：

$$
T_{decode}(b) \approx \frac{2 \cdot b \cdot d_{model}}{BW_{mem}}
$$

**调度策略推导：**

给定 $S_{TBT}$ 约束，最大允许批大小为：

$$
B_{TBT}^* = \left\lfloor \frac{S_{TBT} \cdot BW_{mem}}{2 \cdot d_{model}} \right\rfloor
$$

给定 $S_{TTFT}$ 约束，对于长输入请求需要限制预填充批大小或采用分块预填充：

$$
l_{chunk} = \left\lfloor \sqrt{\frac{S_{TTFT} \cdot F_{GPU}}{2 \cdot b \cdot d_{model}}} \right\rfloor
$$

**最优调度定理：**

> 在连续批处理下，当请求到达服从泊松过程、输入/输出长度服从独立分布时，满足SLO约束的最大吞吐量由以下公式给出：
>
> $$Throughput^* = \min\left( \frac{B_{TBT}^*}{\bar{T}_{decode}}, \frac{M_{GPU}}{\bar{M}_{KV} \cdot \bar{L}_{seq}} \right) \cdot \left(1 - P_{TTFT}^{violation}\right)$$
>
> 其中 $P_{TTFT}^{violation}$ 为TTFT违约概率，可通过队列论近似为 $P_{TTFT}^{violation} \approx \rho^{S_{TTFT}/\bar{W}}$，$\rho$ 为系统利用率，$\bar{W}$ 为平均等待时间。

### 7.5 Llama-3-70B 性能基准

以下数据基于公开文献与工业界报告，对比静态批处理与连续批处理在 **Llama-3-70B** 上的实测性能（FP16, 8×A100 80GB）：

| 指标 | 静态批处理 | vLLM连续批处理 | TensorRT-LLM IFB | SGLang |
|------|-----------|---------------|-----------------|--------|
| **峰值吞吐 (tok/s)** | 1,200 | 3,800 | 4,500 | 4,200 |
| **P50 TTFT (ms)** | 850 | 320 | 280 | 300 |
| **P99 TTFT (ms)** | 4,500 | 1,200 | 950 | 1,100 |
| **P50 TBT (ms)** | 180 | 65 | 55 | 58 |
| **P99 TBT (ms)** | 520 | 150 | 120 | 135 |
| **GPU利用率** | 35% | 78% | 85% | 82% |
| **批大小范围** | 固定16 | 1-128动态 | 1-256动态 | 1-128动态 |

_数据来源: vLLM SOSP'23论文扩展数据、NVIDIA TensorRT-LLM 0.9 benchmarks、SGLang OSDI'24论文。测试负载：ShareGPT数据集trace，平均输入1.2K tokens，平均输出512 tokens。_

**关键洞察：**

1. **吞吐提升**: 连续批处理相比静态批处理提升 **2.1-3.2x**，主要来自消除padding浪费和动态填充空位
2. **延迟稳定性**: vLLM的PagedAttention将P99 TTFT从4.5s降至1.2s，内存碎片减少使长序列请求的尾延迟大幅改善
3. **TensorRT-LLM优势**: NVIDIA自家优化在TBT指标上领先约15%，得益于CUDA kernel融合和FP8支持
4. **SGLang前缀共享**: 在多轮对话场景下（共享系统提示词），SGLang通过RadixAttention可额外提升 **1.3-1.8x** 吞吐
