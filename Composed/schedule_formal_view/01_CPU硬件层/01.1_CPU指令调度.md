## 1 流水线调度

### 1.1 流水线阶段

**经典5级流水线**：

```
IF → ID → EX → MEM → WB
```

| 阶段 | 功能 | 延迟 |
|------|------|------|
| **IF** (取指) | 从指令缓存取指令 | 1周期 |
| **ID** (译码) | 指令译码、寄存器读取 | 1周期 |
| **EX** (执行) | ALU运算、地址计算 | 1-3周期 |
| **MEM** (访存) | 数据缓存访问 | 1-4周期 |
| **WB** (写回) | 结果写回寄存器 | 1周期 |

**现代超标量流水线**（14-19级）：

```
取指 → 预译码 → 译码 → 分派 → 重命名 → 分发 → 执行 → 访存 → 提交
```

### 1.2 流水线调度算法

**静态调度**（编译器完成）：

```c
// 编译器指令调度示例
// 原始代码
ADD R1, R2, R3
SUB R4, R1, R5  // 依赖ADD结果，需要停顿

// 调度后代码
ADD R1, R2, R3
MUL R6, R7, R8  // 插入无关指令
SUB R4, R1, R5  // 消除数据冒险
```

**动态调度**（硬件完成）：

```c
// 硬件动态调度伪代码
typedef struct {
    instruction_t *inst;
    int src1_ready;
    int src2_ready;
    int dst_reg;
    uint64_t issue_cycle;
} reservation_station_entry_t;

// 发射条件
bool can_issue(reservation_station_entry_t *entry) {
    return entry->src1_ready && entry->src2_ready;
}

// 调度循环
void schedule_loop() {
    for (int i = 0; i < RS_SIZE; i++) {
        if (can_issue(&rs[i])) {
            issue_to_execution_unit(&rs[i]);
        }
    }
}
```

### 1.3 流水线冒险处理

**结构冒险**：

```
问题：功能单元冲突
解决：复制功能单元、插入气泡
```

**数据冒险**：

```
RAW (Read After Write): 真依赖，必须等待
WAR (Write After Read): 反依赖，可通过重命名消除
WAW (Write After Write): 输出依赖，可通过重命名消除
```

**控制冒险**：

```
问题：分支指令改变程序流
解决：分支预测、延迟分支、刷新流水线
```

## 2 乱序执行调度

### 2.1 Tomasulo算法

**核心思想**：动态调度指令，消除WAR/WAW冒险

**形式化定义**：

$$
\text{Tomasulo}(I) = \{ (i, t_i) \mid i \in I, t_i = \min\{ t \mid \forall j \prec i, t > t_j \} \}
$$

其中$I$是指令集，$\prec$表示数据依赖关系。

**算法步骤**：

1. **发射（Issue）**：检查空闲保留站，操作数可用则读取，否则标记等待
2. **执行（Execute）**：操作数就绪后开始执行
3. **写结果（Write Result）**：通过CDB广播结果，唤醒等待指令

```c
// Tomasulo算法核心实现
typedef struct {
    int busy;
    int op;
    int vj, vk;      // 操作数值
    int qj, qk;      // 等待的生产保留站
    int dest;        // 目标寄存器
    int addr;        // 内存地址（加载/存储）
} reservation_station_t;

// 发射阶段
bool tomasulo_issue(instruction_t *inst) {
    // 1. 查找空闲保留站
    int rs_id = find_free_rs(inst->type);
    if (rs_id < 0) return false;

    // 2. 读取操作数或标记等待
    for (int i = 0; i < 2; i++) {
        int reg = inst->src[i];
        if (register_status[reg].busy) {
            // 等待其他保留站结果
            rs[rs_id].q[i] = register_status[reg].rs;
        } else {
            // 操作数就绪
            rs[rs_id].v[i] = registers[reg];
            rs[rs_id].q[i] = 0;
        }
    }

    // 3. 标记目标寄存器
    register_status[inst->dst].busy = true;
    register_status[inst->dst].rs = rs_id;

    return true;
}
```

### 2.2 保留站调度

**调度策略**：

| 策略 | 描述 | 复杂度 | 性能 |
|------|------|--------|------|
| **FCFS** | 先到先服务 | O(1) | 一般 |
| **就绪优先** | 优先调度操作数就绪的指令 | O(n) | 较好 |
| **关键路径优先** | 优先调度关键路径上的指令 | O(n log n) | 最优 |

**Intel Skylake保留站配置**：

| 保留站类型 | 条目数 | 功能单元 |
|-----------|--------|----------|
| 整数 | 97 | 4个ALU |
| 浮点 | 72 | 2个FPU |
| 加载 | 56 | 2个加载单元 |
| 存储 | 36 | 1个存储单元 |

### 2.3 重排序缓冲区

**功能**：保证指令按程序序提交，支持精确异常

**结构**：

```c
typedef struct {
    instruction_t *inst;
    int dest_reg;           // 目标寄存器
    int dest_value;         // 结果值
    bool ready;             // 是否就绪
    bool exception;         // 是否有异常
    uint64_t pc;            // 程序计数器
} rob_entry_t;

#define ROB_SIZE 224  // Intel Skylake
```

**提交过程**：

1. 检查ROB头部指令是否完成
2. 更新架构寄存器状态
3. 如果是分支指令，验证预测结果
4. 弹出已提交指令

## 3 分支预测调度

### 3.1 分支预测器类型

**静态预测**：

| 策略 | 准确率 | 适用场景 |
|------|--------|----------|
| 总是不跳转 | 50-60% | 循环结构 |
| 总是跳转 | 40-50% | 错误处理 |
| 后向跳转/前向不跳转 | 65-70% | 循环+顺序 |

**动态预测**：

| 类型 | 准确率 | 硬件成本 |
|------|--------|----------|
| **1位饱和计数器** | 70-80% | 低 |
| **2位饱和计数器** | 85-90% | 中 |
| **两级自适应** | 90-93% | 高 |
| **TAGE** | 95-97% | 很高 |

### 3.2 TAGE预测器

**核心思想**：使用多个基于全局历史的预测表，选择最佳匹配

**结构**：

```
T0: 基础预测表（无历史）
T1: 基于1位历史的预测表
T2: 基于2位历史的预测表
...
Tn: 基于长历史的预测表
```

**预测过程**：

```c
// TAGE预测伪代码
int tage_predict(uint64_t pc, uint64_t *ghist) {
    int pred = base_table[pc % BASE_SIZE];
    int provider = -1;

    for (int i = 0; i < NUM_COMPONENTS; i++) {
        uint64_t tag = compute_tag(pc, ghist, history_lengths[i]);
        uint64_t index = compute_index(pc, ghist, history_lengths[i]);

        if (tage_tables[i][index].tag == tag) {
            if (provider < 0 || i > provider) {
                pred = tage_tables[i][index].counter;
                provider = i;
            }
        }
    }

    return pred;
}
```

**2025年性能**：

| 架构 | 分支预测器 | 准确率 | 误预测惩罚 |
|------|-----------|--------|----------|
| Intel Arrow Lake | TAGE-SC-L | 96.5% | ~15周期 |
| AMD Zen 5 | TAGE+perceptron | 97.2% | ~12周期 |
| Apple M4 | TAGE+loop predictor | 98.1% | ~10周期 |

### 3.3 分支预测对调度的影响

**预测正确**：

- 流水线正常执行
- 指令级并行度最大化

**预测错误**：

- 刷新流水线（Flush）
- 重新从正确路径取指
- 性能损失：15-20周期

**形式化分析**：

$$
\text{IPC}_{\text{actual}} = \text{IPC}_{\text{ideal}} \times (1 - p_{\text{mispredict}} \times \frac{\text{penalty}}{\text{depth}})
$$

其中$p_{\text{mispredict}}$是误预测率，penalty是误预测惩罚周期。

## 4 寄存器重命名调度

### 4.1 重命名原理

**目的**：消除WAR和WAW假依赖，保留RAW真依赖

**实现方式**：

```
架构寄存器（R0-R31）→ 映射表 → 物理寄存器（P0-P223）
```

**重命名过程**：

```c
// 寄存器重命名伪代码
typedef struct {
    int arch_reg;           // 架构寄存器号
    int phys_reg;           // 物理寄存器号
    bool valid;             // 是否有效
} rename_table_entry_t;

int rename_register(int arch_reg) {
    // 1. 分配新的物理寄存器
    int new_phys = allocate_phys_reg();

    // 2. 保存旧映射（用于恢复）
    free_list_push(rename_table[arch_reg].phys_reg);

    // 3. 更新映射表
    rename_table[arch_reg].phys_reg = new_phys;
    rename_table[arch_reg].valid = true;

    return new_phys;
}
```

### 4.2 物理寄存器文件调度

**调度策略**：

| 策略 | 描述 | 延迟 |
|------|------|------|
| **集中式PRF** | 统一物理寄存器文件 | 1-2周期 |
| **分布式PRF** | 按功能单元分布 | 0-1周期 |
| **混合式PRF** | 整数/浮点分离 | 1周期 |

## 5 形式化定义

### 5.1 指令调度问题

**定义1.1（指令调度问题）**：

给定指令集$I = \{i_1, i_2, \ldots, i_n\}$，依赖图$G = (I, E)$，其中$E$表示数据依赖关系，资源约束$R = \{r_1, r_2, \ldots, r_m\}$，调度问题寻找映射$S: I \rightarrow \mathbb{N}$，使得：

1. **依赖约束**：$\forall (i_j, i_k) \in E, S(i_k) \geq S(i_j) + \text{latency}(i_j)$
2. **资源约束**：$\forall t, \forall r \in R, |\{i \in I \mid S(i) \leq t < S(i) + \text{latency}(i), \text{uses}(i, r)\}| \leq \text{capacity}(r)$

**定义1.2（最优调度）**：

调度$S^*$是最优的，当且仅当：

$$
S^* = \arg\min_{S} \max_{i \in I} (S(i) + \text{latency}(i))
$$

### 5.2 调度优化目标

**多目标优化**：

$$
\min_{S} \left( \alpha \cdot \text{makespan}(S) + \beta \cdot \text{resource\_util}(S) + \gamma \cdot \text{power}(S) \right)
$$

其中：

- $\text{makespan}(S)$：调度长度
- $\text{resource\_util}(S)$：资源利用率
- $\text{power}(S)$：功耗
