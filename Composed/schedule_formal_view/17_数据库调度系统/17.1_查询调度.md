## 1 查询调度概述

### 1.1 查询调度的核心挑战

查询调度的核心挑战在于**执行计划优化**和**资源分配**：

- **计划选择**：从指数级计划空间中选择最优计划
- **资源竞争**：CPU、内存、IO资源的竞争
- **并行度**：最大化并行执行效率
- **延迟优化**：最小化查询响应时间

### 1.2 查询执行流程

```text
SQL查询
  ↓ [解析] ~1ms
查询树
  ↓ [语义分析] ~1ms
逻辑查询计划
  ↓ [优化] ~10-100ms
物理执行计划
  ↓ [调度] ~1ms
执行引擎
  ↓ [执行] ~1ms-1s
结果返回
```

**示例：谓词下推**

```sql
-- 原始查询

    SELECT * FROM orders WHERE status = 'completed'
) t JOIN customers c ON t.cust_id = c.id
WHERE c.region = 'Asia';

-- 优化后（谓词下推）
SELECT * FROM orders o
JOIN customers c ON o.cust_id = c.id
WHERE o.status = 'completed' AND c.region = 'Asia';
```

### 2.3 基于成本的优化(CBO)

**成本模型组成**：

```
总成本 = I/O成本 + CPU成本 + 内存成本 + 网络成本

I/O成本 = 顺序读取页数 × seq_page_cost 
        + 随机读取页数 × random_page_cost
        
CPU成本 = 处理元组数 × cpu_tuple_cost
        + 处理操作符开销 × cpu_operator_cost
        
内存成本 = 工作内存使用 × memory_cost_factor

网络成本 = 数据传输量 × network_cost_factor
```

**成本计算示例**：

```
表扫描成本计算:
┌────────────────────────────────────────────────────────────┐
│ 表: orders                                                │
│ - 总行数: 1,000,000                                       │
│ - 页数: 10,000                                            │
│ - seq_page_cost = 1.0                                     │
│ - cpu_tuple_cost = 0.01                                   │
│                                                           │
│ 顺序扫描成本:                                             │
│ Cost = 10,000 × 1.0 + 1,000,000 × 0.01 = 20,000          │
└────────────────────────────────────────────────────────────┘
```

## 3 查询计划生成

### 3.1 计划空间枚举

**连接顺序枚举**：

对于n个表的连接，可能的连接顺序数量：

$$
\text{连接顺序数} = \frac{(2n-2)!}{(n-1)!} = C_{n-1} \times n!
$$

其中 $C_{n-1}$ 是Catalan数。

| 表数量 | 连接顺序数 | 优化时间 |
|--------|-----------|---------|
| 2 | 2 | <1ms |
| 3 | 12 | <1ms |
| 4 | 120 | <10ms |
| 5 | 1,680 | ~50ms |
| 6 | 30,240 | ~500ms |
| 10 | ~6.5×10⁸ | 不可行 |

### 3.2 动态规划搜索

**动态规划算法**：

```python
def dynamic_programming_join_order(tables, statistics):
    """
    使用动态规划寻找最优连接顺序
    """
    n = len(tables)
    
    # dp[S] = (最优子计划, 成本) 对于表集合S
    dp = {}
    
    # 初始化：单表扫描
    for i, table in enumerate(tables):
        S = 1 << i
        plan = create_scan_plan(table, statistics)
        dp[S] = (plan, plan.cost)
    
    # 动态规划：从小到大构建
    for size in range(2, n + 1):
        for S in all_subsets_of_size(n, size):
            best_cost = float('inf')
            best_plan = None
            
            # 尝试所有可能的分割
            for S1 in proper_subsets(S):
                S2 = S - S1
                if S1 not in dp or S2 not in dp:
                    continue
                
                left_plan, left_cost = dp[S1]
                right_plan, right_cost = dp[S2]
                
                # 计算连接成本
                join_cost = estimate_join_cost(left_plan, right_plan, statistics)
                total_cost = left_cost + right_cost + join_cost
                
                if total_cost < best_cost:
                    best_cost = total_cost
                    best_plan = create_join_plan(left_plan, right_plan)
            
            dp[S] = (best_plan, best_cost)
    
    return dp[(1 << n) - 1]
```

### 3.3 启发式剪枝

**剪枝策略**：

```
1. 左深树限制 (Left-Deep Tree)
   - 只考虑左深连接树
   - 减少搜索空间: O(n!) → O(2ⁿ)
   
2. 成本边界剪枝
   - 维护当前最优成本
   - 超过边界的分支直接剪除
   
3. 基数估计剪枝
   - 基于中间结果大小估计
   - 优先探索小中间结果的路径
   
4. 物理属性剪枝
   - 利用排序/哈希属性
   - 避免不必要的排序操作
```

## 4 代价模型

### 4.1 统计信息收集

**统计信息类型**：

```
表级统计:
├── reltuples: 表中行数估计
├── relpages: 表占用页数
└── relallvisible: 全可见页数

列级统计:
├── null_frac: NULL值比例
├── avg_width: 平均列宽
├── n_distinct: 不同值数量
├── correlation: 物理顺序与逻辑顺序相关性
├── most_common_vals: 最常见值列表
├── most_common_freqs: 最常见值频率
├── histogram_bounds: 直方图边界
└── ndistinct_inherited: 继承统计

索引统计:
├── indexrelname: 索引名
├── indrelid: 对应表OID
├── indkey: 索引列
├── indnatts: 索引列数
└── indisunique: 是否唯一索引
```

**直方图示例**：

```
列: age
范围: [0, 100]
直方图桶数: 100

桶边界: [0, 1, 2, 3, ..., 99, 100]
桶频度: [0.02, 0.015, 0.018, ..., 0.005]

选择性估计:
WHERE age BETWEEN 25 AND 35
→ 涉及桶 25-35，共10个桶
→ 估计选择性 ≈ sum(freq[25:35]) ≈ 0.12
```

### 4.2 代价计算公式

**操作符成本公式**：

```
┌─────────────────────────────────────────────────────────────┐
│                    操作符成本模型                            │
├─────────────────────────────────────────────────────────────┤
│ 顺序扫描 (SeqScan)                                          │
│ Cost = pages × seq_page_cost + tuples × cpu_tuple_cost      │
├─────────────────────────────────────────────────────────────┤
│ 索引扫描 (IndexScan)                                        │
│ Cost = index_pages × random_page_cost                       │
│      + table_pages × random_page_cost × selectivity         │
│      + tuples × cpu_tuple_cost × selectivity                │
│      + index_tuples × cpu_index_tuple_cost                  │
├─────────────────────────────────────────────────────────────┤
│ 哈希连接 (HashJoin)                                         │
│ Cost = outer_cost + inner_cost                              │
│      + hash_table_build_cost                                │
│      + probe_cost                                           │
│      = outer_cost + inner_cost                              │
│      + (inner_tuples × cpu_operator_cost)                   │
│      + (outer_tuples × hash_lookup_cost)                    │
├─────────────────────────────────────────────────────────────┤
│ 嵌套循环连接 (NestLoop)                                     │
│ Cost = outer_cost                                           │
│      + (outer_tuples × inner_cost_per_tuple)                │
├─────────────────────────────────────────────────────────────┤
│ 排序 (Sort)                                                 │
│ Cost = input_cost                                           │
│      + (tuples × log₂(tuples) × cpu_operator_cost)          │
│      + (pages × 2 × seq_page_cost)  // 外部排序              │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 选择性估计

**选择性计算公式**：

```
┌─────────────────────────────────────────────────────────────┐
│                    选择性估计公式                            │
├─────────────────────────────────────────────────────────────┤
│ 等值条件: column = value                                    │
│                                                             │
│ 如果 value 在 MCV 列表中:                                    │
│   selectivity = MCV_freq[value]                             │
│ 否则:                                                        │
│   selectivity = (1 - sum(MCV_freqs)) / n_distinct           │
├─────────────────────────────────────────────────────────────┤
│ 范围条件: column > value                                    │
│                                                             │
│ 使用直方图:                                                  │
│   selectivity = (bucket_end - value) / bucket_width         │
│              × bucket_freq                                  │
│              + sum(freq of following buckets)               │
├─────────────────────────────────────────────────────────────┤
│ 连接选择性                                                  │
│                                                             │
│ selectivity = 1 / max(n_distinct(left), n_distinct(right)) │
│                                                             │
│ 对于外键连接:                                                │
│ selectivity = 1 / n_distinct(parent_key)                   │
└─────────────────────────────────────────────────────────────┘
```

## 7 形式化查询优化问题

### 7.1 问题定义

**形式化定义**：

$$
\text{查询优化问题} = (Q, D, P, C, O, R)
$$

其中：

- $Q$：查询集合
  - $Q = \{q_1, q_2, \ldots, q_n\}$
  - 每个查询 $q_i = (sql_i, deadline_i, priority_i)$

- $D$：数据库模式
  - $D = (T, I, S)$
  - $T$：表集合，$I$：索引集合，$S$：统计信息

- $P$：物理计划空间
  - $P(q_i)$：查询 $q_i$ 的所有可能执行计划
  - 每个计划 $p \in P(q_i)$ 包含操作符树

- $C$：约束条件
  - 资源约束：$\sum_{q \in active} resource(q) \leq R_{total}$
  - 时间约束：$latency(q_i) \leq deadline_i$
  - 一致性约束：$isolation \in \{READ\_UNCOMMITTED, \ldots, SERIALIZABLE\}$

- $O$：优化目标
  - 最小化总成本：$\min \sum_i Cost(p_i^*)$
  - 最小化最大延迟：$\min \max_i latency(q_i)$
  - 最大化吞吐量：$\max \frac{n}{T}$

- $R$：资源集合
  - $R = (CPU, Memory, IO, Network)$

### 7.2 复杂度分析

**连接顺序选择复杂度**：

| 表数量 n | 左深树数量 | 浓密树数量 | 优化时间(左深) |
|---------|-----------|-----------|--------------|
| 2 | 2 | 2 | <1ms |
| 3 | 12 | 120 | <1ms |
| 4 | 120 | 1,680 | <10ms |
| 5 | 1,680 | 30,240 | ~50ms |
| 6 | 30,240 | 665,280 | ~500ms |
| 8 | 10,080,000 | 518M | >10s |
| 10 | 6.5×10⁸ | 1.3×10¹¹ | 不可行 |

**复杂度结论**：

- 连接顺序选择是 **NP-hard** 问题
- 对于n个表，左深树搜索空间为 $O(n!)$
- 浓密树搜索空间为 $O(C_{n-1} \times n!)$，其中 $C_{n-1}$ 是Catalan数

### 7.3 定理：查询优化NP完全性

**定理17.1（查询优化NP完全性）**：

> 对于关系数据库中的多表连接查询，在给定代价模型下寻找最优执行计划是NP完全问题。

**证明概要**：

1. **NP成员性**：
   - 给定一个候选计划，可以在多项式时间内计算其成本
   - 因此问题属于NP

2. **NP困难性**（通过3-SAT规约）：
   - 将3-SAT的变量赋值问题映射为连接顺序选择
   - 每个变量对应一个连接决策
   - 可满足性对应于存在低于某阈值的计划成本

∎

**实际启示**：

```
由于NP完全性，实际系统采用启发式方法:

1. 限制搜索空间
   - 左深树限制
   - 连接大小限制 (geqo_threshold in PostgreSQL)
   
2. 遗传算法 (PostgreSQL GEQO)
   - 当表数 > 阈值时使用
   - 非最优但可接受的解
   
3. 贪心算法
   - 每次选择局部最优连接
   - 时间复杂度 O(n²)
```
