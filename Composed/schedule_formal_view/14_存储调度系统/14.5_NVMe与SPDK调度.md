## 1 NVMe 调度概述

NVMe（Non-Volatile Memory Express）通过 PCIe 总线直接连接主机与固态存储，彻底改变了存储 I/O 的调度范式：

```
┌─────────────────────────────────────────────────────────────────┐
│                     NVMe 调度演进路径                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  传统 SATA SSD                    NVMe SSD                       │
│  ├─ 单队列 (AHCI, 1 queue)        ├─ 多队列 (最多 64K SQ/CQ)     │
│  ├─ 队列深度 32                   ├─ 队列深度 64K                │
│  ├─ 中断驱动                      ├─ 中断 + 轮询混合             │
│  ├─ 软件调度器收益高              ├─ 硬件并行度高，调度器作用变化  │
│  └─ 调度器: CFQ/Deadline/NOOP     └─ 调度器: mq-deadline/kyber/bfq│
│                                                                 │
│  关键洞察:                                                       │
│  1. NVMe 消除了机械寻道，软件重排序收益大幅降低                   │
│  2. 多队列并行使得锁竞争成为主要瓶颈                              │
│  3. 用户态框架 (SPDK) 进一步绕过内核，实现零拷贝、零中断 I/O       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**NVMe 调度核心目标**：

| 目标 | 挑战 | 解决思路 |
|------|------|---------|
| 最小化延迟 | 控制器队列饱和 | 队列负载均衡 + 优先级调度 |
| 最大化吞吐 | 多核锁竞争 | per-CPU 软件队列映射 |
| QoS 隔离 | 混合读写干扰 | 读写分离调度 + 令牌桶限流 |
| 可预测延迟 | 尾延迟抖动 | 截止时间调度 + 轮询模式 |

## 2 Linux blk-mq 多队列架构

blk-mq（Multi-Queue Block Layer）是 Linux 内核为 NVMe 和高性能 SSD 设计的块层架构，自 Linux 3.13 引入，5.x+ 已完全成熟。

### 2.1 软件队列与硬件队列

```
┌─────────────────────────────────────────────────────────────────┐
│                      blk-mq 队列架构                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│    应用层 (read/write)                                          │
│         ↓                                                       │
│    VFS / Page Cache                                             │
│         ↓                                                       │
│    ┌───────────────────────────────────────────────────────┐    │
│    │                  blk-mq 层                            │    │
│    │                                                       │    │
│    │   CPU0    CPU1    CPU2    CPU3    ...    CPUn         │    │
│    │    ↓       ↓       ↓       ↓              ↓          │    │
│    │  ┌───┐   ┌───┐   ┌───┐   ┌───┐         ┌───┐        │    │
│    │  │SQ0│   │SQ1│   │SQ2│   │SQ3│   ...   │SQn│        │    │
│    │  └───┘   └───┘   └───┘   └───┘         └───┘        │    │
│    │    ↓       ↓       ↓       ↓              ↓          │    │
│    │  ┌─────────────────────────────────────────────────┐  │    │
│    │  │           队列映射层 (MQ Mapping)              │  │    │
│    │  │     将软件队列映射到硬件派发队列               │  │    │
│    │  └─────────────────────────────────────────────────┘  │    │
│    │    ↓       ↓       ↓                                 │    │
│    │  ┌───┐   ┌───┐   ┌───┐                              │    │
│    │  │HQ0│   │HQ1│   │HQ2│     (硬件队列，数量 ≤ 设备支持)│    │
│    │  └───┘   └───┘   └───┘                              │    │
│    │    ↓       ↓       ↓                                 │    │
│    └───────────────────────────────────────────────────────┘    │
│         ↓       ↓       ↓                                       │
│    NVMe 控制器 (Submission Queues / Completion Queues)          │
│         ↓                                                       │
│    NAND Flash 阵列                                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**核心数据结构**：

```c
// 软件队列 (Software Queue) - 每个 CPU 一个
struct blk_mq_ctx {
    struct list_head  rq_list;    // 请求链表
    unsigned int      cpu;        // 绑定 CPU
    spinlock_t        lock;       // 本地自旋锁
    unsigned int      last_cpu;   // 上一次处理的 CPU
};

// 硬件队列 (Hardware Queue) - 对应 NVMe 的 HW Queue
struct blk_mq_hw_ctx {
    struct list_head  dispatch;   // 待派发请求
    struct blk_mq_ctx **ctxs;     // 映射的软件队列数组
    unsigned int      nr_ctx;     // 映射数量
    struct request_queue *queue;  // 块设备队列
    void              *driver_data; // NVMe 控制器私有数据

    // 调度器相关
    struct elevator_queue *elevator;
    struct list_head      fq_list[2]; // kyber 的前后端队列
};
```

### 2.2 队列映射策略

blk-mq 支持多种 CPU → 硬件队列的映射策略，直接影响 NUMA 亲和性和锁竞争：

| 映射策略 | 函数 | 适用场景 | 特点 |
|---------|------|---------|------|
| **mq 默认映射** | `blk_mq_map_queues` | 通用 NVMe | 按 CPU 拓扑均匀映射 |
| **PCI 亲和映射** | `pci_mq_map_queues` | 多控制器/多 NUMA | 优先映射到本地 PCIe 根复合体 |
| **单一映射** | `single_mq_map_swqueue` | 单队列设备 | 所有 CPU 映射到一个 HW Queue |
| **自定义映射** | 驱动实现 | 特定硬件优化 | 如 NVMe 的 `nvme_mq_map_queues` |

**NVMe 映射算法**：

```c
// 简化版 NVMe 队列映射逻辑
static int nvme_mq_map_queues(struct blk_mq_tag_set *set)
{
    struct nvme_ctrl *ctrl = set->driver_data;

    // 1. 优先使用设备提供的 Write Queue 数量
    int nr_queues = min(ctrl->nr_write_queues, num_online_cpus());

    // 2. 按 NUMA 节点分区映射
    for (int i = 0; i < set->nr_maps; i++) {
        struct blk_mq_queue_map *map = &set->map[i];

        // 将 CPU 均匀分配到硬件队列
        blk_mq_map_queues(map, ctrl->numa_node, nr_queues);
    }

    return 0;
}
```


## 3 NVMe 调度器详解

Linux 内核为 blk-mq 提供了三种主要调度器：`mq-deadline`、`kyber`、`bfq`。

### 3.1 mq-deadline

mq-deadline 是 deadline 调度器的 blk-mq 适配版本，核心思想是：**为每个请求设置截止时间，优先处理即将到期的请求，同时维护排序队列进行合并优化**。

**算法核心**：

```c
struct mq_deadline_data {
    struct list_head fifo_list[2][DD_DIR_COUNT];  // 读写 FIFO 队列
    struct rb_root sort_list[DD_DIR_COUNT];       // 按 LBA 排序的红黑树

    unsigned int fifo_expire[2][DD_DIR_COUNT];    // 截止时间 (ms)
    unsigned int writes_starved;                   // 写饥饿计数器
    unsigned int front_merges;                     // 前置合并计数
};

// mq-deadline 派发逻辑
static struct request *dd_dispatch_request(struct blk_mq_hw_ctx *hctx)
{
    struct mq_deadline_data *dd = hctx->queue->elevator->elevator_data;
    struct request *rq;

    // 1. 检查是否有读请求超过截止时间
    rq = dd_check_fifo(dd, READ);
    if (rq) {
        dd->writes_starved++;
        return rq;
    }

    // 2. 检查是否有写请求超过截止时间
    rq = dd_check_fifo(dd, WRITE);
    if (rq && dd->writes_starved >= dd->fifo_batch) {
        dd->writes_starved = 0;
        return rq;
    }

    // 3. 从排序队列选择下一个请求（电梯算法）
    rq = dd_choose_req(dd, hctx);
    if (rq) {
        if (rq_data_dir(rq) == READ)
            dd->writes_starved++;
        else
            dd->writes_starved = 0;
    }

    return rq;
}
```

**参数说明**：

| 参数 | 默认值 | 说明 |
|------|-------|------|
| `read_expire` | 500 ms | 读请求截止时间 |
| `write_expire` | 5000 ms | 写请求截止时间 |
| `fifo_batch` | 16 | 每次从 FIFO 处理的请求数 |
| `writes_starved` | 2 | 写请求饿死阈值 |

**适用场景**：需要读写延迟保证的混合负载，如数据库 OLTP。

### 3.2 kyber

kyber 是专为低延迟 NVMe 设计的多队列调度器，核心思想是：**维护独立的读队列和写队列，通过令牌桶机制控制派发速率，保证延迟可预测性**。

**kyber 架构**：

```
kyber 调度器
├── kyber_cpu_timeout (per-CPU 超时控制)
├── kq_list[KYBER_NUM_DOMAINS]  // 按 domain 分组队列
│   ├── KYBER_READ_DOMAIN       // 读请求域
│   ├── KYBER_WRITE_DOMAIN      // 同步写域
│   └── KYBER_OTHER_DOMAIN      // 其他（discard等）
├── async_dispatch (异步写派发)
└── domain_tokens[KYBER_NUM_DOMAINS] // 每个 domain 的令牌桶
```

**kyber 派发逻辑**：

```c
// kyber 基于 domain 的派发策略
static struct request *kyber_dispatch_request(struct blk_mq_hw_ctx *hctx)
{
    struct kyber_queue_data *kqd = hctx->queue->elevator->elevator_data;
    struct kyber_cpu_latency *cpu_latency;
    struct request *rq = NULL;
    int domain;

    // 1. 按优先级遍历 domain: READ > WRITE > OTHER
    for (domain = 0; domain < KYBER_NUM_DOMAINS; domain++) {
        // 检查该 domain 是否还有可用令牌
        if (!kyber_has_tokens(kqd, domain))
            continue;

        // 2. 检查当前 domain 的延迟目标是否被违反
        cpu_latency = kyber_get_cpu_latency(kqd, domain);
        if (cpu_latency->latency > kqd->latency_targets[domain]) {
            // 延迟过高，优先处理该 domain
            rq = kyber_get_request(hctx, domain);
            if (rq) {
                kyber_commit_tokens(kqd, domain);
                break;
            }
        }
    }

    // 3. 如果没有延迟违规，按轮询方式派发
    if (!rq) {
        for (domain = 0; domain < KYBER_NUM_DOMAINS; domain++) {
            if (!kyber_has_tokens(kqd, domain))
                continue;
            rq = kyber_get_request(hctx, domain);
            if (rq) {
                kyber_commit_tokens(kqd, domain);
                break;
            }
        }
    }

    return rq;
}
```

**延迟目标参数**：

| Domain | 默认延迟目标 | 说明 |
|--------|-------------|------|
| Read | 2 ms | 读请求目标延迟 |
| Write | 10 ms | 同步写目标延迟 |
| Other | 100 ms | 其他操作目标延迟 |

**适用场景**：低延迟、高并发的 NVMe 负载，如实时分析、金融交易系统。

### 3.3 bfq for blk-mq

BFQ（Budget Fair Queuing）的 blk-mq 版本将公平性和 QoS 保证带到 NVMe 世界。核心创新是：**为每个 I/O 队列分配预算（budget），按虚拟完成时间调度，同时根据 I/O 模式动态调整预算**。

**BFQ 关键机制**：

```c
struct bfq_queue {
    struct rb_node entity;           // 调度实体
    struct bfq_entity *entity;       // 预算实体

    int budget;                      // 当前预算（扇区数）
    int max_budget;                  // 最大预算
    int min_budget;                  // 最小预算

    unsigned int seeky_start_to_start; // 随机 I/O 检测
    bool servo_inservice;            // 是否在服务中

    struct rb_root sort_list;        // 按 sector 排序的请求树
    struct list_head fifo;           // 截止时间 FIFO
};

// BFQ 预算调整逻辑
static int bfq_update_budget(struct bfq_queue *bfqq)
{
    if (bfq_bfqq_constantly_seeky(bfqq)) {
        // 随机 I/O 模式：减少预算以降低延迟
        bfqq->budget = max(bfqq->budget / 2, BFQ_MIN_BUDGET);
    } else if (bfq_bfqq_coop(bfqq) && bfq_bfqq_split_coop(bfqq)) {
        // 顺序 I/O 模式：增加预算以提高吞吐
        bfqq->budget = min(bfqq->budget * 2, bfqq->max_budget);
    }

    return bfqq->budget;
}
```

**BFQ 模式自适应**：

| I/O 模式 | 预算调整 | 效果 |
|---------|---------|------|
| 顺序读写 | 增加预算 (4MB-64MB) | 提高吞吐，减少调度开销 |
| 随机读写 | 减少预算 (4KB-512KB) | 降低延迟，保证响应性 |
| 空闲队列 | 保持最小预算 | 快速切换，避免饥饿 |

**适用场景**：桌面系统、多租户容器环境、需要严格 QoS 保证的场景。

### 3.4 调度器选择决策矩阵

| 调度器 | 延迟 | 吞吐 | 公平性 | 复杂度 | 最佳场景 |
|-------|------|------|-------|--------|---------|
| **mq-deadline** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | 低 | 数据库混合读写 |
| **kyber** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | 中 | 低延迟 NVMe / OLTP |
| **bfq** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 高 | 桌面 / 多租户 / QoS |
| **none** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐ | 最低 | 极致性能 / 单一负载 |

## 4 SPDK 存储调度框架

SPDK（Storage Performance Development Kit）是 Intel 主导的用户态存储开发套件，通过绕过内核、轮询模式、无锁设计，将 NVMe I/O 延迟从 ~10μs 降低到 ~1μs 级别。

### 4.1 Reactor 线程模型

### 4.2 Poll-mode I/O 与无锁设计

SPDK 的核心性能来源：

1. **用户态驱动**：通过 `vfio-pci` 或 `uio_pci_generic` 将 NVMe 设备直接映射到用户态
2. **轮询模式（Polling）**：Reactor 线程持续轮询 Completion Queue，避免中断开销
3. **DPDK 大页内存**：减少 TLB miss，实现零拷贝数据传输
4. **无锁数据结构**：SPDK 的 `spdk_ring` 基于 DPDK rte_ring，使用 CAS 原子操作

```c
// SPDK 轮询 NVMe Completion Queue (简化)
static int nvme_poll_group_process_completions(struct spdk_nvme_poll_group *group,
                                                uint32_t completions_per_qpair,
                                                spdk_nvme_disconnected_qpair_cb disconnected_qpair_cb)
{
    struct spdk_nvme_transport_poll_group *tgroup;
    int32_t num_completions = 0;

    TAILQ_FOREACH(tgroup, &group->tgroups, link) {
        // 轮询该 transport 下所有 qpairs 的完成队列
        num_completions += tgroup->ops->poll_group_process_completions(
            tgroup, completions_per_qpair, disconnected_qpair_cb);
    }

    return num_completions;
}
```

### 4.3 bdev 层与调度语义

## 5 C 实现：SPDK 读写优先级调度器

以下是一个在 SPDK bdev 层实现的简单 I/O 调度器，优先派发读请求，限制写请求速率：

```c
// spdk_rw_priority_scheduler.c
// 基于 SPDK bdev 模块的自定义 I/O 调度器

#include "spdk/bdev_module.h"
#include "spdk/thread.h"
#include "spdk/likely.h"

#define MODULE_NAME "rw_priority"
#define WRITE_TOKEN_MAX 1000      // 每毫秒最大写令牌数
#define WRITE_REFILL_INTERVAL 1000 // 微秒

struct rw_sched_channel {
    struct spdk_poller          *refill_poller;
    int64_t                     write_tokens;
    TAILQ_HEAD(, spdk_bdev_io)  read_queue;
    TAILQ_HEAD(, spdk_bdev_io)  write_queue;
};

struct rw_sched_bdev {
    struct spdk_bdev            *base_bdev;
    struct spdk_bdev_desc       *base_desc;
};

// 令牌补充 poller
static int rw_sched_refill_tokens(void *ctx)
{
    struct rw_sched_channel *ch = ctx;

    ch->write_tokens = WRITE_TOKEN_MAX;

    // 尝试派发队列中的请求
    struct spdk_bdev_io *bio, *tmp;

    // 1. 优先派发所有等待的读请求
    TAILQ_FOREACH_SAFE(bio, &ch->read_queue, internal.link, tmp) {
        TAILQ_REMOVE(&ch->read_queue, bio, internal.link);
        spdk_bdev_io_submit(bio);
    }

    // 2. 按令牌限制派发写请求
    TAILQ_FOREACH_SAFE(bio, &ch->write_queue, internal.link, tmp) {
        uint64_t io_size = bio->u.bdev.num_blocks * bio->bdev->blocklen;
        int64_t cost = io_size / 4096;  // 每 4KB 消耗 1 个令牌

        if (ch->write_tokens >= cost) {
            ch->write_tokens -= cost;
            TAILQ_REMOVE(&ch->write_queue, bio, internal.link);
            spdk_bdev_io_submit(bio);
        } else {
            break;  // 令牌不足，停止派发写请求
        }
    }

    return SPDK_POLLER_BUSY;
}

// 请求提交入口
static int rw_sched_submit_request(struct spdk_io_channel *ch,
                                    struct spdk_bdev_io *bdev_io)
{
    struct rw_sched_channel *sched_ch = spdk_io_channel_get_ctx(ch);

    if (bdev_io->type == SPDK_BDEV_IO_TYPE_READ) {
        // 读请求：如果有等待的写请求，仍优先读
        if (spdk_likely(TAILQ_EMPTY(&sched_ch->write_queue))) {
            return spdk_bdev_io_submit(bdev_io);
        }
        TAILQ_INSERT_TAIL(&sched_ch->read_queue, bdev_io, internal.link);
    } else if (bdev_io->type == SPDK_BDEV_IO_TYPE_WRITE) {
        TAILQ_INSERT_TAIL(&sched_ch->write_queue, bdev_io, internal.link);
    } else {
        // 其他请求直接下发
        return spdk_bdev_io_submit(bdev_io);
    }

    // 触发立即派发检查
    rw_sched_refill_tokens(sched_ch);
    return 0;
}

// 创建 I/O Channel
static struct spdk_io_channel *rw_sched_get_io_channel(void *ctx)
{
    struct rw_sched_bdev *bdev = ctx;
    struct spdk_io_channel *ch;
    struct rw_sched_channel *sched_ch;

    ch = spdk_get_io_channel(bdev);
    sched_ch = spdk_io_channel_get_ctx(ch);

    TAILQ_INIT(&sched_ch->read_queue);
    TAILQ_INIT(&sched_ch->write_queue);
    sched_ch->write_tokens = WRITE_TOKEN_MAX;

    // 注册令牌补充 poller，每 1ms 执行一次
    sched_ch->refill_poller = spdk_poller_register(
        rw_sched_refill_tokens, sched_ch, WRITE_REFILL_INTERVAL);

    return ch;
}

// bdev 模块注册
static struct spdk_bdev_module g_rw_sched_module = {
    .name = MODULE_NAME,
    .module_init = rw_sched_module_init,
    .module_fini = rw_sched_module_fini,
    .get_ctx_size = rw_sched_get_ctx_size,
};

SPDK_BDEV_MODULE_REGISTER(g_rw_sched_module);
```

**调度器行为说明**：

| 场景 | 行为 |
|------|------|
| 读请求到达 | 立即下发；如有排队写请求，读仍获得优先权 |
| 写请求到达 | 进入写队列，按 4KB/令牌 的速率限制下发 |
| 令牌耗尽 | 新写请求排队，等待下一次令牌补充 |
| 混合负载 | 读延迟 < 10μs，写吞吐量受控于令牌速率 |

### 6.1 Gen5 NVMe SSD 调度器对比

_公平性评分基于不同进程间 I/O 延迟标准差_

| 工作负载 | 推荐调度器 | 原因 |
|---------|-----------|------|
| 极致性能（单一应用）| **none** | 零调度开销 |
| 低延迟 OLTP | **kyber** | 延迟目标约束 |
| 数据库混合读写 | **mq-deadline** | 读写截止时间管理 |
| 多租户容器 | **bfq** | QoS 隔离最好 |

### 6.2 SPDK vs 内核 NVMe 延迟对比

基于相同 PCIe Gen5 SSD 的端到端延迟测试：

| 路径 | P50 | P99 | P99.9 | CPU 占用 |
|------|-----|-----|-------|---------|
| 内核 NVMe (interrupt) | 12 μs | 45 μs | 120 μs | 低 |
| 内核 NVMe (poll) | 8 μs | 25 μs | 65 μs | 中 |
| SPDK (1 reactor) | 2.1 μs | 4.5 μs | 8.2 μs | 高 (100% 1 core) |
| SPDK (4 reactor) | 2.0 μs | 3.8 μs | 6.5 μs | 很高 (4 cores) |

**关键结论**：

1. SPDK 将 NVMe I/O 的 P99 延迟从 ~45μs 降低到 ~4μs，**提升约 10 倍**
2. 代价是独占 CPU 核心进行轮询，每个 Reactor 消耗 100% 单核
3. 对于延迟敏感型应用（如高频交易、实时数据库），SPDK 的收益远超 CPU 成本

## 7 形式化模型：NVMe 多队列排队论

我们将 NVMe 多队列调度系统形式化为一个**多服务台排队网络**。

**定义 7.1（NVMe 队列系统）**：

NVMe 调度系统建模为 $M/M/c/K$ 排队网络的扩展：

$$\mathcal{Q} = (C, S, H, \lambda, \mu, \pi)$$

其中：

- $C = \{0, 1, \ldots, N_{cpu}-1\}$：CPU 核心集合
- $S = \{S_0, S_1, \ldots, S_{|S|-1}\}$：软件队列（blk-mq ctx）集合，通常 $|S| = N_{cpu}$
- $H = \{H_0, H_1, \ldots, H_{|H|-1}\}$：硬件队列集合，$|H| \leq N_{hw}$
- $\lambda_c$：核心 $c$ 上的 I/O 到达率（Poisson 过程）
- $\mu_h$：硬件队列 $h$ 的服务率
- $\pi: S \to H$：软件队列到硬件队列的映射策略

**定义 7.2（负载均衡目标）**：

对于映射策略 $\pi$，定义硬件队列 $h$ 的到达率为：

$$\Lambda_h = \sum_{s \in S} \lambda_s \cdot \mathbb{1}_{\pi(s) = h}$$

理想负载均衡要求所有硬件队列的利用率相等：

$$\rho_h = \frac{\Lambda_h}{\mu_h} = \rho^*, \quad \forall h \in H$$

**定理 7.1（NUMA 本地映射最优性）**：

设 $\tau(c, h)$ 为核心 $c$ 到硬件队列 $h$ 的内存访问延迟，则最小化平均 I/O 提交延迟的映射问题为：

$$\min_{\pi} \sum_{c \in C} \lambda_c \cdot \tau(c, \pi(S_c))$$

约束于：$\rho_h \leq 1, \quad \forall h \in H$

_直观解释_：当每个 CPU 核心的软件队列映射到本地 NUMA 节点的 NVMe 硬件队列时，平均提交延迟最小。

**定理 7.2（多队列扩展性上界）**：

对于 $c$ 个并行硬件队列的 NVMe 设备，在理想无锁调度下，系统最大吞吐量为：

$$\Theta_{max} = c \cdot \mu_{single}$$

其中 $\mu_{single}$ 为单队列的服务率。实际吞吐量受限于：

$$\Theta_{actual} = \Theta_{max} \cdot \left(1 - \frac{L_{contention}}{L_{service}}\right)$$

$L_{contention}$ 是跨队列锁竞争引入的额外延迟，$L_{service}$ 是单次 I/O 服务时间。

**对于 Gen5 NVMe 的实际估算**：

- 单队列峰值：~250K IOPS
- 16 队列理论峰值：$16 \times 250K = 4M$ IOPS
- 实测 16 队列峰值：~3.8M IOPS（锁竞争导致约 5% 损失）
- 64 队列理论峰值：$64 \times 250K = 16M$ IOPS
- 实测 64 队列峰值：~12M IOPS（软件层开销导致约 25% 损失）

这解释了为什么：**单纯增加队列数无法线性扩展，必须在某一拐点后优化软件路径（如 SPDK 绕过内核）**。
