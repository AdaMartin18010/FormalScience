### 1.2 IO调度的核心挑战

**调度目标矩阵**：

| 目标 | 描述 | 权衡因素 |
|------|------|---------|
| **最小化延迟** | 减少单个IO响应时间 | 可能降低吞吐量 |
| **最大化吞吐量** | 提高单位时间数据量 | 可能增加延迟 |
| **公平性** | 多进程IO公平分配 | 可能降低整体效率 |
| **实时性** | 保证截止时间 | 实现复杂度高 |

### 1.3 调度延迟分解

```

  ↓ [未命中]

  ↓ [块映射] ~1μs

  ↓ [队列调度] ~5-10μs

  └─ SSD: 控制器 50μs + 闪存 50-100μs
```

## 2 IO调度器详解

### 2.1 CFQ调度器

**CFQ（Completely Fair Queuing）**：完全公平队列调度器

**核心算法**：

```python
class CFQScheduler:
    """
    CFQ调度器实现

    核心思想:
    1. 每个进程维护独立的IO队列
    2. 使用时间片轮询调度
    3. 保证每个进程的IO带宽公平
    """

    def __init__(self):
        self.cfq_queues = {}  # pid -> CFQQueue
        self.service_tree = RBTree()  # 按虚拟时间排序
        self.active_queue = None
        self.slice_end_time = 0

    def cfq_enqueue(self, request, pid):
        """将请求加入对应进程的CFQ队列"""
        if pid not in self.cfq_queues:
            self.cfq_queues[pid] = CFQQueue(pid)

        queue = self.cfq_queues[pid]
        queue.add_request(request)

        # 插入服务树
        if queue not in self.service_tree:
            self.service_tree.insert(queue.virtual_time, queue)

    def cfq_dispatch(self):
        """选择下一个要处理的请求"""
        # 1. 检查当前队列是否还有时间片
        if self.active_queue and time.now() < self.slice_end_time:
            if self.active_queue.has_requests():
                return self.active_queue.next_request()

        # 2. 时间片用完，选择下一个队列
        if self.active_queue:
            # 更新虚拟时间
            self.active_queue.virtual_time += self.slice_used

        # 3. 选择虚拟时间最小的队列
        self.active_queue = self.service_tree.find_min()
        if self.active_queue:
            self.service_tree.remove(self.active_queue)
            self.slice_end_time = time.now() + SLICE_DURATION
            return self.active_queue.next_request()

        return None

    def cfq_completed(self, request, pid):
        """请求完成后的处理"""
        queue = self.cfq_queues[pid]
        queue.slice_used += request.sectors
```

**CFQ队列结构**：

```
CFQ调度器
├── 同步队列 (每个进程一个)
│   ├── 按sector排序的红黑树
│   └── FIFO队列（相同sector的请求）
├── 异步队列 (全局)
│   └── 按时间排序
└── 空闲队列
    └── 等待新请求
```

**CFQ参数**：

| 参数 | 默认值 | 说明 |
|------|-------|------|
| quantum | 8 | 每次分发请求数 |
| fifo_expire_sync | 125ms | 同步请求过期时间 |
| fifo_expire_async | 250ms | 异步请求过期时间 |
| slice_sync | 100ms | 同步队列时间片 |
| slice_async | 40ms | 异步队列时间片 |

### 2.2 Deadline调度器

**Deadline调度器**：截止时间调度器

**核心算法**：

```python
class DeadlineScheduler:
    """
    Deadline调度器实现

    核心思想:
    1. 维护读队列和写队列
    2. 为每个请求设置截止时间
    3. 优先处理接近截止时间的请求
    4. 同时维护排序队列进行合并优化
    """

    def __init__(self):
        # 按LBA排序的队列 (用于合并)
        self.sorted_read_queue = RBTree()   # 读排序队列
        self.sorted_write_queue = RBTree()  # 写排序队列

        # 按截止时间排序的队列
        self.read_fifo = deque()  # 读FIFO队列
        self.write_fifo = deque() # 写FIFO队列

        self.read_expire = 500   # 读请求500ms过期
        self.write_expire = 5000 # 写请求5s过期
        self.writes_starved = 0  # 写请求饥饿计数
        self.fifo_batch = 16     # 每次处理FIFO请求数

    def deadline_enqueue(self, request):
        """将请求加入队列"""
        request.deadline = time.now() + \
            (self.read_expire if request.is_read else self.write_expire)

        # 加入排序队列 (按LBA)
        if request.is_read:
            self.sorted_read_queue.insert(request.lba, request)
            self.read_fifo.append(request)
        else:
            self.sorted_write_queue.insert(request.lba, request)
            self.write_fifo.append(request)

    def deadline_dispatch(self):
        """选择下一个请求"""
        # 1. 检查是否有请求超过截止时间
        expired_read = self.check_expired(self.read_fifo)
        expired_write = self.check_expired(self.write_fifo)

        # 2. 优先处理过期读请求
        if expired_read:
            self.writes_starved += 1
            return expired_read

        # 3. 优先处理过期写请求 (如果读没有过度饥饿写)
        if expired_write and self.writes_starved >= 0:
            self.writes_starved = 0
            return expired_write

        # 4. 从排序队列选择 (电梯算法)
        next_read = self.sorted_read_queue.next_request()
        next_write = self.sorted_write_queue.next_request()

        # 5. 读写交替，避免写饿死
        if next_read and (not next_write or self.writes_starved > -2):
            self.writes_starved += 1
            return next_read
        elif next_write:
            self.writes_starved -= 1
            return next_write

        return next_read

    def check_expired(self, fifo_queue):
        """检查队列中是否有过期请求"""
        if fifo_queue and fifo_queue[0].deadline < time.now():
            return fifo_queue.popleft()
        return None
```

**Deadline调度特点**：

| 特性 | 说明 |
|------|------|
| 读优先 | 读请求截止时间更短 (500ms vs 5s) |
| 写聚合 | 写请求批量处理，提升吞吐量 |
| 饿死防护 | 计数器保证写请求不会饿死 |
| 请求合并 | 排序队列支持高效合并 |

### 2.3 NOOP调度器

**NOOP（No Operation）**：无操作调度器

**核心思想**：

```python
class NOOPScheduler:
    """
    NOOP调度器 - 最简单的调度器

    核心思想:
    1. 简单的FIFO队列
    2. 仅做基本的请求合并
    3. 适用于SSD/NVMe等无寻道延迟设备
    """

    def __init__(self):
        self.queue = deque()  # FIFO队列
        self.next_lba = 0     # 下一个期望的LBA (用于合并)

    def noop_enqueue(self, request):
        """将请求加入队列"""
        # 尝试与前一个请求合并
        if self.queue:
            last = self.queue[-1]
            if last.lba + last.sectors == request.lba:
                # 相邻请求，可以合并
                last.sectors += request.sectors
                last.length += request.length
                return

        self.queue.append(request)

    def noop_dispatch(self):
        """取出下一个请求"""
        if self.queue:
            return self.queue.popleft()
        return None
```

**NOOP适用场景**：

| 场景 | 原因 |
|------|------|
| SSD | 无寻道延迟，不需要重排序 |
| NVMe | 多队列并行，软件调度无益 |
| 虚拟化 | 调度留给宿主机处理 |
| 存储阵列 | 阵列控制器已优化 |

### 2.4 BFQ调度器

**BFQ（Budget Fair Queuing）**：预算公平队列调度器

**核心算法**：

```python
class BFQScheduler:
    """
    BFQ调度器实现

    核心思想:
    1. 为每个队列分配预算 (budget)，单位为扇区数
    2. 队列消耗预算进行IO
    3. 预算耗尽后切换到下一个队列
    4. 根据IO模式动态调整预算
    """

    def __init__(self):
        self.queues = {}  # pid -> BFQQueue
        self.active_tree = RBTree()  # 活跃队列
        self.idle_tree = RBTree()    # 空闲队列
        self.in_service_queue = None
        self.default_budget = 16 * 1024  # 8192扇区 = 4MB

    def bfq_enqueue(self, request, pid):
        """将请求加入队列"""
        if pid not in self.queues:
            self.queues[pid] = BFQQueue(pid, self.default_budget)

        queue = self.queues[pid]
        queue.add_request(request)

        # 如果是新活跃队列，加入活跃树
        if queue not in self.active_tree and queue not in self.idle_tree:
            self.active_tree.insert(queue.virtual_finish_time, queue)

    def bfq_dispatch(self):
        """选择下一个请求"""
        # 1. 检查当前队列预算
        if self.in_service_queue:
            if self.in_service_queue.budget_exhausted():
                # 预算耗尽，保存状态并切换
                self.update_queue_budget(self.in_service_queue)
                self.in_service_queue = None
            else:
                # 还有预算，继续服务
                return self.in_service_queue.next_request()

        # 2. 选择虚拟完成时间最小的队列
        if not self.in_service_queue and self.active_tree:
            self.in_service_queue = self.active_tree.extract_min()
            self.in_service_queue.budget = self.calculate_budget(
                self.in_service_queue
            )
            return self.in_service_queue.next_request()

        return None

    def calculate_budget(self, queue):
        """动态计算队列预算"""
        # 根据历史IO模式调整预算
        if queue.is_sequential():
            # 顺序IO，增加预算以提高吞吐
            return min(queue.budget * 2, 64 * 1024)
        else:
            # 随机IO，减少预算以保证延迟
            return max(queue.budget // 2, 4 * 1024)

    def update_queue_budget(self, queue):
        """更新队列预算和虚拟时间"""
        # 根据实际使用量调整
        used = queue.budget - queue.remaining_budget
        queue.virtual_finish_time += used / queue.weight

        if queue.has_requests():
            self.active_tree.insert(queue.virtual_finish_time, queue)
        else:
            self.idle_tree.insert(queue.virtual_finish_time, queue)
```

**BFQ优势**：

| 场景 | BFQ优化 | 效果 |
|------|--------|------|
| 启动应用 | 识别启动IO模式 | 启动时间减少30-50% |
| 视频播放 | 保证带宽 | 无卡顿 |
| 多任务 | 公平分配 | 响应性提升 |
| 大文件复制 | 后台低优先级 | 不阻塞前台 |

## 3 磁盘物理优化

### 3.1 寻道优化算法

**电梯算法（SCAN）**：

```python
def scan_schedule(requests, current_position, direction='up'):
    """
    SCAN电梯算法

    磁头单向移动，服务沿途请求
    到达端点后反向
    """
    sorted_requests = sorted(requests, key=lambda r: r.lba)

    # 分离当前位置左右两侧的请求
    left = [r for r in sorted_requests if r.lba < current_position]
    right = [r for r in sorted_requests if r.lba >= current_position]

    if direction == 'up':
        # 先向右，再向左
        schedule = right + left[::-1]
    else:
        # 先向左，再向右
        schedule = left[::-1] + right

    return schedule
```

**C-SCAN（循环扫描）**：

```python
def cscan_schedule(requests, current_position):
    """
    C-SCAN循环扫描算法

    磁头单向移动，到达端点后直接返回起点
    更公平的等待时间
    """
    sorted_requests = sorted(requests, key=lambda r: r.lba)

    right = [r for r in sorted_requests if r.lba >= current_position]
    left = [r for r in sorted_requests if r.lba < current_position]

    # 只向右扫描，到达端点后返回起点继续向右
    return right + left
```

**LOOK算法**（实际实现）：

```python
def look_schedule(requests, current_position, direction='up'):
    """
    LOOK算法 - 实际调度器使用的算法

    与SCAN类似，但只到达最远请求位置就反向
    不需要到达物理端点
    """
    if not requests:
        return []

    sorted_requests = sorted(requests, key=lambda r: r.lba)
    min_lba = sorted_requests[0].lba
    max_lba = sorted_requests[-1].lba

    left = [r for r in sorted_requests if r.lba < current_position]
    right = [r for r in sorted_requests if r.lba >= current_position]

    if direction == 'up':
        if right:  # 有右侧请求，继续向右
            return right + left[::-1]
        else:  # 无右侧请求，直接向左
            return left[::-1]
    else:
        if left:
            return left[::-1] + right
        else:
            return right
```

**FIFO (First In First Out) / FCFS 算法详解**：

```python
def fifo_schedule(requests):
    """
    FIFO先到先服务调度算法

    特点:
    - 按照请求到达顺序处理
    - 实现简单，无需重排序
    - 公平性最好，无饥饿问题

    缺点:
    - 寻道时间可能很长
    - 无寻道优化
    """
    queue = deque(requests)
    schedule = []

    while queue:
        request = queue.popleft()
        schedule.append(request)

    return schedule
```

**SSTF (Shortest Seek Time First) 算法详解**：

```python
def sstf_schedule(requests, current_position):
    """
    SSTF最短寻道时间优先算法

    特点:
    - 优先处理距离当前磁头位置最近的请求
    - 最小化每次寻道时间
    - 可能导致远端请求饥饿

    数学分析:
    对于n个请求，平均寻道距离为:
    E[d] = (max_lba - min_lba) / (n + 1)

    时间复杂度:
    - 朴素实现: O(n²)
    - 使用优先队列: O(n log n)
    """
    pending = list(requests)
    schedule = []
    current = current_position

    while pending:
        # 找到距离最近的请求
        closest = min(pending, key=lambda r: abs(r.lba - current))
        schedule.append(closest)
        pending.remove(closest)
        current = closest.lba

    return schedule
```

**电梯算法（SCAN/C-SCAN/LOOK）数学分析**：

```
磁盘参数:
- C: 磁道数 ( cylinders)
- N: 请求数
- v: 磁头移动速度 (磁道/毫秒)
- t_seek: 单位距离寻道时间

SCAN算法分析:
设磁头从0到C-1扫描，请求位置为 x₁, x₂, ..., xₙ

平均寻道长度 = (2C) / 3  (均匀分布时)
总寻道时间 = Σ|xᵢ - xᵢ₋₁| / v

LOOK算法优化:
只需扫描到最远请求位置 x_max:
平均寻道长度 = (2 × x_max) / 3

C-SCAN公平性分析:
最大等待时间 = 2C / v
相比SCAN减少了双向扫描的不公平性
```

**寻道优化效果对比矩阵**：

| 算法 | 平均寻道 | 最大等待 | 公平性 | 时间复杂度 | 饥饿风险 |
|------|---------|---------|--------|-----------|---------|
| **FIFO** | 最高 | 低 | ⭐⭐⭐⭐⭐ | O(1) | 无 |
| **SSTF** | 低 | 高 | ⭐⭐ | O(n log n) | 高 |
| **SCAN** | 中 | 中 | ⭐⭐⭐⭐ | O(n log n) | 无 |
| **C-SCAN** | 中 | 低 | ⭐⭐⭐⭐⭐ | O(n log n) | 无 |
| **LOOK** | 低 | 中 | ⭐⭐⭐⭐ | O(n log n) | 无 |
| **C-LOOK** | 低 | 低 | ⭐⭐⭐⭐⭐ | O(n log n) | 无 |

**算法选择决策表**：

| 工作负载特征 | 推荐算法 | 原因 |
|-------------|---------|------|
| 请求均匀分布 | C-SCAN | 公平性+效率平衡 |
| 请求集中在局部 | SSTF | 最小化寻道 |
| 实时性要求高 | LOOK | 快速响应 |
| 简单实现需求 | FIFO | 无状态维护 |

### 3.2 旋转延迟优化

**旋转位置感知（RPO）**：

```python
def rpo_schedule(requests, current_position, current_rotation, rpm=7200):
    """
    旋转位置感知调度

    同时考虑寻道时间和旋转延迟
    选择总时间最短的请求
    """
    sector_time = 60 / (rpm * sectors_per_track)  # 每个扇区旋转时间

    best_request = None
    best_time = float('inf')

    for req in requests:
        # 计算寻道时间
        seek_distance = abs(req.cylinder - current_position.cylinder)
        seek_time = seek_distance * seek_rate

        # 计算旋转等待时间
        arrival_rotation = (current_rotation + seek_time * rpm / 60) % sectors_per_track
        rotation_wait = (req.sector - arrival_rotation) % sectors_per_track
        rotation_time = rotation_wait * sector_time

        total_time = seek_time + rotation_time

        if total_time < best_time:
            best_time = total_time
            best_request = req

    return best_request
```

### 3.3 请求合并与排序

**请求合并算法**：

```python
def merge_requests(queue, new_request):
    """
    请求合并 - 相邻请求合并为一个大请求

    前置合并: [已存在请求] + [新请求]
    后置合并: [新请求] + [已存在请求]
    双向合并: [前置] + [新请求] + [后置]
    """
    merged = False

    # 尝试前置合并
    for req in queue:
        if req.lba + req.sectors == new_request.lba:
            req.sectors += new_request.sectors
            req.length += new_request.length
            merged = True
            break
        elif new_request.lba + new_request.sectors == req.lba:
            # 后置合并
            req.lba = new_request.lba
            req.sectors += new_request.sectors
            req.length += new_request.length
            merged = True
            break

    if not merged:
        queue.append(new_request)

    return queue
```

### 4.3 带宽与IOPS限制

**令牌桶限流器**：

```python
import time
import threading

class IOTokenBucket:
    """
    IO令牌桶限流器

    支持:
    - IOPS限制 (请求数/秒)
    - 带宽限制 (字节/秒)
    """

    def __init__(self, iops_limit=None, bps_limit=None):
        self.iops_limit = iops_limit
        self.bps_limit = bps_limit

        self.iops_tokens = iops_limit if iops_limit else float('inf')
        self.bps_tokens = bps_limit if bps_limit else float('inf')

        self.last_update = time.time()
        self.lock = threading.Lock()

    def consume(self, io_size):
        """
        尝试消费令牌

        返回:
            (success, wait_time)
        """
        with self.lock:
            now = time.time()
            elapsed = now - self.last_update

            # 补充令牌
            if self.iops_limit:
                self.iops_tokens = min(
                    self.iops_limit,
                    self.iops_tokens + elapsed * self.iops_limit
                )
            if self.bps_limit:
                self.bps_tokens = min(
                    self.bps_limit,
                    self.bps_tokens + elapsed * self.bps_limit
                )

            self.last_update = now

            # 检查令牌是否足够
            need_iops = 1
            need_bps = io_size

            if self.iops_tokens >= need_iops and self.bps_tokens >= need_bps:
                self.iops_tokens -= need_iops
                self.bps_tokens -= need_bps
                return True, 0

            # 计算需要等待的时间
            wait_iops = (need_iops - self.iops_tokens) / self.iops_limit if self.iops_limit else 0
            wait_bps = (need_bps - self.bps_tokens) / self.bps_limit if self.bps_limit else 0
            wait_time = max(wait_iops, wait_bps)

            return False, wait_time

    def throttle(self, io_size):
        """限流，需要时等待"""
        success, wait_time = self.consume(io_size)
        if not success:
            time.sleep(wait_time)
            return self.consume(io_size)[0]
        return True
```

## 6 形式化模型

### 6.1 IO调度问题定义

**形式化定义**：

$$
\text{IO调度问题} = (R, Q, C, O)
$$

其中：

- $R = \{r_1, r_2, \ldots, r_n\}$：存储资源
  - $r_i = (\text{type}, \text{capacity}, \text{bandwidth}, \text{latency})$
- $Q = \{q_1, q_2, \ldots, q_m\}$：IO请求队列
  - $q_j = (lba_j, size_j, dir_j, deadline_j, priority_j)$
- $C$：约束条件
  - 物理约束：$\text{seek\_time} = f(|lba_i - lba_j|)$
  - 时间约束：$t_{completion} \leq deadline_j$
  - 资源约束：$\sum bandwidth_i \leq bandwidth_{max}$
- $O$：优化目标
  - 最小化平均延迟：$\min \frac{1}{m}\sum_{j} t_{latency}(q_j)$
  - 最大化吞吐量：$\max \sum_{j} size_j / t_{total}$
  - 公平性：$\min \sigma(t_{wait}(q_j))$

### 6.2 调度算法复杂度

| 算法 | 插入复杂度 | 调度复杂度 | 空间复杂度 | 最优性 |
|------|-----------|-----------|-----------|--------|
| FIFO | $O(1)$ | $O(1)$ | $O(n)$ | 非最优 |
| SSTF | $O(n)$ | $O(n)$ | $O(n)$ | 局部最优 |
| SCAN | $O(\log n)$ | $O(1)$ | $O(n)$ | 最优（单指标） |
| CFQ | $O(\log n)$ | $O(\log n)$ | $O(n)$ | 公平性最优 |
| Deadline | $O(\log n)$ | $O(1)$ | $O(n)$ | 延迟保证 |
| BFQ | $O(\log n)$ | $O(\log n)$ | $O(n)$ | 多目标最优 |

### 6.3 定理：寻道优化下界

**定理14.1（寻道优化下界）**：

对于$n$个随机IO请求，最优寻道调度（电梯算法）的寻道时间下界为：

$$
T_{seek} \geq \frac{\alpha \cdot \sum_{i=1}^{n-1} |lba_{i+1} - lba_i|}{n}
$$

其中$\alpha$是单位距离寻道时间。

**证明**：

由三角不等式，任意调度路径长度不小于按LBA排序的路径长度。

电梯算法按LBA排序后单向扫描，达到此下界。

∎

**定理14.2（延迟-吞吐量权衡）**：

对于给定的IO负载，平均延迟$L$和吞吐量$T$满足：

$$
L \cdot T \geq \frac{1}{2} \cdot \frac{\sum_{i} size_i^2}{\sum_{i} size_i}
$$

等号成立当且仅当所有请求大小相同且均匀到达。

**调优脚本示例**：

```bash
#!/bin/bash
# Linux IO调度器自动调优脚本

# 检测设备类型并设置调度器
for disk in /sys/block/sd* /sys/block/nvme*; do
    if [ -d "$disk" ]; then
        device=$(basename $disk)

        # 检查是否为旋转设备
        rotational=$(cat $disk/queue/rotational 2>/dev/null || echo 1)

        if [ "$rotational" -eq 0 ]; then
            # SSD/NVMe
            if [[ $device == nvme* ]]; then
                echo none > $disk/queue/scheduler
                echo "NVMe $device: 设置调度器为 NONE"
            else
                echo noop > $disk/queue/scheduler
                echo "SSD $device: 设置调度器为 NOOP"
            fi
            # 增加队列深度
            echo 1024 > $disk/queue/nr_requests
        else
            # HDD
            echo bfq > $disk/queue/scheduler
            echo "HDD $device: 设置调度器为 BFQ"
            # 启用原生命令队列
            echo 31 > $disk/queue/iosched/slice_idle
        fi
    fi
done
```
