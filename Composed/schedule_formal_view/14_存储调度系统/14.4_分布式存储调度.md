## 1 分布式存储调度概述

### 1.1 分布式存储特征

**分布式存储核心特征**：

| 特征 | 描述 | 调度影响 |
|------|------|---------|
| **数据分布** | 数据分布在多个物理节点 | 需要数据定位与路由调度 |
| **副本机制** | 数据多副本保证可靠性 | 需要副本放置与选择调度 |
| **一致性** | 保证多副本数据一致性 | 需要一致性协议调度 |
| **可扩展性** | 支持水平扩展 | 需要动态负载均衡调度 |
| **容错性** | 节点故障自动恢复 | 需要故障检测与恢复调度 |

**分布式存储架构**：

```
┌─────────────────────────────────────────────────────────────┐
│                      客户端层                                │
│         (应用程序 / SDK / 文件系统客户端)                     │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                    元数据服务层                              │
│    (命名空间管理 / 数据定位 / 副本管理 / 一致性协调)           │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                     数据存储层                               │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐        │
│  │ 数据节点1 │  │ 数据节点2 │  │ 数据节点3 │  │ 数据节点N │        │
│  │ (副本1)  │  │ (副本2)  │  │ (副本3)  │  │ (副本...)│        │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘        │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 分布式存储调度的核心挑战

**四大核心挑战**：

1. **数据分布调度**：如何均匀分布数据，避免热点
2. **副本管理调度**：如何选择副本位置，平衡可靠性与性能
3. **一致性调度**：如何在一致性与性能之间权衡
4. **负载均衡调度**：如何动态均衡节点负载

**延迟分解**（典型分布式存储访问）：

```
客户端请求
  ↓ [客户端处理] ~0.1ms
网络传输
  ↓ [网络往返] ~0.5-2ms (同机房)
元数据查询
  ↓ [元数据服务] ~1-5ms
数据定位
  ↓ [副本选择] ~0.1ms
数据读取
  ↓ [存储IO] ~0.5-10ms (SSD/HDD)
响应返回
  ↓ [网络传输] ~0.5-2ms
客户端接收

总延迟: 2.7-19.7ms
```

### 1.3 分布式存储架构模型

**三种主流架构**：

| 架构类型 | 代表系统 | 优点 | 缺点 |
|---------|---------|------|------|
| **集中式元数据** | HDFS, GFS | 实现简单，一致性易保证 | 元数据服务器瓶颈 |
| **分布式元数据** | Ceph, GlusterFS | 无单点瓶颈，扩展性好 | 实现复杂，一致性难保证 |
| **无中心架构** | IPFS, Dynamo | 完全去中心化 | 查找开销大，延迟高 |

## 2 数据副本调度

### 2.1 副本放置策略

**副本放置目标**：

- **可靠性**：副本分布在不同故障域
- **性能**：副本靠近访问客户端
- **均衡**：存储负载均匀分布

**机架感知放置算法**：

```python
# 机架感知副本放置伪代码
def place_replicas(data_id, replication_factor, cluster_topology):
    """
    机架感知副本放置算法

    输入:
        data_id: 数据标识
        replication_factor: 副本因子
        cluster_topology: 集群拓扑信息
    输出:
        replica_nodes: 副本节点列表
    """
    selected_nodes = []
    selected_racks = set()

    # 第一副本：随机选择（考虑负载）
    first_node = select_node_by_load(cluster_topology.nodes)
    selected_nodes.append(first_node)
    selected_racks.add(first_node.rack)

    # 第二副本：不同机架
    for node in cluster_topology.nodes:
        if node.rack not in selected_racks and node not in selected_nodes:
            selected_nodes.append(node)
            selected_racks.add(node.rack)
            break

    # 第三副本：不同机架或随机
    if len(selected_nodes) < replication_factor:
        for node in cluster_topology.nodes:
            if node not in selected_nodes:
                # 优先选择不同机架，如果机架不足则随机
                if node.rack not in selected_racks or len(selected_racks) >= cluster_topology.num_racks:
                    selected_nodes.append(node)
                    selected_racks.add(node.rack)
                    break

    return selected_nodes
```

**一致性哈希放置**：

```
哈希空间：[0, 2^160)

数据键 → 哈希值 → 顺时针找到第一个节点

d1 ---- d2 ---- d3 ---- d4 ---- d5 ---- d6
  ↓       ↓       ↓       ↓       ↓       ↓
 N1      N2      N3      N1      N2      N3

其中 N1, N2, N3 是数据节点，均匀分布在哈希环上
```

**虚拟节点**：每个物理节点对应多个虚拟节点，改善负载均衡

| 策略 | 负载均衡 | 扩展性 | 实现复杂度 | 适用场景 |
|------|---------|--------|-----------|---------|
| 随机放置 | 差 | 差 | 低 | 测试环境 |
| 一致性哈希 | 好 | 好 | 中 | 大规模系统 |
| 机架感知 | 好 | 中 | 高 | 高可靠性要求 |
| 网络拓扑感知 | 最优 | 中 | 极高 | 跨数据中心 |

### 2.2 副本选择算法

**副本选择目标**：最小化读取延迟

```python
# 自适应副本选择算法
def select_replica(data_replicas, client_info, network_metrics):
    """
    自适应副本选择算法

    考虑因素:
        - 网络延迟
        - 节点负载
        - 节点健康状态
        - 数据本地性
    """
    candidates = []

    for replica in data_replicas:
        score = 0.0

        # 网络延迟权重 40%
        latency = network_metrics.get_latency(client_info, replica.node)
        score += 0.4 * (1.0 / (1.0 + latency))

        # 节点负载权重 30%
        load = replica.node.get_load()
        score += 0.3 * (1.0 - load)

        # 数据本地性权重 20%
        if replica.node.rack == client_info.rack:
            score += 0.2
        elif replica.node.dc == client_info.dc:
            score += 0.1

        # 健康度权重 10%
        health = replica.node.get_health_score()
        score += 0.1 * health

        candidates.append((replica, score))

    # 选择得分最高的副本
    return max(candidates, key=lambda x: x[1])[0]
```

### 2.3 副本一致性协议

**主从复制 (Primary-Backup)**：

```
写入流程:
客户端 → 主副本 → 同步到从副本 → 确认 → 返回客户端

读取流程:
客户端 → 任意副本 → 返回数据

特点:
- 强一致性（同步复制）
- 主节点瓶颈
- 写入延迟高
```

**多主复制 (Multi-Master)**：

```
写入流程:
客户端 → 任一主副本 → 异步复制到其他主副本

冲突解决:
- 时间戳优先 (Last-Writer-Wins)
- 向量时钟
- 应用层合并

特点:
- 高可用性
- 可能冲突
- 最终一致性
```

### 2.4 副本调度形式化模型

**副本放置问题**：

$$
\text{ReplicaPlacement} = \arg\min_{P} \left[ \alpha \cdot \text{Latency}(P) + \beta \cdot \text{Cost}(P) + \gamma \cdot \text{Risk}(P) \right]
$$

其中：

- $P$：副本放置方案
- $\text{Latency}(P)$：访问延迟
- $\text{Cost}(P)$：存储成本
- $\text{Risk}(P)$：故障风险
- $\alpha, \beta, \gamma$：权重系数

## 3 一致性调度

### 3.1 强一致性调度

**线性一致性 (Linearizability)**：

```
要求:
1. 每次读取返回最新写入的值
2. 所有操作看起来按某种全局顺序执行
3. 与实时顺序一致

实现:
- 分布式锁
- 共识协议 (Paxos/Raft)
- 两阶段提交 (2PC)

代价:
- 高延迟 (网络往返)
- 低吞吐量 (串行化)
```

**顺序一致性 (Sequential Consistency)**：

```
要求:
1. 所有操作看起来按某种全局顺序执行
2. 每个进程的操作顺序保持不变
3. 不保证实时顺序

比线性一致性弱，但实现更简单
```

### 3.2 最终一致性调度

**最终一致性模型**：

```
特性:
- 不保证立即一致性
- 保证最终达到一致
- 允许临时不一致

适用场景:
- DNS系统
- CDN缓存
- 购物车系统

优化:
- 读修复 (Read Repair)
- 反熵 (Anti-Entropy)
- 提示移交 (Hinted Handoff)
```

### 3.3 一致性协议详解

**Raft协议**：

```
三种角色:
1. Leader: 处理所有客户端请求
2. Follower: 被动接收日志
3. Candidate: 竞选Leader

流程:
1. Leader选举
   - 超时触发选举
   - 获得多数票成为Leader

2. 日志复制
   - Leader接收请求
   - 追加到本地日志
   - 并行复制到Followers
   - 获得多数确认后提交

3. 安全性保证
   - 只有一个Leader
   - 已提交日志不丢失
   - 日志顺序一致
```

**Raft算法伪代码**：

```python
class RaftNode:
    def __init__(self, node_id, peers):
        self.id = node_id
        self.peers = peers
        self.state = 'FOLLOWER'
        self.current_term = 0
        self.voted_for = None
        self.log = []
        self.commit_index = 0

    def request_vote(self, candidate_id, term):
        """处理投票请求"""
        if term > self.current_term:
            self.current_term = term
            self.state = 'FOLLOWER'
            self.voted_for = None

        if term == self.current_term and \
           (self.voted_for is None or self.voted_for == candidate_id) and \
           self.is_log_up_to_date(candidate_id):
            self.voted_for = candidate_id
            return True
        return False

    def append_entries(self, leader_id, term, entries, leader_commit):
        """处理日志复制请求"""
        if term < self.current_term:
            return False

        self.current_term = term
        self.state = 'FOLLOWER'

        # 一致性检查
        if len(self.log) > 0 and \
           (len(self.log) < entries[0].index or \
            self.log[entries[0].index - 1].term != entries[0].term):
            return False

        # 追加新条目
        for entry in entries:
            if entry.index > len(self.log):
                self.log.append(entry)
            elif self.log[entry.index - 1].term != entry.term:
                self.log = self.log[:entry.index - 1]
                self.log.append(entry)

        # 更新commit_index
        if leader_commit > self.commit_index:
            self.commit_index = min(leader_commit, len(self.log))

        return True
```

### 3.4 一致性-性能权衡

**CAP定理**：

```
分布式系统最多同时满足以下两项:
C - Consistency (一致性)
A - Availability (可用性)
P - Partition Tolerance (分区容错性)

由于网络分区不可避免，实际在C和A之间选择
```

**PACELC定理**（CAP扩展）：

```
如果有分区(P)，在可用性(A)和一致性(C)之间选择
否则(E)，在延迟(L)和一致性(C)之间选择
```

**一致性级别选择矩阵**：

| 应用场景 | 推荐一致性 | 延迟要求 | 可用性要求 |
|---------|-----------|---------|-----------|
| 银行转账 | 强一致性 | 高 | 中 |
| 库存系统 | 强一致性 | 中 | 高 |
| 社交网络 | 最终一致性 | 低 | 高 |
| 日志收集 | 最终一致性 | 低 | 高 |
| 配置管理 | 强一致性 | 中 | 高 |

## 4 负载均衡调度

### 4.1 数据分片策略

**范围分片 (Range Sharding)**：

```
数据键: [0, 1000)

分片1: [0, 250)    → 节点A
分片2: [250, 500)  → 节点B
分片3: [500, 750)  → 节点C
分片4: [750, 1000) → 节点D

优点: 范围查询高效
缺点: 热点风险 (如按时间分片，最新数据集中)
```

**哈希分片 (Hash Sharding)**：

```
分片 = hash(key) % num_shards

优点: 数据均匀分布
缺点: 范围查询低效
```

**一致性哈希分片**：

```
哈希环: [0, 2^160)

节点映射到环上多个点 (虚拟节点)
数据映射到顺时针第一个节点

优点:
- 节点增删只影响相邻数据
- 自动负载均衡
- 支持异构节点
```

### 4.2 负载均衡算法

**最小连接数算法**：

```python
def least_connections(nodes, request):
    """选择连接数最少的节点"""
    return min(nodes, key=lambda n: n.active_connections)
```

**加权轮询算法**：

```python
class WeightedRoundRobin:
    def __init__(self, nodes, weights):
        self.nodes = nodes
        self.weights = weights
        self.current_index = 0
        self.current_weight = 0
        self.max_weight = max(weights)
        self.gcd = gcd(weights)  # 最大公约数

    def select(self):
        while True:
            self.current_index = (self.current_index + 1) % len(self.nodes)
            if self.current_index == 0:
                self.current_weight -= self.gcd
                if self.current_weight <= 0:
                    self.current_weight = self.max_weight

            if self.weights[self.current_index] >= self.current_weight:
                return self.nodes[self.current_index]
```

**自适应负载均衡**：

```python
def adaptive_load_balance(nodes, request, metrics):
    """
    自适应负载均衡
    综合考虑CPU、内存、磁盘、网络负载
    """
    scores = []
    for node in nodes:
        # 归一化各指标
        cpu_score = 1 - node.cpu_usage
        mem_score = 1 - node.memory_usage
        disk_score = 1 - node.disk_io_utilization
        network_score = 1 - node.network_utilization

        # 加权综合
        total_score = (0.3 * cpu_score +
                      0.2 * mem_score +
                      0.3 * disk_score +
                      0.2 * network_score)

        scores.append((node, total_score))

    # 按得分排序，选择得分最高的节点
    scores.sort(key=lambda x: x[1], reverse=True)
    return scores[0][0]
```

### 4.3 动态负载均衡

**数据迁移触发条件**：

| 条件 | 阈值 | 动作 |
|------|------|------|
| 节点负载 | > 80% | 迁移数据到负载低节点 |
| 存储容量 | > 85% | 迁移数据到其他节点 |
| 访问热点 | QPS > 阈值 | 增加副本 |
| 节点故障 | 心跳超时 | 重建副本 |

**数据迁移调度**：

```python
def schedule_data_migration(cluster_state):
    """
    数据迁移调度器

    目标: 最小化迁移开销，同时达到负载均衡
    """
    migrations = []

    # 识别过载和欠载节点
    avg_load = sum(n.load for n in cluster_state.nodes) / len(cluster_state.nodes)
    overloaded = [n for n in cluster_state.nodes if n.load > avg_load * 1.2]
    underloaded = [n for n in cluster_state.nodes if n.load < avg_load * 0.8]

    for src in overloaded:
        # 计算需要迁移的数据量
        excess_load = src.load - avg_load

        for dst in underloaded:
            if excess_load <= 0:
                break

            available_capacity = avg_load - dst.load
            migrate_amount = min(excess_load, available_capacity)

            # 选择要迁移的分片 (优先选择访问频率低的)
            shard = select_shard_for_migration(src, migrate_amount)

            if shard:
                migrations.append(MigrationTask(src, dst, shard))
                excess_load -= shard.load_contribution
                dst.load += shard.load_contribution

    return migrations
```

## 5 纠删码调度

### 5.1 纠删码原理

**纠删码 vs 副本**：

| 特性 | 3副本 | RS(6,3)纠删码 |
|------|-------|--------------|
| 存储冗余度 | 3x | 1.5x |
| 可靠性 | 2节点容错 | 3节点容错 |
| 读取性能 | 好 | 中等 |
| 写入性能 | 好 | 差 (需计算) |
| 网络开销 | 低 | 高 |
| 计算开销 | 低 | 高 |

**Reed-Solomon编码**：

```
数据块: D1, D2, ..., Dk
编码块: P1, P2, ..., Pm

生成矩阵 G (Vandermonde矩阵):

[D1]   [1  0  ...  0]   [D1]
[D2]   [0  1  ...  0]   [D2]
[...] = [...      ...] x [...]
[Dk]   [0  0  ...  1]   [Dk]
[P1]   [g11 g12 ... g1k]
[P2]   [g21 g22 ... g2k]
[...]  [...         ...]
[Pm]   [gm1 gm2 ... gmk]

解码: 从任意k个块恢复原始数据
```

**故障检测与恢复**：

```python
def handle_node_failure(failed_node, cluster_state):
    """
    节点故障处理

    1. 识别受影响的数据块
    2. 调度恢复任务
    3. 优先恢复关键数据
    """
    affected_blocks = cluster_state.get_blocks_on_node(failed_node)

    recovery_tasks = []
    for block in affected_blocks:
        # 获取同组的其他块
        stripe_group = cluster_state.get_stripe_group(block)
        available_blocks = [b for b in stripe_group if b.node != failed_node]

        # 选择恢复目标节点
        target_node = select_recovery_target(cluster_state)

        # 创建恢复任务
        task = RecoveryTask(
            source_blocks=available_blocks,
            target_node=target_node,
            missing_block=block,
            priority=calculate_priority(block)
        )
        recovery_tasks.append(task)

    # 按优先级排序，调度执行
    recovery_tasks.sort(key=lambda t: t.priority, reverse=True)
    return recovery_tasks
```

**恢复带宽优化**：

```
传统恢复: 从幸存节点读取所有数据块
优化恢复: 只读取必要的数据块进行解码

例如 RS(6,3):
- 1个数据块丢失
- 传统: 读取6个数据块
- 优化: 读取任意6个可用块（可能包括校验块）
- 恢复计算: 矩阵求逆 + 矩阵乘法
```

**令牌桶算法**：

```python
class TokenBucket:
    """令牌桶限流器"""

    def __init__(self, rate, capacity):
        """
        rate: 令牌产生速率 (tokens/second)
        capacity: 桶容量 (最大突发流量)
        """
        self.rate = rate
        self.capacity = capacity
        self.tokens = capacity
        self.last_update = time.time()

    def consume(self, tokens):
        """尝试消费指定数量的令牌"""
        now = time.time()
        elapsed = now - self.last_update

        # 添加新产生的令牌
        self.tokens = min(self.capacity, self.tokens + elapsed * self.rate)
        self.last_update = now

        if self.tokens >= tokens:
            self.tokens -= tokens
            return True
        else:
            return False

    def get_wait_time(self, tokens):
        """获取需要等待的时间"""
        if self.tokens >= tokens:
            return 0
        return (tokens - self.tokens) / self.rate

# 使用示例
bucket = TokenBucket(rate=1000, capacity=2000)  # 1000 IOPS，突发2000

if bucket.consume(1):
    process_io_request()
else:
    wait_time = bucket.get_wait_time(1)
    delay_io_request(wait_time)
```

**权重轮询调度**：

```python
class WeightedFairQueue:
    """加权公平队列调度器"""

    def __init__(self):
        self.queues = {}  # cgroup -> queue
        self.weights = {}  # cgroup -> weight
        self.virtual_time = 0
        self.last_finish_time = {}

    def enqueue(self, cgroup, request):
        """将请求加入队列"""
        if cgroup not in self.queues:
            self.queues[cgroup] = []
            self.weights[cgroup] = get_cgroup_weight(cgroup)

        # 计算虚拟开始时间
        start_time = max(self.virtual_time, self.last_finish_time.get(cgroup, 0))
        request.virtual_start_time = start_time
        request.virtual_finish_time = start_time + request.cost / self.weights[cgroup]

        self.queues[cgroup].append(request)

    def dequeue(self):
        """选择下一个执行的请求"""
        min_finish_time = float('inf')
        selected_cgroup = None

        for cgroup, queue in self.queues.items():
            if queue:
                request = queue[0]
                if request.virtual_finish_time < min_finish_time:
                    min_finish_time = request.virtual_finish_time
                    selected_cgroup = cgroup

        if selected_cgroup:
            request = self.queues[selected_cgroup].pop(0)
            self.last_finish_time[selected_cgroup] = request.virtual_finish_time
            self.virtual_time = request.virtual_start_time
            return request

        return None
```

**CRUSH算法**：

```python
def crush_select(bucket, replica_num, rule):
    """
    CRUSH: Controlled Replication Under Scalable Hashing

    输入:
        bucket: 存储桶 (根节点)
        replica_num: 副本编号
        rule: 放置规则
    输出:
        选中的OSD列表
    """
    result = []

    for step in rule.steps:
        if step.op == 'take':
            current = bucket
        elif step.op == 'choose':
            # 伪随机选择子节点
            for _ in range(step.n):
                hash_val = crush_hash(step.type, replica_num, current.id)
                child = select_child_by_weight(current, hash_val)
                if step.type == 'osd':
                    result.append(child)
                else:
                    current = child
        elif step.op == 'emit':
            return result

    return result

def crush_hash(x, r, id):
    """CRUSH哈希函数"""
    return hash((x, r, id)) % 0xFFFFFFFF
```

**CRUSH算法数学分析**：

```
CRUSH核心特性:
1. 伪随机分布: hash(x, r, id) 产生确定性但均匀的分布
2. 权重感知: 节点权重决定数据分布比例
   P(选择节点i) = weight_i / Σweight_j
3. 故障域隔离: 通过规则确保副本分布在不同故障域

数据分布均匀性分析:
对于N个OSD，每个权重相等，数据对象O的副本R放置:
  OSD_R = argmax_i (hash(O, R, i) × weight_i)

一致性哈希对比:
- 传统一致性哈希: 虚拟节点数V，迁移数据量 ≈ 1/V
- CRUSH: 基于权重和层级，迁移数据量 ≈ weight_change / total_weight

CRUSH层级结构:
root (数据中心)
  ├── room (机房)
  │     ├── rack (机架)
  │     │     ├── host (主机)
  │     │     │     ├── osd.1
  │     │     │     └── osd.2
  │     │     └── host
  │     └── rack
  └── room

故障域保证:
- 副本1: root → room1 → rack1 → host1 → osd.1
- 副本2: root → room1 → rack2 → host3 → osd.5  (不同机架)
- 副本3: root → room2 → rack5 → host8 → osd.12 (不同机房)
```

**GlusterFS弹性哈希调度**：

def glusterfs_elastic_hash(path):
    """
    GlusterFS弹性哈希算法 (EHA)
    
    特点:
    - 无中心化元数据服务器
    - 128-bit哈希空间
    - 哈希值直接映射到存储节点
    """
    # 计算文件路径的128位哈希
    hash_val = fnv1a_128(path)
    
    # 子卷选择 (子卷 = 一组brick)
    subvol_index = hash_val % num_subvols
    
    # 在子卷内选择具体brick (基于DHT)
    brick_index = (hash_val // num_subvols) % bricks_per_subvol
    
    return subvol_index, brick_index

def glusterfs_rebalance(old_bricks, new_bricks, files):
    """
    GlusterFS再平衡调度
    
    当添加/删除brick时，需要迁移数据
    """
    migrations = []
    
    for file in files:
        old_location = glusterfs_hash(file, old_bricks)
        new_location = glusterfs_hash(file, new_bricks)
        
        if old_location != new_location:
            migrations.append({
                'file': file,
                'from': old_location,
                'to': new_location
            })
    
    return migrations

```python
def hdfs_block_placement(block, replication_factor, cluster_topology):
    """
    HDFS默认副本放置策略 (机架感知)

    第1副本: 本地节点 (如果本地是DataNode)
    第2副本: 不同机架的节点
    第3副本: 与第2副本同机架的不同节点
    第4+副本: 随机分布
    """
    writer = get_writer_node()
    writer_rack = writer.rack

    # 第1副本
    replica1 = writer if writer.is_datanode else select_random_node()

    # 第2副本: 不同机架
    replica2 = select_random_node(exclude_rack=writer_rack)
    replica2_rack = replica2.rack

    # 第3副本: 与第2副本同机架
    replica3 = select_random_node(
        require_rack=replica2_rack,
        exclude_nodes=[replica2]
    )

    replicas = [replica1, replica2, replica3]

    # 第4+副本: 随机选择
    while len(replicas) < replication_factor:
        replica = select_random_node(exclude_nodes=replicas)
        replicas.append(replica)

    return replicas
```

| MQ-DEADLINE | HDD/SSD | 截止时间保证 | HDD默认 |
| BFQ | HDD/SSD | 预算公平队列 | 桌面默认 |
| Kyber | NVMe | 延迟目标 | NVMe推荐 |
| NONE | NVMe | 无调度 | 部分NVMe |

**BFQ调度器实现**：

```c
// BFQ核心数据结构
struct bfq_queue {
    struct request *next_rq;
    int pid;                    // 进程ID
    unsigned long budget;       // IO预算
    int service;                // 已服务IO
    int weight;                 // 队列权重
    struct rb_node entity;      // 红黑树节点
    int dispatched;             // 已分发请求
    bool in_service;            // 是否正在服务
};

struct bfq_data {
    struct bfq_queue *in_service_queue;
    struct rb_root_active;      // 活跃队列红黑树
    struct rb_root_idle;        // 空闲队列红黑树
    unsigned int peak_rate;     // 峰值带宽
};

// BFQ调度流程
static struct request *bfq_dispatch_request(struct blk_mq_hw_ctx *hctx)
{
    struct bfq_data *bfqd = hctx->queue->elevator->elevator_data;
    struct bfq_queue *bfqq = bfqd->in_service_queue;

    // 1. 检查当前队列预算
    if (bfqq && bfqq->service >= bfqq->budget) {
        // 预算耗尽，选择下一个队列
        bfqq = bfq_select_next_queue(bfqd);
        bfqd->in_service_queue = bfqq;
    }

    // 2. 从选中队列取请求
    if (bfqq && !list_empty(&bfqq->fifo)) {
        struct request *rq = list_first_entry(&bfqq->fifo, struct request, queuelist);
        bfqq->service += blk_rq_sectors(rq);
        return rq;
    }

    return NULL;
}
```

## 9 形式化模型

### 9.1 分布式存储调度问题定义

**形式化定义**：

$$
\text{分布式存储调度问题} = (N, D, R, C, O, T)
$$

其中：

- $N = \{n_1, n_2, \ldots, n_k\}$：存储节点集合
  - $n_i = (\text{capacity}_i, \text{bandwidth}_i, \text{latency}_i, \text{location}_i)$
- $D = \{d_1, d_2, \ldots, d_m\}$：数据对象集合
  - $d_j = (\text{size}_j, \text{access\_pattern}_j, \text{qos\_req}_j)$
- $R$：副本关系，$R: D \times N \rightarrow \{0,1\}$
- $C$：约束条件
  - 一致性约束：$\text{consistency}(d_j) \in \{\text{strong}, \text{eventual}\}$
  - 容量约束：$\sum_{d_j \in n_i} \text{size}_j \leq \text{capacity}_i$
  - 带宽约束：$\sum_{d_j \in n_i} \text{traffic}_j \leq \text{bandwidth}_i$
  - 副本约束：$\sum_{n_i} R(d_j, n_i) = \text{replication\_factor}_j$
- $O$：优化目标
  - 最小化延迟：$\min \sum_{d_j} \text{latency}(d_j)$
  - 最大化吞吐量：$\max \sum_{d_j} \text{throughput}(d_j)$
  - 负载均衡：$\min \sigma(\text{load}(n_i))$
- $T$：时间维度（动态调度）

### 9.2 调度算法复杂度

| 算法 | 时间复杂度 | 空间复杂度 | 适用场景 |
|------|-----------|-----------|---------|
| 随机放置 | $O(1)$ | $O(1)$ | 小规模 |
| 一致性哈希 | $O(\log n)$ | $O(n)$ | 大规模 |
| 机架感知 | $O(n)$ | $O(n)$ | 高可靠 |
| 负载均衡 | $O(n^2)$ | $O(n)$ | 动态调度 |
| Raft共识 | $O(n)$ | $O(n)$ | 一致性 |

### 9.3 定理：CAP定理形式化

**定理14.4（CAP定理形式化）**：

对于分布式存储系统，在存在网络分区($P$)的情况下，强一致性($C$)和可用性($A$)不可兼得：

$$
\neg (C \land A \land P) \equiv P \rightarrow \neg (C \land A)
$$

**证明**：

假设存在网络分区，将系统分为两个分区$P_1$和$P_2$。

情况1：保证可用性($A$)

- $P_1$和$P_2$都继续服务请求
- 可能导致$P_1$和$P_2$对同一数据返回不同值
- 违反强一致性($C$)

情况2：保证强一致性($C$)

- 需要协调$P_1$和$P_2$
- 由于网络分区，协调无法完成
- 必须拒绝部分请求
- 违反可用性($A$)

因此，$C$和$A$不可兼得。∎
