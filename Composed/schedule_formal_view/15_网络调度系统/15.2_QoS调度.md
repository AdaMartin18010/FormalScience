### 📊 2. 多维对比矩阵

#### 2.1 15.2 QoS调度对比矩阵

| 维度 | 优先级调度 | 带宽保证 | 延迟保证 | 丢包控制 |
|------|-----------|---------|---------|---------|
| **性能** | 低延迟<10ms | 精确带宽分配 | 有界延迟保证 | 低丢包率<0.1% |
| **复杂度** | 简单(O(1)) | 中等(需队列管理) | 复杂(需延迟监测) | 复杂(需AQM算法) |
| **适用场景** | 实时应用 | 多租户网络 | 实时语音/视频 | 高可靠性网络 |
| **技术成熟度** | 成熟(>20年) | 成熟(>20年) | 成熟(>15年) | 成熟(>10年) |

#### 2.2 技术特性对比矩阵

| 技术 | 优势 | 劣势 | 适用场景 | 性能 |
|------|------|------|---------|------|
| **IntServ (RFC 1633)** | 端到端QoS保证、资源预留 | 可扩展性差、状态维护复杂 | 小规模网络、严格QoS要求 | 延迟<5ms，丢包率<0.1% |
| **DiffServ (RFC 2474)** | 可扩展性好、无状态 | 缺乏端到端保证、粒度粗 | 大规模网络、Internet骨干网 | 延迟<50ms，吞吐量>10Gbps |
| **MPLS QoS** | 流量工程、路径优化 | 需要MPLS支持 | 运营商网络、企业WAN | 延迟<10ms，带宽利用率>90% |
| **5G QoS (5QI)** | 端到端切片、低延迟 | 需要5G网络支持 | 5G网络、边缘计算 | 延迟<1ms(URLLC)，带宽>1Gbps |
| **802.1p (VLAN优先级)** | 简单、硬件支持 | 仅8个优先级级别 | 局域网、数据中心 | 延迟<1ms，简单场景 |
| **DSCP (Differentiated Services)** | 64个优先级级别、标准 | 需要端到端支持 | Internet、企业网络 | 延迟<100ms，标准兼容 |

#### 2.3 实现方式对比矩阵

| 实现方式 | 复杂度 | 性能 | 可维护性 | 扩展性 |
|---------|-------|------|---------|-------|
| **硬件实现(TCAM)** | 低 | 极高(<1μs延迟) | 低(需硬件支持) | 低(硬件固定) |
| **内核实现(Linux tc)** | 中等 | 高(<10μs延迟) | 中等(需内核开发) | 中等(内核版本依赖) |
| **用户态实现(DPDK)** | 高 | 极高(<3μs延迟) | 高(用户态开发) | 高(独立升级) |
| **eBPF/XDP实现** | 中等 | 高(<5μs延迟) | 高(可动态加载) | 高(无需重启) |

## 1 QoS调度概述

### 1.1 QoS的核心需求

**QoS（Quality of Service）**核心需求：

- **带宽保证**：保证最小带宽或限制最大带宽
- **延迟保证**：保证最大延迟或平均延迟
- **丢包率控制**：控制丢包率在可接受范围
- **抖动控制**：控制延迟抖动（Jitter）

**QoS服务模型**：

- **IntServ（Integrated Services）**：端到端资源预留（RFC 1633）
- **DiffServ（Differentiated Services）**：基于类的服务质量（RFC 2474）
- **MPLS QoS**：基于标签的QoS
- **5G网络切片QoS**：端到端网络切片QoS保证（3GPP TS 23.501）

### 1.3 QoS服务模型详解

#### 1.3.1 IntServ模型（RFC 1633）

**IntServ（Integrated Services）**模型提供端到端的资源预留服务，保证每个流的QoS需求。

**核心机制**：

```text
应用请求资源预留（RSVP）
  ↓
网络节点检查资源可用性
  ↓
建立端到端资源预留路径
  ↓
传输期间保证QoS
  ↓
释放资源
```

**IntServ服务类型**：

| **服务类型** | **延迟保证** | **带宽保证** | **丢包保证** | **适用场景** |
|------------|------------|------------|------------|------------|
| **Guaranteed Service** | 严格上限 | 精确保证 | 几乎为零 | 实时控制、关键业务 |
| **Controlled Load** | 尽力保证 | 近似保证 | 低丢包 | 自适应实时应用 |
| **Best Effort** | 无保证 | 无保证 | 可能高丢包 | 普通数据传输 |

**RSVP（Resource Reservation Protocol）**：

RSVP是IntServ的信令协议，用于资源预留请求和响应。

$$
\text{RSVP}_{path} = \{(sender, receiver, TSpec, RSpec)\}
$$

其中：

- $TSpec$（Traffic Specification）：流量规格，描述流量特征
- $RSpec$（Reservation Specification）：预留规格，描述QoS要求

**RSVP消息类型**：

| **消息类型** | **方向** | **功能** | **字段** |
|------------|---------|---------|---------|
| **Path** | 发送→接收 | 建立路径状态 | Session, Sender TSpec |
| **Resv** | 接收→发送 | 预留资源请求 | FlowSpec, FilterSpec |
| **PathErr** | 双向 | 路径错误通知 | Error Spec |
| **ResvErr** | 双向 | 预留错误通知 | Error Spec |
| **PathTear** | 发送→接收 | 拆除路径状态 | Session |
| **ResvTear** | 接收→发送 | 释放预留资源 | Session |

**IntServ优势与局限**：

| **维度** | **优势** | **局限** |
|---------|---------|---------|
| **QoS保证** | 端到端精确保证 | 网络复杂度高 |
| **可扩展性** | 每流独立控制 | 状态爆炸问题 |
| **灵活性** | 支持多种服务类型 | 配置复杂度高 |
| **适用规模** | 小规模网络（<1000流） | 大规模网络不适用 |

**形式化模型**：

$$
\text{IntServ}_{guarantee} = \{f_i \mid \forall f_i \in F: bw_i \geq bw_{req,i} \land d_i \leq D_{req,i}\}
$$

#### 1.3.2 DiffServ模型（RFC 2474）

**DiffServ（Differentiated Services）**模型提供基于类别的QoS服务，通过DSCP标记实现逐跳行为（PHB）。

**核心机制**：

```text
入口路由器标记DSCP值
  ↓
核心路由器根据DSCP执行PHB
  ↓
出口路由器可能重标记DSCP
  ↓
端到端逐跳QoS保证
```

**DiffServ域架构**：

| **组件** | **功能** | **位置** | **关键行为** |
|---------|---------|---------|------------|
| **DS边界节点** | 分类、标记、计量、整形 | 域边缘 | 流量分类与DSCP标记 |
| **DS内部节点** | PHB执行 | 域内部 | 基于DSCP的转发 |
| **DS区域** | 连续的DS域 | 网络范围 | 统一PHB策略 |

**DSCP（Differentiated Services Code Point）**：

DSCP是IP头部的6位字段（原ToS字段），定义64种服务类别。

$$
\text{DSCP} = \{0, 1, 2, \ldots, 63\} \subset \text{IPv4 ToS Byte}
$$

**标准PHB定义**：

| **PHB** | **DSCP值** | **二进制** | **转发行为** | **典型应用** |
|--------|-----------|-----------|------------|------------|
| **EF** | 46 | 101110 |  Expedited Forwarding（加速转发） | 语音、视频 |
| **AF41** | 34 | 100010 |  Assured Forwarding Class 4 | 交互式视频 |
| **AF42** | 36 | 100100 |  Assured Forwarding Class 4 | 流媒体 |
| **AF43** | 38 | 100110 |  Assured Forwarding Class 4 | 低优先级视频 |
| **AF31** | 26 | 011010 |  Assured Forwarding Class 3 | 关键业务 |
| **AF32** | 28 | 011100 |  Assured Forwarding Class 3 | 商业数据 |
| **AF33** | 30 | 011110 |  Assured Forwarding Class 3 | 批量数据 |
| **AF21** | 18 | 010010 |  Assured Forwarding Class 2 | 低延迟数据 |
| **AF22** | 20 | 010100 |  Assured Forwarding Class 2 | 普通数据 |
| **AF23** | 22 | 010110 |  Assured Forwarding Class 2 | 高吞吐数据 |
| **AF11** | 10 | 001010 |  Assured Forwarding Class 1 | 尽力数据 |
| **AF12** | 12 | 001100 |  Assured Forwarding Class 1 | 批量数据 |
| **AF13** | 14 | 001110 |  Assured Forwarding Class 1 | 背景流量 |
| **CS** | 0-7 | 000xxx |  Class Selector（兼容IP Precedence） | 向后兼容 |
| **BE** | 0 | 000000 |  Best Effort（尽力而为） | 默认流量 |

**AF PHB丢弃优先级**：

$$
\text{DropPriority} = \begin{cases}
1 & \text{(最低丢弃概率，DSCP x1)} \\
2 & \text{(中等丢弃概率，DSCP x2)} \\
3 & \text{(最高丢弃概率，DSCP x3)}
\end{cases}
$$

**DiffServ优势与局限**：

| **维度** | **优势** | **局限** |
|---------|---------|---------|
| **可扩展性** | 每类而非每流，支持大规模 | 端到端保证弱 |
| **复杂度** | 边缘复杂、核心简单 | 需要全网协调 |
| **灵活性** | 基于类的聚合管理 | 粒度较粗 |
| **部署** | 渐进式部署可行 | 跨域协调困难 |

#### 1.3.3 MPLS QoS模型

**MPLS（Multi-Protocol Label Switching）QoS**模型通过标签交换路径（LSP）提供QoS保证，支持流量工程和快速重路由。

**核心机制**：

```text
入口LSR分类并压入标签
  ↓
核心LSR根据标签交换
  ↓
出口LSR弹出标签并转发
  ↓
LSP提供确定的QoS路径
```

**MPLS QoS架构**：

| **组件** | **功能** | **协议/技术** | **QoS特性** |
|---------|---------|-------------|------------|
| **LSR（Label Switching Router）** | 标签交换 | MPLS | 快速转发 |
| **LSP（Label Switched Path）** | 标签交换路径 | RSVP-TE/LDP | 显式路由 |
| **TE（Traffic Engineering）** | 流量工程 | RSVP-TE | 带宽预留 |
| **FRR（Fast Reroute）** | 快速重路由 | MPLS-TE | 故障恢复<50ms |

**MPLS EXP字段**：

EXP（Experimental）字段是MPLS标签头部的3位字段，用于QoS标记。

$$
\text{EXP} = \{0, 1, 2, 3, 4, 5, 6, 7\}
$$

**EXP到PHB映射**：

| **EXP值** | **PHB** | **服务类型** | **转发优先级** |
|----------|--------|------------|--------------|
| 7 | Network Control | 网络控制 | 最高 |
| 6 | Internetwork Control | 网间控制 | 极高 |
| 5 | EF | 语音、视频 | 高 |
| 4 | AF41 | 实时交互 | 中高 |
| 3 | AF31 | 关键业务 | 中 |
| 2 | AF21 | 优先级数据 | 中低 |
| 1 | AF11 | 普通数据 | 低 |
| 0 | BE | 尽力而为 | 最低 |

**DSCP到EXP映射规则**：

$$
\text{EXP} = \left\lfloor \frac{\text{DSCP}}{8} \right\rfloor \quad \text{(默认映射)}
$$

**MPLS-TE带宽预留**：

$$
\text{LSP}_{bandwidth} = \min\left(\text{ReqBW}, \min_{\text{links} \in \text{path}}(\text{ReservableBW}_{link})\right)
$$

**MPLS QoS优势**：

| **特性** | **MPLS QoS优势** | **传统IP QoS对比** |
|---------|-----------------|------------------|
| **转发效率** | 标签交换O(1)，无需最长前缀匹配 | IP路由查找O(log n) |
| **流量工程** | 显式路径控制，避开拥塞 | 依赖IGP最短路径 |
| **VPN支持** | 天然支持MPLS VPN | 需要额外封装 |
| **故障恢复** | FRR<50ms | 路由收敛数秒 |
| **QoS粒度** | 基于LSP的精细控制 | 基于流的粗粒度 |

#### 1.3.4 5G网络切片QoS模型（3GPP TS 23.501）

### 1.2 QoS调度目标

QoS调度需要在以下目标之间权衡：

1. **服务质量保证**：满足不同流的QoS要求
2. **资源利用率**：最大化链路利用率
3. **公平性**：保证各流获得公平资源
4. **可扩展性**：支持大量流和复杂QoS要求

**QoS调度目标模型**（2025年11月最新）：

$$
\text{QoS调度目标} = \arg\max_{\text{Schedule}} \left\{ \alpha \cdot \text{QoS}_{satisfaction} + \beta \cdot \text{Utilization} + \gamma \cdot \text{Fairness} - \delta \cdot \text{Complexity} \right\}
$$

其中：

- $\alpha + \beta + \gamma = 1$：权重系数
- $\text{QoS}_{satisfaction}$：QoS满足率
- $\text{Utilization}$：资源利用率
- $\text{Fairness}$：公平性指标
- $\text{Complexity}$：调度复杂度

**QoS满足率**：

$$
\text{QoS}_{satisfaction} = \frac{1}{n} \sum_{i=1}^{n} \mathbb{1}(\text{QoS}_i \text{ satisfied})
$$

其中$\mathbb{1}(\cdot)$是指示函数，$\text{QoS}_i \text{ satisfied}$表示流$i$的QoS要求得到满足。

**资源利用率**：

$$
\text{Utilization} = \frac{\sum_i bw_i}{B_{total}} \times 100\%
$$

**公平性指标**（Jain's Fairness Index）：

$$
\text{Fairness} = \frac{(\sum_i bw_i)^2}{n \cdot \sum_i bw_i^2}
$$

**QoS调度性能指标**（2025年11月最新）：

| **指标** | **目标值** | **2025年最新** | **提升** |
|---------|----------|--------------|---------|
| **QoS满足率** | > 85% | 98%+ | +15%+ |
| **资源利用率** | > 70% | 90%+ | +29%+ |
| **公平性指数** | > 0.8 | 0.95+ | +19%+ |
| **调度延迟** | < 10μs | < 5μs | 50%+ |
| **可扩展性** | 1000流 | 10000+流 | 10x+ |

**QoS调度权衡**（2025年11月最新）：

- **QoS保证 vs 资源利用率**：严格QoS保证可能降低资源利用率（60-70%），灵活QoS保证可提高利用率（90%+）
- **公平性 vs 性能**：完全公平可能降低性能，加权公平可在公平性和性能间平衡
- **可扩展性 vs 复杂度**：简单算法可扩展性好但QoS保证弱，复杂算法QoS保证强但可扩展性差

#### 1.4.1 IEEE 802.1p（二层优先级）

**IEEE 802.1p**定义了以太网帧的优先级标记机制，使用VLAN标签的PCP（Priority Code Point）字段。

**PCP字段定义**：

PCP是802.1Q VLAN标签的3位字段，位于TCI（Tag Control Information）中。

## 2 优先级调度

### 2.1 严格优先级调度

**严格优先级调度（Strict Priority）**：

```text
多个优先级队列
  ↓
高优先级队列优先调度
  ↓
高优先级队列为空时才调度低优先级
  ↓
低优先级可能饿死
```

**特点**：

- **延迟最低**：高优先级流延迟最低
- **不公平**：低优先级流可能饿死
- **简单高效**：实现简单，开销低

**严格优先级调度模型**（2025年11月最新）：

$$
\text{Schedule}(packet) = \begin{cases}
\text{HighPriorityQueue} & \text{if } \text{Priority}(packet) = \text{High} \\
\text{LowPriorityQueue} & \text{if } \text{HighPriorityQueue} = \emptyset
\end{cases}
$$

**延迟分析**：

- **高优先级流延迟**：$d_{high} \approx \frac{L_{max}}{C}$（接近理论最小值）
- **低优先级流延迟**：$d_{low} \leq \infty$（可能无限延迟，饿死）

**性能指标**（2025年11月最新）：

- **高优先级流延迟**：< 1ms（关键流量）
- **调度开销**：$O(1)$（常数时间）
- **资源利用率**：85-95%（高优先级流占用大部分资源）

**适用场景**：关键业务流量、实时流量、语音/视频流

### 2.2 加权优先级调度

**加权优先级调度（Weighted Priority）**：

```text
为每个优先级分配权重
  ↓
按权重比例分配带宽
  ↓
保证最低带宽
  ↓
避免低优先级饿死
```

**带宽分配**：

$$
bw_i = \frac{w_i}{\sum_j w_j} \times B_{total}
$$

**特点**：

- **公平性**：保证各优先级获得最小带宽
- **灵活性**：通过权重调整带宽分配
- **复杂度**：实现复杂度中等

**加权优先级调度模型**（2025年11月最新）：

$$
\text{Bandwidth}_i = \max\left(bw_{min,i}, \frac{w_i}{\sum_j w_j} \times B_{total}\right)
$$

其中：

- $bw_{min,i}$：优先级$i$的最小带宽保证
- $w_i$：优先级$i$的权重
- $B_{total}$：总带宽

**延迟分析**：

$$
d_i \leq \frac{L_{max}}{r_i} + \frac{\sum_{j \neq i} L_j}{C}
$$

其中$r_i = \frac{w_i}{\sum_j w_j} \times C$是优先级$i$的服务速率。

**权重分配策略**（2025年11月最新）：

- **等权重**：$w_i = w_j$（公平分配）
- **比例权重**：$w_i \propto bw_{min,i}$（按最小带宽需求分配）
- **优先级权重**：$w_i = 2^{priority_i}$（指数权重，高优先级获得更多资源）

**性能指标**（2025年11月最新）：

- **带宽保证**：> 95%（最小带宽保证）
- **延迟上界**：$O(\frac{L}{r})$（有界延迟）
- **资源利用率**：80-90%（权重分配影响利用率）
- **调度开销**：$O(\log n)$（优先级队列）

**适用场景**：通用场景、多租户网络、混合流量

## 3 带宽保证调度

### 3.1 最小带宽保证

**最小带宽保证（Minimum Bandwidth Guarantee）**：

```text
为每个流分配最小带宽
  ↓
保证最小带宽可用
  ↓
剩余带宽按权重分配
```

**带宽分配模型**：

$$
bw_i = \max(bw_{min,i}, \frac{w_i}{\sum_j w_j} \times B_{remaining})
$$

其中$B_{remaining} = B_{total} - \sum_j bw_{min,j}$。

**最小带宽保证机制**（2025年11月最新）：

最小带宽保证通过资源预留和调度策略实现。

**资源预留检查**：

$$
\text{Admit}(flow_i) \iff \sum_j bw_{min,j} + bw_{min,i} \leq B_{total}
$$

**带宽分配算法**：

1. **预留最小带宽**：为每个流预留$bw_{min,i}$
2. **分配剩余带宽**：剩余带宽$B_{remaining}$按权重分配
3. **动态调整**：根据流量变化动态调整带宽分配

**带宽分配算法实现**（2025年11月最新）：

**算法伪代码**：

```python
def allocate_bandwidth(flows, total_bandwidth):
    """
    带宽分配算法
    :param flows: 流列表，每个流包含 (min_bw, weight, current_bw)
    :param total_bandwidth: 总带宽
    :return: 分配后的带宽字典
    """
    # 步骤1：预留最小带宽
    reserved_bw = sum(flow.min_bw for flow in flows)
    remaining_bw = total_bandwidth - reserved_bw

    # 准入控制检查
    if reserved_bw > total_bandwidth:
        return None  # 拒绝新流

    # 步骤2：按权重分配剩余带宽
    total_weight = sum(flow.weight for flow in flows)
    allocations = {}

    for flow in flows:
        # 最小带宽保证
        min_allocation = flow.min_bw

        # 按权重分配的剩余带宽
        weighted_allocation = (flow.weight / total_weight) * remaining_bw

        # 最终分配 = max(最小带宽, 加权分配)
        allocations[flow.id] = max(min_allocation, weighted_allocation)

    return allocations

def dynamic_adjust(flows, allocations, network_state):
    """
    动态调整带宽分配
    :param flows: 流列表
    :param allocations: 当前分配
    :param network_state: 网络状态（拥塞、利用率等）
    :return: 调整后的分配
    """
    # 根据网络状态调整权重
    for flow in flows:
        if network_state.congested:
            # 拥塞时降低低优先级流的权重
            if flow.priority < Priority.HIGH:
                flow.weight *= 0.9
        else:
            # 非拥塞时恢复权重
            flow.weight = flow.original_weight

    # 重新分配带宽
    return allocate_bandwidth(flows, network_state.total_bandwidth)
```

**算法复杂度分析**（2025年11月最新）：

- **时间复杂度**：$O(n)$，其中$n$是流的数量
- **空间复杂度**：$O(n)$，存储每个流的分配信息
- **更新频率**：每$T$时间单位更新一次，$T$通常为10-100ms

**性能指标**（2025年11月最新）：

- **最小带宽保证率**：> 99%（严格保证）
- **资源利用率**：85-95%（剩余带宽充分利用）
- **带宽分配精度**：±2%（动态调整后）
- **准入控制响应时间**：< 10ms（资源预留检查）

**适用场景**：关键业务流量、SLA保证、多租户网络

### 3.2 最大带宽限制

**最大带宽限制（Maximum Bandwidth Limit）**：

```text
为每个流设置最大带宽
  ↓
限制流的最大带宽
  ↓
防止单个流占用过多资源
```

**带宽限制**：

$$
bw_i \leq bw_{max,i}
$$

**最大带宽限制机制**（2025年11月最新）：

最大带宽限制通过流量整形和调度策略实现。

**带宽限制模型**：

$$
\text{Bandwidth}_{actual} = \min(\text{Bandwidth}_{requested}, bw_{max,i})
$$

**限制策略**：

- **硬限制**：严格限制，超过$bw_{max,i}$的流量被丢弃或延迟
- **软限制**：允许短期突发，长期平均不超过$bw_{max,i}$
- **动态限制**：根据网络状况动态调整$bw_{max,i}$

**带宽限制实现**（2025年11月最新）：

使用令牌桶或漏桶算法实现带宽限制：

$$
\text{Transmit}(packet) \iff \text{TokenBucket}(packet) \land \text{Bandwidth}_{current} < bw_{max,i}
$$

**性能指标**（2025年11月最新）：

- **带宽限制精度**：±3%（令牌桶算法）
- **突发容忍度**：支持短期突发（软限制）
- **资源保护**：防止单个流占用过多资源
- **公平性**：保证各流公平共享资源

**适用场景**：多租户网络、流量控制、资源保护

### 3.3 流量整形（2025年新增）

**令牌桶算法（Token Bucket）**：

流量整形通过令牌桶算法限制流量速率，保证QoS。

**令牌桶模型**：

```text
令牌以固定速率生成
  ↓
数据包消耗令牌
  ↓
令牌不足时延迟或丢弃
  ↓
平滑流量突发
```

**令牌桶参数**：

- **CIR（Committed Information Rate）**：承诺信息速率（令牌生成速率）
- **CBS（Committed Burst Size）**：承诺突发大小（桶容量）
- **PIR（Peak Information Rate）**：峰值信息速率（可选）

**流量整形模型**：

$$
\text{Transmit}(packet) = \begin{cases}
\text{Allow} & \text{if } \text{tokens} \geq \text{packet\_size} \\
\text{Delay} & \text{otherwise}
\end{cases}
$$

**令牌更新**：

$$
\text{tokens}(t) = \min(\text{CBS}, \text{tokens}(t-1) + \text{CIR} \times \Delta t)
$$

**双速率令牌桶（Dual Rate Token Bucket）**（2025年11月最新）：

双速率令牌桶使用两个令牌桶：C桶（承诺速率）和P桶（峰值速率）。

**双速率令牌桶模型**：

$$
\text{Transmit}(packet) = \begin{cases}
\text{Allow} & \text{if } \text{tokens}_C \geq \text{packet\_size} \land \text{tokens}_P \geq \text{packet\_size} \\
\text{Delay} & \text{if } \text{tokens}_C < \text{packet\_size} \land \text{tokens}_P \geq \text{packet\_size} \\
\text{Drop} & \text{if } \text{tokens}_P < \text{packet\_size}
\end{cases}
$$

其中：

- $\text{tokens}_C$：C桶令牌数（承诺速率）
- $\text{tokens}_P$：P桶令牌数（峰值速率）

**漏桶算法（Leaky Bucket）**（2025年11月最新）：

漏桶算法以固定速率输出流量，平滑突发流量。

**漏桶模型**：

$$
\text{OutputRate}(t) = \min(\text{InputRate}(t), \text{LeakRate})
$$

**漏桶 vs 令牌桶对比**（2025年11月最新）：

| **特性** | **令牌桶** | **漏桶** |
|---------|----------|---------|
| **突发容忍** | 支持（CBS） | 不支持 |
| **输出速率** | 可变（≤CIR） | 固定 |
| **延迟** | 低（有令牌时） | 高（平滑输出） |
| **适用场景** | 允许突发的流量 | 需要平滑输出的流量 |

**流量整形性能指标**（2025年11月最新）：

- **速率控制精度**：±5% → ±3%（双速率令牌桶优化后）
- **突发容忍度**：支持CBS大小的突发（令牌桶）
- **延迟增加**：< 10ms → < 5ms（优化后）
- **平滑度**：输出速率波动 < 5%（漏桶算法）

**流量整形应用场景**（2025年11月最新）：

- **企业网络**：限制用户带宽，保证关键业务
- **云服务**：限制租户带宽，防止资源滥用
- **CDN**：平滑流量突发，保护后端服务器
- **5G网络**：流量整形保证网络切片QoS

## 4 延迟保证调度

### 4.1 有界延迟调度

**有界延迟调度（Bounded Delay Scheduling）**：

**延迟保证**：

$$
d_i \leq D_i
$$

其中$D_i$是流$i$的最大延迟要求。

**调度算法**：

- **EDF（Earliest Deadline First）**：最早截止时间优先
  - 延迟保证：$d_i \leq D_i$（严格保证）
  - 适用场景：硬实时流量
  - 复杂度：$O(n \log n)$

- **WFQ（Weighted Fair Queuing）**：加权公平队列（有界延迟）
  - 延迟保证：$d_i \leq \frac{L_{max}}{r_i} + \frac{\sum_{j \neq i} L_j}{C}$
  - 其中$L_{max}$是最大包长度，$r_i$是流的权重，$C$是链路容量
  - 适用场景：QoS网络
  - 复杂度：$O(\log n)$

- **Virtual Clock**：虚拟时钟算法
  - 延迟保证：基于虚拟时钟的延迟上界
  - 适用场景：实时流量调度
  - 复杂度：$O(\log n)$

**延迟保证调度模型**：

$$
\text{Schedule}(packet) = \arg\min_{queue} \left\{ \text{Deadline}(packet), \text{Priority}(packet), \text{Delay}(queue) \right\}
$$

**延迟约束检查**：

$$
\text{Admit}(packet) \iff \text{CurrentDelay} + \text{EstimatedDelay} \leq D_{max}
$$

### 4.2 抖动控制

**抖动（Jitter）**：延迟变化

**抖动控制**：

```text
平滑延迟变化
  ↓
使用缓冲区吸收抖动
  ↓
保证延迟一致性
```

**抖动度量**：

$$
\text{jitter} = \max_i d_i - \min_j d_j
$$

**抖动控制机制**（2025年11月最新）：

抖动控制通过缓冲区管理和调度策略，平滑延迟变化，保证延迟一致性。

**抖动控制模型**：

$$
\text{JitterBuffer}(t) = \max(0, \text{JitterBuffer}(t-1) + \text{ArrivalDelay}(t) - \text{TargetDelay})
$$

**抖动控制策略**：

- **自适应缓冲区**：根据抖动情况动态调整缓冲区大小
- **延迟平滑**：使用移动平均平滑延迟变化
- **优先级调度**：抖动敏感流优先调度
- **预测性调整**：基于历史数据预测抖动，提前调整

**抖动控制性能指标**（2025年11月最新）：

- **抖动减少**：>70%（抖动控制后）
- **延迟一致性**：>95%（抖动控制后）
- **缓冲区利用率**：>80%（自适应缓冲区后）

## 5 丢包控制

### 5.1 主动队列管理（AQM）

**AQM（Active Queue Management）**：

**RED（Random Early Detection）**：

```text
监控队列长度
  ↓
队列长度 > 阈值时随机丢包
  ↓
避免队列溢出
  ↓
提前通知拥塞
```

**RED算法模型**：

$$
p_{drop} = \begin{cases}
0 & \text{if } q_{avg} < q_{min} \\
\frac{q_{avg} - q_{min}}{q_{max} - q_{min}} \times p_{max} & \text{if } q_{min} \leq q_{avg} < q_{max} \\
1 & \text{if } q_{avg} \geq q_{max}
\end{cases}
$$

其中：

- $q_{avg}$：平均队列长度
- $q_{min}$：最小阈值
- $q_{max}$：最大阈值
- $p_{max}$：最大丢包概率

**RED参数调优**（2025年11月最新）：

- **自适应阈值**：根据流量特征动态调整$q_{min}$和$q_{max}$
- **加权平均**：使用指数加权移动平均（EWMA）计算$q_{avg}$
- **丢包概率平滑**：平滑丢包概率，避免突发丢包

**CoDel（Controlled Delay）**：

```text
监控队列延迟
  ↓
延迟 > 阈值时丢包
  ↓
控制队列延迟
  ↓
避免缓冲区膨胀
```

**CoDel算法模型**：

$$
\text{Drop}(packet) \iff \text{SojournTime} > \text{TargetDelay} \land \text{Interval} > \text{IntervalMin}
$$

其中：

- $\text{SojournTime}$：包在队列中的停留时间
- $\text{TargetDelay}$：目标延迟（默认5ms）
- $\text{Interval}$：上次丢包后的时间间隔
- $\text{IntervalMin}$：最小间隔（默认100ms）

**CoDel优势**（2025年11月最新）：

- **延迟控制**：直接控制队列延迟，而非队列长度
- **自适应**：自动适应流量特征，无需手动调优
- **低延迟**：保持低延迟，避免缓冲区膨胀
- **公平性**：保证各流公平丢包

**AQM算法对比**（2025年11月最新）：

| **算法** | **控制指标** | **复杂度** | **适用场景** | **优势** |
|---------|------------|-----------|------------|---------|
| **RED** | 队列长度 | $O(1)$ | 通用网络 | 简单高效 |
| **CoDel** | 队列延迟 | $O(1)$ | 低延迟网络 | 延迟保证 |
| **PIE** | 队列延迟+长度 | $O(1)$ | 混合场景 | 平衡性能 |
| **FQ-CoDel** | 队列延迟+公平性 | $O(\log n)$ | QoS网络 | 公平+延迟 |

### 5.2 丢包率控制

**丢包率保证**：

$$
\text{loss\_rate}_i \leq L_i
$$

其中$L_i$是流$i$的最大丢包率。

**控制策略**：

- **优先级丢包**：低优先级流优先丢包
- **加权丢包**：按权重比例丢包
- **公平丢包**：各流公平丢包

**丢包率控制模型**（2025年11月最新）：

$$
\text{LossRate}_i = \frac{\text{DroppedPackets}_i}{\text{TotalPackets}_i} \leq L_i
$$

**优先级丢包策略**：

$$
\text{Drop}(packet_i) \iff \text{Priority}(packet_i) < \text{PriorityThreshold} \land \text{LossRate}_i < L_i
$$

**加权丢包策略**：

$$
\text{DropProbability}_i = \frac{w_i}{\sum_j w_j} \times \text{TotalDropProbability}
$$

其中$w_i$是流$i$的权重。

**丢包率控制性能指标**（2025年11月最新）：

- **丢包率控制精度**：±2%（优先级丢包后）
- **SLA满足率**：>98%（丢包率控制后）
- **公平性**：>95%（加权丢包后）

## 6 形式化模型

### 6.1 QoS调度问题定义

$$
\text{QoS调度问题} = (F, B, Q, C, O)
$$

其中：

- $F = \{f_1, f_2, \ldots, f_n\}$：流集合
- $B$：带宽约束（$B_{total}$）
- $Q$：QoS要求集合
  - 带宽要求：$bw_{min,i} \leq bw_i \leq bw_{max,i}$
  - 延迟要求：$d_i \leq D_i$
  - 丢包率要求：$loss_i \leq L_i$
- $C$：约束条件
  - 带宽约束：$\sum_i bw_i \leq B_{total}$
  - 资源限制：$\sum_i r_i \leq R_{total}$
- $O$：优化目标
  - 最大化QoS满足率：$\max \sum_i \mathbb{1}(QoS_i \text{ satisfied})$
  - 最小化延迟：$\min \sum_i d_i$
  - 最大化吞吐量：$\max \sum_i bw_i$

### 6.2 调度算法复杂度

| **算法** | **时间复杂度** | **QoS保证** | **公平性** | **适用场景** |
|---------|--------------|------------|-----------|------------|
| **严格优先级** | $O(1)$ | 高优先级最优 | 无 | 关键流量 |
| **加权优先级** | $O(\log n)$ | 带宽保证 | 中 | 通用场景 |
| **WFQ** | $O(\log n)$ | 带宽+延迟 | 高 | QoS网络 |
| **EDF** | $O(n \log n)$ | 延迟保证 | 中 | 实时流量 |

### 6.3 定理：QoS可调度性

**定理15.2（QoS可调度性）**：

对于流集合$F = \{f_1, f_2, \ldots, f_n\}$，如果满足：

**条件1（带宽约束）**：

$$
\sum_i \frac{bw_{min,i}}{B_{total}} \leq 1
$$

**条件2（延迟约束）**：

$$
\sum_i \frac{bw_i}{B_{total}} \times \frac{D_i}{d_i} \leq 1
$$

则存在调度算法满足所有QoS要求（带宽、延迟、丢包率）。

**证明**：

**步骤1：带宽保证**

由条件1，所有流的最小带宽需求之和不超过总带宽：

$$
\sum_i bw_{min,i} \leq B_{total}
$$

因此，可以使用加权公平队列（WFQ）算法，为每个流$f_i$分配权重：

$$
w_i = \frac{bw_{min,i}}{B_{total}}
$$

WFQ保证每个流获得的最小带宽为：

$$
bw_i \geq \frac{w_i}{\sum_j w_j} \times B_{total} = bw_{min,i}
$$

**步骤2：延迟保证**

由条件2，延迟约束满足：

$$
\sum_i \frac{bw_i}{B_{total}} \times \frac{D_i}{d_i} \leq 1
$$

对于WFQ算法，流的延迟上界为：

$$
d_i \leq \frac{L_{max}}{r_i} + \frac{\sum_{j \neq i} L_j}{C}
$$

其中：

- $L_{max}$：最大包长度
- $r_i$：流$i$的权重（服务速率）
- $C$：链路容量

当$r_i = bw_{min,i}$时，延迟上界为：

$$
d_i \leq \frac{L_{max}}{bw_{min,i}} + \frac{\sum_{j \neq i} L_j}{B_{total}}
$$

如果满足条件2，则可以保证$d_i \leq D_i$。

**步骤3：丢包率保证**

使用主动队列管理（AQM）算法（如RED或CoDel），根据优先级丢包策略，保证高优先级流的丢包率低于阈值$L_i$。

**步骤4：构造调度算法**

综合以上步骤，可以构造满足所有QoS要求的调度算法：

1. **带宽分配**：使用WFQ算法，权重$w_i = \frac{bw_{min,i}}{B_{total}}$
2. **延迟保证**：WFQ提供有界延迟，满足$d_i \leq D_i$
3. **丢包控制**：使用AQM算法，保证$loss_i \leq L_i$

因此，存在调度算法满足所有QoS要求。∎

**推论15.2.1（QoS可调度性充分条件）**：

如果流集合$F$满足：

$$
\sum_i \frac{bw_{min,i}}{B_{total}} \leq \alpha \quad \text{且} \quad \sum_i \frac{bw_i}{B_{total}} \times \frac{D_i}{d_i} \leq \beta
$$

其中$\alpha + \beta \leq 1$，则存在调度算法满足所有QoS要求。

**证明**：由定理15.2，当$\alpha = \beta = 0.5$时，条件满足。∎

## 7 跨领域洞察

### 7.1 QoS调度与资源预留

**资源预留模型**：

```text
流请求资源
  ↓
检查资源可用性
  ↓
预留资源
  ↓
保证QoS
```

**预留策略**：

- **静态预留**：长期预留资源
- **动态预留**：按需预留资源
- **软预留**：尽力保证，不严格预留

**资源预留模型**（2025年11月最新）：

$$
\text{Reserve}(flow_i) \iff \text{Available}(resource) \geq \text{Require}(flow_i) \land \text{Admit}(flow_i)
$$

其中：

- $\text{Available}(resource)$：可用资源
- $\text{Require}(flow_i)$：流$i$的资源需求
- $\text{Admit}(flow_i)$：流$i$的准入决策

**资源预留效率**：

$$
\text{Efficiency} = \frac{\text{Utilized}(resource)}{\text{Reserved}(resource)} \times 100\%
$$

**预留策略对比**（2025年11月最新）：

| **策略** | **资源利用率** | **QoS保证** | **灵活性** | **适用场景** |
|---------|------------|------------|-----------|------------|
| **静态预留** | 60-70% | 高 | 低 | 长期稳定流量 |
| **动态预留** | 80-90% | 中高 | 高 | 变化流量 |
| **软预留** | 90-95% | 中 | 高 | 尽力而为流量 |

**关键洞察**：**资源预留是QoS保证的基础**，但可能导致资源利用率低。动态预留在QoS保证和资源利用率之间取得平衡。

### 7.2 QoS与网络拥塞

**拥塞时的QoS**：

```text
网络拥塞
  ↓
资源不足
  ↓
QoS无法保证
  ↓
需要拥塞控制
```

**QoS与拥塞控制协同**：

- **优先级丢包**：拥塞时优先保证高优先级流
- **带宽调整**：动态调整带宽分配
- **延迟容忍**：允许低优先级流延迟增加

**拥塞时的QoS调整模型**（2025年11月最新）：

$$
\text{QoS}_{adjusted}(flow_i) = \begin{cases}
\text{QoS}_{original}(flow_i) & \text{if } \text{CongestionLevel} < \text{Threshold} \\
\text{QoS}_{degraded}(flow_i) & \text{if } \text{Priority}(flow_i) < \text{PriorityThreshold} \\
\text{QoS}_{guaranteed}(flow_i) & \text{if } \text{Priority}(flow_i) \geq \text{PriorityThreshold}
\end{cases}
$$

**拥塞检测**：

$$
\text{CongestionLevel} = \frac{\text{QueueLength}}{\text{QueueCapacity}} \times \alpha + \frac{\text{LossRate}}{\text{MaxLossRate}} \times \beta
$$

其中$\alpha + \beta = 1$，$\alpha = 0.6$，$\beta = 0.4$。

**QoS降级策略**（2025年11月最新）：

当检测到拥塞时，根据流的优先级进行QoS降级：

$$
\text{Bandwidth}_{adjusted} = \text{Bandwidth}_{original} \times (1 - \text{CongestionLevel} \times \text{PriorityFactor})
$$

其中$\text{PriorityFactor}$根据流的优先级确定：

- 高优先级流：$\text{PriorityFactor} = 0.1$（轻微降级）
- 中优先级流：$\text{PriorityFactor} = 0.5$（中等降级）
- 低优先级流：$\text{PriorityFactor} = 0.9$（大幅降级）

**关键洞察**：**QoS保证受网络拥塞影响**，需要与拥塞控制协同。通过优先级丢包和动态带宽调整，可以在拥塞时保证关键流的QoS。

### 7.3 多租户QoS隔离

**多租户场景**：

```text
多个租户共享网络
  ↓
每个租户有QoS要求
  ↓
需要租户间隔离
  ↓
保证各租户QoS
```

**隔离机制**：

- **资源隔离**：为每个租户预留资源
- **优先级隔离**：租户间优先级隔离
- **带宽隔离**：租户间带宽隔离

**多租户QoS隔离模型**（2025年11月最新）：

$$
\text{QoS}_{tenant_i} = f(\text{Reserved}(tenant_i), \text{Isolation}(tenant_i), \text{Shared}(resource))
$$

**资源隔离模型**：

$$
\text{Bandwidth}_{tenant_i} = \min\left(\text{Reserved}_{tenant_i}, \frac{\text{Weight}_{tenant_i}}{\sum_j \text{Weight}_{tenant_j}} \times \text{TotalBandwidth}\right)
$$

**隔离度度量**（2025年11月最新）：

$$
\text{IsolationDegree} = 1 - \frac{\text{Interference}(tenant_i, tenant_j)}{\text{TotalResource}}
$$

其中：

- $\text{Interference}(tenant_i, tenant_j)$：租户$i$和$j$之间的资源干扰
- $\text{TotalResource}$：总资源

**隔离策略对比**（2025年11月最新）：

| **策略** | **隔离度** | **资源利用率** | **复杂度** | **适用场景** |
|---------|----------|------------|-----------|------------|
| **硬隔离** | >99% | 60-70% | 高 | 关键租户 |
| **软隔离** | 85-95% | 80-90% | 中 | 一般租户 |
| **共享隔离** | 70-85% | 90-95% | 低 | 非关键租户 |

**多租户QoS保证**（2025年11月最新）：

$$
\text{QoS}_{guaranteed}(tenant_i) \geq \text{QoS}_{SLA}(tenant_i) \quad \text{s.t.} \quad \text{IsolationDegree} \geq \text{Threshold}
$$

**关键洞察**：**多租户QoS需要隔离机制**，防止租户间相互影响。通过资源隔离、优先级隔离和带宽隔离，可以保证各租户的QoS要求，同时提高资源利用率。

\text{QoS}_{slice}(t) = \arg\min_{\text{Resource}} \left\{ \text{Latency}(t) \leq L_{slice}, \text{Bandwidth}(t) \geq B_{slice}, \text{Isolation}(t) \geq I_{slice} \right\}
$$

其中：

- $L_{slice}$：切片延迟要求（eMBB: <50ms, uRLLC: <1ms, mMTC: <500ms）
- $B_{slice}$：切片最小带宽保证
- $I_{slice}$：切片间隔离度要求（>95%）

**5G切片类型与QoS要求**：

| **切片类型** | **延迟要求** | **带宽要求** | **可靠性** | **典型应用** |
|------------|------------|------------|----------|------------|
| **eMBB** | < 50ms | 100Mbps-1Gbps | 99.9% | 高清视频、AR/VR |
| **uRLLC** | < 1ms | 1-10Mbps | 99.999% | 工业控制、自动驾驶 |
| **mMTC** | < 500ms | 1-100Kbps | 99.9% | 物联网传感器 |

### 9.2 存储QoS调度（2025年11月）

**IO优先级调度**：

$$
\text{QoS}_{IO}(task) = f(\text{IOPattern}, \text{StorageState}, \text{HistoricalQoS}, \text{MLPrediction}, \text{WorkloadType})
$$

**核心机制**：

- **IO模式识别**：使用深度学习自动识别IO模式（顺序/随机、读/写、大小）
- **延迟预测**：预测IO延迟，提前调整QoS策略

**存储QoS调度模型**：

$$
\begin{aligned}
\text{IOPS}_i &= \min\left(\text{IOPS}_{max,i}, \frac{w_i}{\sum_j w_j} \times \text{TotalIOPS}\right) \\
\text{Bandwidth}_i &= \min\left(\text{BW}_{max,i}, \frac{w_i}{\sum_j w_j} \times \text{TotalBW}\right) \\
\text{Latency}_i &\leq L_i \quad \text{(SLA约束)}
\end{aligned}
$$

其中：

- $w_i$：任务$i$的权重
- $\text{IOPS}_{max,i}$：任务$i$的最大IOPS限制
- $\text{BW}_{max,i}$：任务$i$的最大带宽限制
- $L_i$：任务$i$的延迟SLA要求

### 10.1 QoS调度算法对比

| **算法** | **带宽保证** | **延迟保证** | **公平性** | **复杂度** | **适用场景** | **延迟上界** | **资源利用率** |

| **严格优先级** | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐ | ⭐⭐⭐⭐⭐ | 关键流量 | $O(1)$ | 85-95% |
| **加权优先级** | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ | 通用场景 | $O(\frac{L}{r})$ | 80-90% |
| **WFQ** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | QoS网络 | $\frac{L_{max}}{r_i} + \frac{\sum L_j}{C}$ | 90-95% |

**算法详细说明**（2025年11月最新）：

**严格优先级调度**：

- **延迟上界**：高优先级流延迟接近0，低优先级流可能无限延迟
- **资源利用率**：85-95%（高优先级流占用大部分资源）
- **适用场景**：关键业务流量、实时流量、语音/视频流

**加权优先级调度**：

- **延迟上界**：$d_i \leq \frac{L}{r_i}$，其中$L$是包长度，$r_i$是流的服务速率
- **资源利用率**：80-90%（权重分配影响利用率）
- **适用场景**：通用场景、多租户网络、混合流量

**WFQ（加权公平队列）**：

- **延迟上界**：$d_i \leq \frac{L_{max}}{r_i} + \frac{\sum_{j \neq i} L_j}{C}$，提供有界延迟保证
- **资源利用率**：90-95%（公平分配，利用率高）
- **适用场景**：QoS网络、企业网络、数据中心网络

**EDF（最早截止时间优先）**：

- **延迟上界**：严格保证$d_i \leq D_i$（截止时间）
- **资源利用率**：70-85%（严格延迟保证限制利用率）
- **适用场景**：实时流量、硬实时系统、工业控制

**令牌桶**：

- **延迟上界**：$d_i \leq \frac{CBS}{CIR}$，其中$CBS$是承诺突发大小，$CIR$是承诺信息速率
- **资源利用率**：85-95%（流量整形，利用率高）
- **适用场景**：流量整形、带宽限制、QoS保证

   - 监控带宽使用情况，及时调整

3. **流量整形**：
   - 使用令牌桶算法平滑流量
   - 配置合理的CIR和CBS参数
   - 根据业务需求选择单速率或双速率令牌桶

**延迟保证最佳实践**：

1. **延迟要求分析**：
   - 明确各业务的延迟SLA要求
   - 区分硬实时和软实时需求
   - 设置合理的延迟上界

2. **调度算法选择**：
   - 硬实时：使用EDF算法
   - 软实时：使用WFQ算法
   - 一般业务：使用加权优先级

3. **抖动控制**：
   - 使用自适应缓冲区平滑抖动
   - 监控抖动指标，及时调整
   - 优先保证抖动敏感流的QoS

**丢包控制最佳实践**：

1. **AQM算法选择**：
   - 低延迟网络：使用CoDel算法
   - 通用网络：使用RED算法
   - QoS网络：使用FQ-CoDel算法

2. **丢包策略**：
   - 高优先级流：避免丢包
   - 中优先级流：适度丢包
   - 低优先级流：优先丢包

3. **丢包率监控**：
   - 实时监控各流的丢包率
   - 设置丢包率告警阈值
   - 及时调整QoS策略

```python
# 伪代码：链路质量感知QoS调度
def select_link(application_qos, link_states):
    """
    根据应用QoS和链路状态选择最优链路
    """
    candidates = []
    for link in link_states:
        if link.available_bandwidth >= application_qos.min_bw:
            if link.latency <= application_qos.max_latency:
                if link.loss_rate <= application_qos.max_loss:
                    score = calculate_score(link, application_qos)
                    candidates.append((link, score))

    return max(candidates, key=lambda x: x[1])[0]

def calculate_score(link, qos):
    """
    计算链路适配分数
    """
    bw_score = min(link.available_bandwidth / qos.req_bw, 1.0)
    latency_score = 1.0 - (link.latency / qos.max_latency)
    loss_score = 1.0 - (link.loss_rate / qos.max_loss)
    stability_score = link.stability_factor

    return 0.3*bw_score + 0.3*latency_score + 0.2*loss_score + 0.2*stability_score
```

```c
#include <linux/bpf.h>
#include <bpf/bpf_helpers.h>

struct qos_config {
    __u64 rate_limit;      // 带宽限制 (bytes/s)
    __u64 burst_size;      // 突发大小
    __u32 priority;        // 优先级 (0-7)
};

struct {
    __uint(type, BPF_MAP_TYPE_HASH);
    __type(key, __u32);    // Pod IP
    __type(value, struct qos_config);
    __uint(max_entries, 10000);
} qos_map SEC(".maps");

SEC("tc/egress")
int tc_qos_limit(struct __sk_buff *skb) {
    __u32 ip = skb->local_ipv4;
    struct qos_config *config = bpf_map_lookup_elem(&qos_map, &ip);

    if (!config)
        return TC_ACT_OK;  // 无配置，正常转发

    // 令牌桶算法实现
    if (!token_bucket_check(config, skb->len)) {
        if (config->priority == 0)
            return TC_ACT_SHOT;  // 低优先级，丢弃
        else
            return TC_ACT_STOLEN; // 高优先级，延迟发送
    }

    // 设置DSCP标记
    set_dscp(skb, config->priority);

    return TC_ACT_OK;
}
```

### 形式化定义补充

#### 通用QoS参数形式化定义

**定义15.1（QoS流量规范 - TSpec）**：

根据RFC 2215，流量规范TSpec定义为五元组：

$$
\text{TSpec} = (r, b, p, m, M)
$$

其中：

- $r$：令牌桶速率（token bucket rate），单位 bytes/second
- $b$：令牌桶深度（token bucket size），单位 bytes
- $p$：峰值速率（peak rate），单位 bytes/second
- $m$：最小包大小（minimum policed unit），单位 bytes
- $M$：最大包大小（maximum packet size），单位 bytes

**定义15.2（QoS服务规范 - RSpec）**：

资源预留规范RSpec定义为：

$$
\text{RSpec} = (R, S)
$$

其中：

- $R$：预留速率（reservation rate），单位 bytes/second
- $S$：松弛项（slack term），单位 microseconds

**定义15.3（端到端延迟上界）**：

根据RFC 2212，对于Guaranteed Service，端到端延迟上界为：

$$
D_{end-to-end} \leq \frac{b}{R} + \sum_{i=1}^{n} \frac{M}{C_i} + \sum_{i=1}^{n} \frac{L_i}{R} + S
$$

其中：

- $b$：令牌桶深度
- $R$：预留速率
- $M$：最大包大小
- $C_i$：第$i$跳链路容量
- $L_i$：第$i$跳的最大包大小
- $S$：松弛项
- $n$：跳数

#### 加权公平队列（WFQ）算法

**算法15.1：加权公平队列（WFQ）**

```python
class WeightedFairQueuing:
    """
    加权公平队列（WFQ）算法实现
    参考：Demers, Keshav, Shenker (1989) 与 Kurose-Ross 7.5.4节
    """

    def __init__(self, flows, total_bandwidth):
        """
        初始化WFQ调度器

        参数：
            flows: 流列表，每个流包含 weight（权重）属性
            total_bandwidth: 总带宽（bytes/second）

        时间复杂度：O(n)，其中 n 为流数量
        空间复杂度：O(n)
        """
        self.flows = flows
        self.total_bandwidth = total_bandwidth
        self.virtual_time = 0.0  # 虚拟时间
        self.flow_states = {}

        # 初始化每个流的状态
        for flow in flows:
            self.flow_states[flow.id] = {
                'weight': flow.weight,
                'finish_time': 0.0,  # 上次完成时间
                'queue': [],         # 该流的队列
                'arrival_rate': 0.0  # 到达速率
            }

    def enqueue(self, packet, flow_id):
        """
        将包加入对应流的队列

        参数：
            packet: 数据包（包含 length 属性）
            flow_id: 流标识符

        时间复杂度：O(1)
        """
        state = self.flow_states[flow_id]
        state['queue'].append(packet)

        # 计算包的完成时间（Finish Number）
        # F_i^k = max(F_i^{k-1}, V(t)) + L_i^k / w_i
        start_time = max(state['finish_time'], self.virtual_time)
        packet.finish_time = start_time + packet.length / state['weight']
        state['finish_time'] = packet.finish_time

        # 更新到达速率估计
        state['arrival_rate'] = self._update_arrival_rate(flow_id, packet)

    def dequeue(self):
        """
        出队一个包（选择最小完成时间的包）

        返回：
            packet: 被调度的数据包

        时间复杂度：O(n)，使用优先队列可优化到 O(log n)
        空间复杂度：O(1)
        """
        # 找到具有最小完成时间的非空队列
        min_finish_time = float('inf')
        selected_flow = None

        for flow_id, state in self.flow_states.items():
            if state['queue']:
                packet = state['queue'][0]
                if packet.finish_time < min_finish_time:
                    min_finish_time = packet.finish_time
                    selected_flow = flow_id

        if selected_flow is None:
            return None  # 所有队列为空

        # 出队
        packet = self.flow_states[selected_flow]['queue'].pop(0)

        # 更新虚拟时间
        # V(t) 增加量为 包长度 / (总带宽 * 权重归一化因子)
        self.virtual_time += packet.length / self.total_bandwidth

        return packet

    def _update_arrival_rate(self, flow_id, packet):
        """更新到达速率估计（指数加权移动平均）"""
        state = self.flow_states[flow_id]
        alpha = 0.7  # EWMA平滑因子
        current_rate = packet.length / (time.now() - state['last_arrival'])
        state['last_arrival'] = time.now()
        return alpha * current_rate + (1 - alpha) * state['arrival_rate']

# WFQ性能保证（理论分析）
def wfq_delay_bound(flow_weight, max_packet_size, total_bandwidth, n_flows):
    """
    计算WFQ的延迟上界

    根据Kurose-Ross 7.5.4节公式：
    d_i <= L_max / r_i + (n-1) * L_max / C

    其中：
    - r_i = (w_i / sum(w_j)) * C 是流i的保证速率
    - L_max 是最大包长度
    - C 是链路容量
    - n 是流数量

    时间复杂度：O(1)
    """
    w_i = flow_weight
    W = sum(f.weight for f in flows)  # 总权重
    r_i = (w_i / W) * total_bandwidth

    delay_bound = max_packet_size / r_i + (n_flows - 1) * max_packet_size / total_bandwidth
    return delay_bound

# WFQ公平性保证
def wfq_fairness_guarantee(flow_i, flow_j, time_window):
    """
    WFQ公平性保证：在任意时间窗口内，两个流的带宽分配与权重成正比

    |S_i(t1, t2) / w_i - S_j(t1, t2) / w_j| <= L_max / w_i + L_max / w_j

    其中 S_i(t1, t2) 是流i在[t1, t2]内获得的带宽
    """
    L_max = max_packet_size
    fairness_error = L_max / flow_i.weight + L_max / flow_j.weight
    return fairness_error
```

#### 最早截止时间优先（EDF）算法

**算法15.2：最早截止时间优先（EDF）**

```python
import heapq

class EDFScheduler:
    """
    最早截止时间优先（EDF）调度器
    参考：Liu & Layland (1973) 实时调度理论

    EDF是最优的动态优先级实时调度算法，在利用率 U <= 1 时保证可调度性
    """

    def __init__(self):
        """
        初始化EDF调度器

        时间复杂度：O(1)
        空间复杂度：O(1)
        """
        self.packet_heap = []  # 最小堆，按截止时间排序
        self.packet_counter = 0  # 用于打破平局

    def enqueue(self, packet, deadline):
        """
        将包加入EDF队列

        参数：
            packet: 数据包
            deadline: 绝对截止时间（微秒）

        时间复杂度：O(log n)
        空间复杂度：O(1)
        """
        # 使用 (截止时间, 计数器, 包) 作为堆元素
        # 计数器确保相同截止时间时的FIFO顺序
        heapq.heappush(self.packet_heap, (deadline, self.packet_counter, packet))
        self.packet_counter += 1

    def dequeue(self):
        """
        出队最早截止时间的包

        返回：
            packet: 被调度的数据包，或 None（如果队列为空）

        时间复杂度：O(log n)
        空间复杂度：O(1)
        """
        if not self.packet_heap:
            return None

        deadline, _, packet = heapq.heappop(self.packet_heap)
        return packet

    def peek_next_deadline(self):
        """查看下一个包的截止时间"""
        if self.packet_heap:
            return self.packet_heap[0][0]
        return None

    @staticmethod
    def schedulability_test(flows):
        """
        EDF可调度性测试（Liu & Layland 利用率测试）

        参数：
            flows: 流列表，每个流有 period（周期）和 exec_time（执行时间/包传输时间）

        返回：
            bool: 是否可调度

        时间复杂度：O(n)
        空间复杂度：O(1)

        理论依据：
        对于周期任务，如果总利用率 U = sum(C_i / T_i) <= 1，则EDF可调度
        其中 C_i 是计算时间，T_i 是周期
        """
        total_utilization = sum(
            flow.exec_time / flow.period for flow in flows
        )
        return total_utilization <= 1.0

    @staticmethod
    def density_test(flows):
        """
        密度测试（考虑截止时间的更精确测试）

        对于非周期或截止时间不等于周期的任务：
        如果密度 D = sum(C_i / D_i) <= 1，则可调度
        其中 D_i 是相对截止时间
        """
        total_density = sum(
            flow.exec_time / flow.deadline for flow in flows
        )
        return total_density <= 1.0

# EDF延迟保证计算
def edf_delay_guarantee(flow, link_capacity):
    """
    EDF的延迟保证分析

    对于Guaranteed Service（RFC 2212）：
    - 延迟上界 = 排队延迟 + 传输延迟
    - 排队延迟 <= D_i - L_i/C （截止时间约束）
    - 传输延迟 = L_i / C

    因此总延迟 <= D_i （截止时间）

    参数：
        flow: 流对象，包含 deadline（截止时间）和 packet_size（包大小）
        link_capacity: 链路容量

    返回：
        max_delay: 最大延迟保证
    """
    transmission_delay = flow.packet_size / link_capacity
    queuing_delay = flow.deadline - transmission_delay

    return {
        'max_delay': flow.deadline,
        'transmission_delay': transmission_delay,
        'queuing_delay_bound': queuing_delay,
        'guarantee': 'hard' if queuing_delay > 0 else 'none'
    }
```

#### 令牌桶算法（Token Bucket）

**算法15.3：令牌桶流量整形**

```python
import time

class TokenBucket:
    """
    令牌桶算法实现
    参考：RFC 2697（单速率三色标记器）、RFC 2698（双速率三色标记器）
    """

    def __init__(self, cir, cbs, pbs=0, pir=0):
        """
        初始化令牌桶

        参数：
            cir: 承诺信息速率（Committed Information Rate），bytes/second
            cbs: 承诺突发大小（Committed Burst Size），bytes
            pbs: 峰值突发大小（Peak Burst Size），bytes（双速率时使用）
            pir: 峰值信息速率（Peak Information Rate），bytes/second（双速率时使用）

        时间复杂度：O(1)
        空间复杂度：O(1)
        """
        self.cir = cir  # 承诺速率
        self.cbs = cbs  # 承诺桶容量
        self.pir = pir  # 峰值速率（双速率）
        self.pbs = pbs  # 峰值桶容量（双速率）

        # 当前令牌数
        self.tc = cbs  # 承诺桶令牌
        self.tp = pbs if pbs > 0 else 0  # 峰值桶令牌（双速率）

        self.last_update = time.time()

    def _update_tokens(self):
        """
        根据时间更新令牌数量

        时间复杂度：O(1)
        """
        now = time.time()
        elapsed = now - self.last_update
        self.last_update = now

        # 增加令牌（承诺桶）
        self.tc = min(self.cbs, self.tc + self.cir * elapsed)

        # 增加令牌（峰值桶，双速率时）
        if self.pir > 0:
            self.tp = min(self.pbs, self.tp + self.pir * elapsed)

    def check_single_rate(self, packet_size):
        """
        单速率三色标记器（RFC 2697）

        返回：
            color: 包颜色 - 'green', 'yellow', 'red'

        算法：
        - Green: 令牌充足，不超出CBS
        - Yellow: 令牌不足，但可借用
        - Red: 完全超出限制

        时间复杂度：O(1)
        空间复杂度：O(1)
        """
        self._update_tokens()

        if packet_size <= self.tc:
            # 绿色：在承诺范围内
            self.tc -= packet_size
            return 'green'
        elif packet_size <= self.tp:
            # 黄色：超出承诺但在峰值内
            self.tp -= packet_size
            return 'yellow'
        else:
            # 红色：超出峰值
            return 'red'

    def check_dual_rate(self, packet_size):
        """
        双速率三色标记器（RFC 2698）

        与单速率的区别：
        - PIR和CIR独立
        - 两个桶独立更新

        返回：
            color: 'green', 'yellow', 'red'
        """
        self._update_tokens()

        if packet_size <= self.tc:
            # 绿色：符合CIR和PIR
            self.tc -= packet_size
            self.tp -= packet_size
            return 'green'
        elif packet_size <= self.tp:
            # 黄色：超出CIR但符合PIR
            self.tp -= packet_size
            return 'yellow'
        else:
            # 红色：超出PIR
            return 'red'

    def conform(self, packet_size, mode='single'):
        """
        检查包是否符合流量规范（RFC 2697/2698）

        参数：
            packet_size: 包大小
            mode: 'single'（单速率）或 'dual'（双速率）

        返回：
            (conforms, color): (是否符合, 颜色标记)
        """
        if mode == 'single':
            color = self.check_single_rate(packet_size)
        else:
            color = self.check_dual_rate(packet_size)

        # Green和Yellow都符合，Red不符合
        return (color != 'red', color)

# 令牌桶复杂度分析
class TokenBucketComplexity:
    """令牌桶算法复杂度分析"""

    @staticmethod
```

```c
// 1. HTB（Hierarchical Token Bucket）层次化令牌桶
// 文件：net/sched/sch_htb.c

struct htb_class {
    // 速率参数
    u64 rate;           // CIR
    u64 ceil;           // PIR（峰值速率）
    s64 buffer;         // CBS
    s64 cbuffer;        // PBS

    // 令牌桶状态
    s64 tokens;         // 承诺桶令牌
    s64 ctokens;        // 峰值桶令牌
    psched_time_t t_c;  // 上次更新时间

    // 队列
    struct Qdisc *qdisc;

    // 优先级
    int priority;
    int level;

    // 父类（层次结构）
    struct htb_class *parent;
};

// HTB入队函数复杂度：O(log n)，n为类数量
static int htb_enqueue(struct sk_buff *skb, struct Qdisc *sch,
                       struct sk_buff **to_free)
{
    struct htb_sched *q = qdisc_priv(sch);
    struct htb_class *cl;

    // 1. 分类（查找对应类）- O(log n)
    cl = htb_classify(skb, sch, &q->cl_hash);
    if (!cl)
        return qdisc_drop(skb, sch, to_free);

    // 2. 检查令牌桶 - O(1)
    if (htb_may_xmit(cl, skb)) {
        // 3. 入队或发送
        return qdisc_enqueue(skb, cl->qdisc, to_free);
    }

    // 4. 不可发送，处理延迟
    htb_charge_class(sch, cl, skb);
    return NET_XMIT_SUCCESS;
}

// 2. FQ-CoDel（Fair Queueing + Controlled Delay）
// 文件：net/sched/sch_fq_codel.c

struct fq_codel_flow {
    struct sk_buff *head;   // 队列头
    struct sk_buff *tail;   // 队列尾
    struct list_head flowchain;
    int deficit;            // DRR赤字计数器
    u32 dropped;            // 丢弃计数
    u32 backlog;            // 队列积压
};

// FQ-CoDel出队复杂度：O(1) 均摊
static struct sk_buff *fq_codel_dequeue(struct Qdisc *sch)
{
    struct fq_codel_sched_data *q = qdisc_priv(sch);
    struct fq_codel_flow *flow;
    struct sk_buff *skb;

    // 1. 从活跃流列表选择（RR轮询）- O(1)
    flow = fq_codel_bucket_get(q);
    if (!flow)
        return NULL;

    // 2. CoDel延迟控制检查 - O(1)
    if (codel_should_drop(&q->cparams, &q->cvars,
                          flow->head, sch, flow->backlog)) {
        // 执行丢包
        fq_codel_drop(sch, flow);
    }

    // 3. DRR调度 - O(1)
    skb = flow->head;
    flow->deficit -= skb->len;

    if (flow->deficit <= 0) {
        flow->deficit += q->quantum;
        list_move_tail(&flow->flowchain, &q->new_flows);
    }

    return skb;
}
```

```c
/* DPDK QoS框架（lib/librte_sched/）
 * 参考：rte_sched.h, rte_sched.c

// DPDK QoS架构层次
// 1. Port（物理端口）
// 2. Subport（子端口）- 如不同租户
// 3. Pipe（管道）- 如不同用户
// 4. Traffic Class（流量类别）- 如语音、视频、数据
// 5. Queue（队列）- 每TC的多个队列

// DPDK QoS调度器配置
struct rte_sched_port_params {
    uint32_t rate;              // 端口速率（bytes/second）
    uint32_t mtu;               // MTU
    uint32_t frame_overhead;    // 帧开销
    uint32_t n_subports_per_port;  // 每端口子端口数
    uint32_t n_pipes_per_subport;  // 每子端口管道数
    uint32_t n_pipes_per_subport_log2;
    uint32_t n_traffic_classes_per_pipe;  // 每管道TC数
    uint32_t n_queue_profiles;
    struct rte_sched_subport_profile_params *subport_profiles;
};

// 子端口参数（令牌桶）
struct rte_sched_subport_params {
    uint64_t tb_rate;           // 令牌桶速率
    uint64_t tb_size;           // 令牌桶大小
    struct rte_sched_pipe_params *pipe_profiles;
};

// DPDK QoS入队函数（时间复杂度：O(1)）
int rte_sched_port_enqueue(struct rte_sched_port *port,
                           struct rte_mbuf **pkts,
                           uint32_t n_pkts)
{
    uint32_t i;

    for (i = 0; i < n_pkts; i++) {
        struct rte_mbuf *pkt = pkts[i];
        uint32_t subport = pkt->subport;
        uint32_t pipe = pkt->pipe;
        uint32_t tc = pkt->traffic_class;
        uint32_t queue = pkt->queue;

        // 1. 确定队列位置 - O(1)
        struct rte_sched_queue *q = &port->queue[subport][pipe][tc][queue];

        // 2. 令牌桶检查 - O(1)
        if (rte_sched_subport_check(port, subport, pkt->pkt_len)) {
            // 3. 入队 - O(1)
            rte_mbuf_enqueue(q, pkt);
        } else {
            // 超出速率限制，丢弃
            rte_pktmbuf_free(pkt);
        }
    }

    return i;
}

// DPDK QoS出队函数（时间复杂度：O(1)均摊）
int rte_sched_port_dequeue(struct rte_sched_port *port,
                           struct rte_mbuf **pkts,
                           uint32_t n_pkts)
{
    uint32_t count = 0;

    while (count < n_pkts) {
        // 1. 轮询子端口（RR）
        uint32_t subport = port->subport_next;

        // 2. 轮询管道（RR）
        uint32_t pipe = port->pipe_next[subport];

        // 3. 按优先级选择TC（严格优先级）
        for (uint32_t tc = 0; tc < port->n_tcs; tc++) {
            struct rte_sched_queue *q = get_queue(port, subport, pipe, tc);

            if (!rte_sched_queue_empty(q)) {
                // 4. 出队 - O(1)
                pkts[count++] = rte_mbuf_dequeue(q);
                break;  // 处理下一个包
            }
        }

        // 更新轮询指针
        port->pipe_next[subport] = (pipe + 1) % port->n_pipes;
        if (port->pipe_next[subport] == 0)
            port->subport_next = (subport + 1) % port->n_subports;
    }

    return count;
}
```

```python
class CrossLayerQoSMapper:
    """
    跨层QoS映射器
    实现RFC 8325推荐的DSCP到802.1p/802.11e映射
    """

    # DSCP到802.1p映射表（RFC 8325）
    DSCP_TO_8021P = {
        # 网络控制
        56: 7,   # CS7 -> NC (Network Control)
        48: 6,   # CS6 -> IC (Internetwork Control)

        # 语音
        46: 6,   # EF -> VO (Voice)
        44: 6,   # VA -> VO

        # 视频
        40: 5,   # CS5 -> VI (Video)
        34: 5,   # AF41 -> VI
        32: 4,   # CS4 -> CL (Controlled Load)

        # 关键数据
        26: 3,   # AF31 -> EE (Excellent Effort)
        24: 3,   # CS3 -> EE

        # 优先级数据
        18: 2,   # AF21 -> - (Spare)
        16: 2,   # CS2 -> -

        # 普通数据
        10: 1,   # AF11 -> BK (Background)
        8: 1,    # CS1 -> BK

        # 尽力而为
        0: 0,    # BE -> BE (Best Effort)
    }

    # 802.1p到DSCP反向映射（当需要向上传递QoS信息时）
    P8021_TO_DSCP = {
        7: 56,   # NC -> CS7
        6: 46,   # VO -> EF
        5: 34,   # VI -> AF41
        4: 26,   # CL -> AF31
        3: 18,   # EE -> AF21
        2: 10,   # - -> AF11
        1: 8,    # BK -> CS1
        0: 0,    # BE -> BE
    }

    # 5QI到DSCP映射（3GPP TS 23.501与RFC 8325对齐）
    QQI_TO_DSCP = {
        # GBR流量
        1: 46,   # VoIP -> EF
        2: 34,   # Live Video -> AF41
        3: 26,   # Real-time Gaming -> AF31
        4: 18,   # Non-conversational Video -> AF21
        65: 46,  # MCPTT -> EF
        66: 34,  # MCVideo -> AF41

        # Non-GBR流量
        5: 48,   # IMS Signaling -> CS6
        6: 10,   # Video Buffered -> AF11
        7: 26,   # Voice/Video/Gaming -> AF31
        8: 10,   # Video Buffered -> AF11
        9: 0,    # Internet -> BE
        69: 48,  # Mission Critical -> CS6
        70: 26,  # Mission Critical Data -> AF31

        # Delay Critical GBR
        82: 46,  # Discrete Automation -> EF
        83: 46,  # Discrete Automation Small -> EF
        84: 46,  # Intelligent Transport -> EF
        85: 46,  # Electricity Distribution -> EF
    }

    @classmethod
    def dscp_to_8021p(cls, dscp):
        """DSCP到802.1p映射"""
        return cls.DSCP_TO_8021P.get(dscp, 0)

    @classmethod
    def p8021_to_dscp(cls, pcp):
        """802.1p到DSCP映射"""
        return cls.P8021_TO_DSCP.get(pcp, 0)

    @classmethod
    def _5qi_to_dscp(cls, _5qi):
        """5QI到DSCP映射"""
        return cls.QQI_TO_DSCP.get(_5qi, 0)

    @classmethod
    def map_cross_layer(cls, qos_params, from_layer, to_layer):
        """
        执行跨层QoS映射

        参数：
            qos_params: 源层的QoS参数
            from_layer: 源层（'app', 'transport', 'network', 'link'）
            to_layer: 目标层

        返回：
            target_qos: 目标层的QoS参数

        时间复杂度：O(1)
        空间复杂度：O(1)
        """
        mapping_chain = {
            ('app', 'transport'): cls._app_to_transport,
            ('transport', 'network'): cls._transport_to_network,
            ('network', 'link'): cls._network_to_link,
            ('app', 'network'): cls._app_to_network,
            ('app', 'link'): cls._app_to_link,
        }

        mapper = mapping_chain.get((from_layer, to_layer))
        if mapper:
            return mapper(qos_params)

        raise ValueError(f"Unsupported mapping: {from_layer} -> {to_layer}")

    @staticmethod
    def _app_to_transport(app_qos):
        """应用层到传输层映射"""
        transport_qos = {}

        # 协议选择
        if app_qos['latency_req'] < 100:  # 低延迟
            if app_qos['reliability'] > 0.99:
                transport_qos['protocol'] = 'QUIC'  # 或 TCP BBR
                transport_qos['congestion_control'] = 'bbr'
            else:
                transport_qos['protocol'] = 'UDP'
                transport_qos['congestion_control'] = 'none'
        else:
            transport_qos['protocol'] = 'TCP'
            transport_qos['congestion_control'] = 'cubic'

        # 缓冲区大小
        if app_qos['latency_req'] < 50:
            transport_qos['buffer_size'] = 'small'  # 低延迟模式
        else:
            transport_qos['buffer_size'] = 'auto'   # 自动调优

        # 端口范围（根据应用类型）
        transport_qos['port_range'] = app_qos.get('port_range', 'default')

        return transport_qos

    @classmethod
    def _transport_to_network(cls, transport_qos):
        """传输层到网络层映射"""
        network_qos = {}

        # 基于协议和端口确定DSCP
        if transport_qos['protocol'] == 'UDP':
            if transport_qos.get('port', 0) in range(10000, 20000):
                # RTP语音
                network_qos['dscp'] = 46  # EF
            else:
                network_qos['dscp'] = 34  # AF41
        else:  # TCP
            if transport_qos.get('congestion_control') == 'bbr':
                network_qos['dscp'] = 18  # AF21 (低延迟数据)
            else:
                network_qos['dscp'] = 10  # AF11 (普通数据)

        # ECN标记
        if transport_qos['congestion_control'] in ['bbr', 'cubic', 'dctcp']:
            network_qos['ecn'] = 'ect0'  # 支持ECN
        else:
            network_qos['ecn'] = 'none'

        return network_qos

    @classmethod
    def _network_to_link(cls, network_qos):
        """网络层到链路层映射"""
        link_qos = {}

        # DSCP到802.1p
        dscp = network_qos.get('dscp', 0)
        link_qos['802.1p'] = cls.dscp_to_8021p(dscp)

        # DSCP到MPLS EXP
        link_qos['mpls_exp'] = min(dscp // 8, 7)

        # 队列调度策略
        if dscp >= 46:  # EF/CS6/CS7
            link_qos['scheduling'] = 'strict_priority'
        elif dscp >= 26:  # AF3x/AF4x
            link_qos['scheduling'] = 'llq'  # 低延迟队列
        else:
            link_qos['scheduling'] = 'cbwfq'  # 基于类的WFQ

        return link_qos

# 端到端QoS一致性检查
def check_qos_consistency(qos_layers):
    """
    检查跨层QoS一致性

    参数：
        qos_layers: 各层QoS参数字典

    返回：
        consistency_score: 一致性分数 (0-1)
        issues: 不一致问题列表

    时间复杂度：O(n)，n为层数
    """
    issues = []

    # 检查DSCP在各层的一致性
    app_dscp = qos_layers['app'].get('target_dscp')
    network_dscp = qos_layers['network'].get('dscp')
    link_dscp = CrossLayerQoSMapper.p8021_to_dscp(
        qos_layers['link'].get('802.1p', 0)
    )

    if app_dscp != network_dscp:
        issues.append(f"应用层DSCP({app_dscp})与网络层({network_dscp})不一致")

    if network_dscp != link_dscp:
        issues.append(f"网络层DSCP({network_dscp})与链路层({link_dscp})不一致")

    # 计算一致性分数
    consistency_score = 1.0 - (len(issues) * 0.1)
    return max(0, consistency_score), issues
```
