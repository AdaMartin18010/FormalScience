## 1 SDN调度概述

### 1.1 SDN架构

**SDN（Software-Defined Networking）架构**：

```text
应用层
  ↓
控制层（SDN控制器）
  ↓
数据层（交换机）
```

**SDN特征**：

- **控制与转发分离**：控制逻辑集中
- **可编程性**：网络可编程
- **集中管理**：集中式网络管理

### 1.2 SDN调度的核心挑战

SDN调度的核心挑战在于**控制器调度**和**流表管理**：

- **控制器调度**：控制器处理能力有限
- **流表管理**：流表大小有限
- **更新调度**：流表更新需要调度
- **可扩展性**：支持大规模网络

## 2 SDN协议分析

### 2.1 OpenFlow协议版本演进（1.0-1.5）

**OpenFlow协议演进历程**：

| 版本 | 发布时间 | 关键特性 | 匹配字段 | 动作类型 | 组表支持 |
|-----|---------|---------|---------|---------|---------|
| **OpenFlow 1.0** | 2009年12月 | 基础版本 | 12个 | 4种 | 不支持 |
| **OpenFlow 1.3** | 2012年4月 | IPv6、计量表 | 40个 | 16种 | 支持 |
| **OpenFlow 1.4** | 2013年10月 | 同步表、捆绑消息 | 41个 | 18种 | 增强 |
| **OpenFlow 1.5** | 2015年3月 | 流表监视、控制器类型 | 44个 | 20种 | 完整支持 |

#### 2.1.1 OpenFlow 1.0 核心特性

**匹配字段（12个）**：

- 入端口、VLAN ID、VLAN优先级、以太网源/目的MAC、以太网类型

### 2.2 P4编程语言分析

**P4（Programming Protocol-Independent Packet Processors）** 是一种用于编程网络数据平面的领域特定语言。

#### 2.2.1 P4核心特性

| 特性 | 描述 | 优势 |
|-----|------|------|
| **协议无关性** | 不绑定特定协议 | 支持任意协议解析 |

#### 2.2.2 P4程序结构

```p4
// P4程序基本结构
#include <core.p4>
#include <v1model.p4>

// 头部定义
header ethernet_t {
    bit<48> dstAddr;
    bit<48> srcAddr;
    bit<16> etherType;
}

// 解析器
parser MyParser(packet_in packet, out headers hdr, inout metadata meta, inout standard_metadata_t standard_metadata) {
    state start {
        packet.extract(hdr.ethernet);
        transition accept;
    }
}

// 控制流
control MyIngress(inout headers hdr, inout metadata meta, inout standard_metadata_t standard_metadata) {
    action drop() {
        mark_to_drop(standard_metadata);
    }

    action forward(bit<9> port) {
        standard_metadata.egress_spec = port;
    }

    table ipv4_forward {
        key = {
            hdr.ipv4.dstAddr: lpm;
        }
        actions = {
            forward;
            drop;
        }
        default_action = drop;
    }

    apply {
        ipv4_forward.apply();
    }
}
```

## 3 SDN控制器调度

### 3.1 控制器架构

**控制器架构**：

- **单控制器**：单一控制器，简单但存在单点故障
- **多控制器**：多个控制器协同，提高可靠性
- **分布式控制器**：分布式控制器集群，支持大规模网络

**控制器性能指标**：

- **吞吐量**：10K-100K flows/s（取决于控制器实现）
- **延迟**：< 10ms（P99）
- **可扩展性**：支持10K-100K交换机

### 3.2 控制器调度策略

**优先级调度**：

```text
关键流（如控制流）优先处理
  ↓
普通流按FIFO处理
  ↓
保证控制平面响应时间
```

**批处理调度**：

```text
收集多个流请求
  ↓
批量计算路由
  ↓
批量下发流表
  ↓
减少控制器开销
```

**批处理模型**：

$$
\text{Process}(batch) = \begin{cases}
\text{Immediate} & \text{if } |batch| < threshold \\
\text{Batch} & \text{otherwise}
\end{cases}
$$

**负载均衡**：

```text
多个控制器协同
  ↓
按交换机或流分配负载
  ↓
均衡控制器负载
  ↓
提高整体吞吐量
```

**负载分配模型**：

$$
\text{Assign}(switch, controller) = \arg\min_{c} \text{Load}(c)
$$

## 4 流表调度

### 4.1 流表管理

**流表大小限制**：

- **TCAM容量**：1K-32K entries（取决于硬件）
- **SRAM容量**：64K-1M entries（软件实现）
- **流表溢出**：需要替换策略

**流表替换策略**：

- **LRU（Least Recently Used）**：替换最久未使用的流表项
- **LFU（Least Frequently Used）**：替换使用频率最低的流表项
- **随机替换**：随机选择流表项替换
- **优先级替换**：低优先级流表项优先替换

**流表替换模型**：

$$
\text{Replace}(entry) = \arg\min_{e \in \text{Table}} \text{Priority}(e) \times \text{Usage}(e)
$$

**流表聚合**：

```text
识别相似流表项
  ↓
合并为通配符规则
  ↓
减少流表占用
  ↓
提高流表利用率
```

**聚合策略**：

- **前缀聚合**：合并相同前缀的IP地址
- **端口范围聚合**：合并连续端口范围
- **协议聚合**：合并相同协议的流

### 4.2 流表更新调度

**原子更新**：

```text
准备新流表项
  ↓
原子替换旧流表项
  ↓
保证数据包转发连续性
  ↓
避免转发中断
```

**批量更新**：

```text
收集多个流表更新
  ↓
批量计算更新顺序
  ↓
批量下发更新
  ↓
减少控制器-交换机通信开销
```

**批量更新模型**：

$$
\text{Update}(batch) = \text{Minimize} \sum_{i} \text{Cost}(\text{update}_i) + \text{CommunicationCost}(batch)
$$

**一致性更新**：

```text
多交换机流表更新
  ↓
保证更新顺序一致性
  ↓
避免转发环路
  ↓
保证网络一致性
```

**一致性保证**：

- **两阶段提交**：准备阶段 + 提交阶段
- **版本号机制**：使用版本号保证更新顺序
- **回滚机制**：更新失败时回滚

### 4.3 OpenFlow调度

**OpenFlow协议**：

OpenFlow是SDN的标准南向接口协议，定义了控制器与交换机之间的通信。

**OpenFlow流表匹配**：

```text
数据包到达
  ↓
匹配流表项（按优先级）
  ↓
执行动作（转发、丢弃、修改）
  ↓
更新计数器
```

**流表匹配模型**：

$$
\text{Match}(packet, table) = \arg\max_{entry \in table} \text{Priority}(entry) \times \text{Match}(packet, entry)
$$

**OpenFlow调度优化**：

- **流表优化**：优化流表项顺序，提高匹配效率
- **动作缓存**：缓存常用动作，减少处理延迟
- **批量处理**：批量处理数据包，提高吞吐量

**性能指标**：

- **匹配延迟**：< 1μs（硬件实现）
- **吞吐量**：> 10Mpps（单交换机）
- **流表容量**：1K-32K entries（TCAM）

## 5 网络虚拟化调度与NFV

### 5.1 虚拟网络调度

**网络切片（Network Slicing）**：

```text
物理网络资源
  ↓
虚拟化为多个逻辑网络
  ↓
每个切片独立配置
  ↓
支持多租户隔离
```

**资源隔离**：

- **带宽隔离**：每个虚拟网络分配独立带宽
- **流表隔离**：每个虚拟网络使用独立流表空间
- **计算隔离**：每个虚拟网络使用独立计算资源

**隔离模型**：

$$
\text{Isolate}(slice_i, slice_j) \iff \forall r \in \text{Resources}: \text{Allocate}(slice_i, r) \cap \text{Allocate}(slice_j, r) = \emptyset
$$

**动态分配**：

```text
监控虚拟网络资源使用
  ↓
根据需求动态调整资源
  ↓
提高资源利用率
  ↓
保证SLA
```

### 5.2 资源分配

**带宽分配**：

$$
\text{Bandwidth}(slice_i) = \frac{\text{weight}(slice_i)}{\sum_j \text{weight}(slice_j)} \times \text{TotalBandwidth}
$$

**流表分配**：

$$
\text{FlowTable}(slice_i) = \frac{\text{weight}(slice_i)}{\sum_j \text{weight}(slice_j)} \times \text{TotalFlowTable}
$$

**计算分配**：

- **CPU分配**：为每个虚拟网络分配CPU资源
- **内存分配**：为每个虚拟网络分配内存资源
- **存储分配**：为每个虚拟网络分配存储资源

### 5.3 VNF放置算法

**VNF放置问题定义**：

给定物理网络 $G=(N, E)$，其中 $N$ 是服务器节点集合，$E$ 是链路集合，以及一组VNF请求 $V = \{v_1, v_2, ..., v_m\}$，每个VNF $v_i$ 有资源需求 $(cpu_i, mem_i, disk_i)$ 和处理能力 $cap_i$。

**目标**：最小化部署成本，同时满足所有VNF的资源需求和性能约束。

$$
\min \sum_{n \in N} \sum_{v \in V} x_{v,n} \cdot (\alpha \cdot \text{Cost}_{cpu}(n) + \beta \cdot \text{Cost}_{mem}(n) + \gamma \cdot \text{Latency}(v,n))
$$

约束条件：

- $\forall n \in N: \sum_{v \in V} x_{v,n} \cdot cpu_v \leq CPU_n$
- $\forall n \in N: \sum_{v \in V} x_{v,n} \cdot mem_v \leq MEM_n$
- $\forall v \in V: \sum_{n \in N} x_{v,n} = 1$

#### 5.3.1 VNF放置算法对比

| 算法 | 时间复杂度 | 优化目标 | 适用场景 | 特点 |
|-----|-----------|---------|---------|------|
| **贪心算法** | $O(n \cdot m)$ | 成本最小化 | 小规模部署 | 简单快速，局部最优 |
| **整数线性规划(ILP)** | NP-hard | 全局最优 | 离线规划 | 最优解，计算复杂 |
| **启发式算法** | $O(n^2)$ | 多目标优化 | 大规模部署 | 平衡性能与复杂度 |
| **遗传算法** | $O(g \cdot n \cdot m)$ | 多目标优化 | 动态环境 | 适应性好，收敛慢 |
| **强化学习** | 训练开销大 | 长期收益 | 动态负载 | 自适应，需训练数据 |

#### 5.3.2 基于聚类的VNF放置算法

**算法步骤**：

```
1. 根据流量模式对VNF进行聚类
   ↓
2. 将同一聚类的VNF放置在同一服务器或相邻服务器
   ↓
3. 优化服务器间流量，减少带宽消耗
   ↓
4. 动态调整以适应负载变化
```

**形式化表示**：

$$
\text{Cluster}(V) = \{C_1, C_2, ..., C_k\}
$$

$$
\text{Place}(C_i) = \arg\min_{n \in N} \sum_{v \in C_i} \text{Latency}(v, n) + \lambda \cdot \text{InterTraffic}(C_i, n)
$$

### 5.4 服务链调度

**服务链（Service Function Chaining, SFC）**：将多个网络功能（VNF）按特定顺序连接，形成处理流水线。

**服务链模型**：

$$
SFC = (v_1 \rightarrow v_2 \rightarrow ... \rightarrow v_k)
$$

其中每个箭头表示VNF之间的流量转发。

#### 5.4.1 服务链调度目标

| 目标 | 描述 | 数学表达 |
|-----|------|---------|
| **延迟最小化** | 减少端到端处理延迟 | $\min \sum_{i=1}^{k-1} \text{Latency}(v_i, v_{i+1})$ |
| **带宽优化** | 减少服务链带宽占用 | $\min \sum_{i=1}^{k-1} \text{Bandwidth}(v_i, v_{i+1})$ |
| **负载均衡** | 均衡各服务器负载 | $\min \max_{n \in N} \text{Load}(n)$ |
| **可靠性保障** | 避免单点故障 | $\max \text{Availability}(SFC)$ |

#### 5.4.2 服务链部署策略

**策略1：集中式部署**

```
将所有VNF部署在同一服务器
  ↓
优点：零网络延迟，高吞吐量
  ↓
缺点：资源利用率低，扩展性差
```

**策略2：分布式部署**

```
将VNF分布部署在多个服务器
  ↓
优点：负载均衡，资源利用率高
  ↓
缺点：增加网络延迟，管理复杂
```

**策略3：混合式部署**

```
关键VNF集中部署，其他分布式部署
  ↓
优点：平衡延迟和资源利用
  ↓
适用：复杂服务链场景
```

### 5.5 资源分配优化

#### 5.5.1 动态资源调整

**垂直扩展（Scale Up/Down）**：

- 增加/减少VNF分配的CPU、内存资源
- 适用于负载变化可预测的场景

**水平扩展（Scale Out/In）**：

- 增加/减少VNF实例数量
- 使用负载均衡器分发流量
- 适用于突发负载场景

#### 5.5.2 资源超分配策略

**超分配模型**：

$$
\text{OverallocationRatio} = \frac{\sum_{v \in V} \text{Allocated}(v)}{\sum_{n \in N} \text{Physical}(n)}
$$

**超分配策略**：

| 策略 | 超分配比 | 风险 | 适用场景 |
|-----|---------|------|---------|
| **保守策略** | 1.0-1.2 | 低 | 关键业务 |
| **适中策略** | 1.2-1.5 | 中 | 一般业务 |
| **激进策略** | 1.5-2.0 | 高 | 开发测试环境 |

## 6 形式化模型

### 6.1 SDN调度问题定义

$$
\text{SDN调度问题} = (C, S, F, R, O)
$$

其中：

- $C = \{c_1, c_2, \ldots, c_k\}$：控制器集合
- $S = \{s_1, s_2, \ldots, s_n\}$：交换机集合
- $F = \{f_1, f_2, \ldots, f_m\}$：流集合
- $R$：资源约束
  - 控制器容量：$\sum_{s \in S} \text{Load}(s, c) \leq \text{Capacity}(c)$
  - 流表容量：$|\text{FlowTable}(s)| \leq \text{MaxFlowTable}(s)$
  - 带宽约束：$\sum_{f \in F} \text{Bandwidth}(f) \leq \text{TotalBandwidth}$
- $O$：优化目标
  - 最小化延迟：$\min \sum_i \text{latency}(f_i)$
  - 最大化吞吐量：$\max \sum_i \text{throughput}(f_i)$
  - 最小化控制器负载：$\min \max_{c \in C} \text{Load}(c)$

### 6.2 控制器调度算法复杂度

| **算法** | **时间复杂度** | **可扩展性** | **可靠性** | **适用场景** |
|---------|--------------|------------|-----------|------------|
| **单控制器** | $O(1)$ | ⭐ | ⭐ | 小规模网络 |
| **多控制器（静态）** | $O(n)$ | ⭐⭐⭐ | ⭐⭐⭐ | 中等规模网络 |
| **多控制器（动态）** | $O(n \log n)$ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 大规模网络 |
| **分布式控制器** | $O(n \log n)$ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 超大规模网络 |

### 6.3 流表调度算法复杂度

| **算法** | **匹配复杂度** | **更新复杂度** | **内存占用** | **适用场景** |
|---------|--------------|--------------|------------|------------|
| **线性匹配** | $O(n)$ | $O(1)$ | 低 | 小规模流表 |
| **哈希匹配** | $O(1)$ | $O(1)$ | 中 | 精确匹配 |
| **TCAM匹配** | $O(1)$ | $O(1)$ | 高 | 通配符匹配 |
| **Trie树匹配** | $O(\log n)$ | $O(\log n)$ | 中 | 前缀匹配 |

### 6.4 NFV调度形式化模型

**NFV调度问题**可以形式化为：

$$
\text{NFV调度} = (V, N, E, D, R, O)
$$

其中：

- $V = \{v_1, v_2, ..., v_m\}$：VNF实例集合
- $N = \{n_1, n_2, ..., n_k\}$：物理服务器节点集合
- $E \subseteq N \times N$：物理链路集合
- $D = \{d_1, d_2, ..., d_l\}$：服务链请求集合
- $R$：资源约束
  - CPU约束：$\sum_{v \in V_n} cpu_v \leq CPU_n$
  - 内存约束：$\sum_{v \in V_n} mem_v \leq MEM_n$
  - 带宽约束：$\sum_{e \in E_n} bw_e \leq BW_n$
- $O$：优化目标
  - 最小化部署成本：$\min \sum_{n} \text{Cost}(n)$
  - 最小化服务延迟：$\min \sum_{d} \text{Latency}(d)$
  - 最大化资源利用率：$\max \frac{\sum \text{Used}}{\sum \text{Total}}$

#### 7.1.3 B4调度算法

**核心算法**：集中式流量工程（Centralized TE）

```
输入：
  - 网络拓扑 G = (V, E)
  - 流量需求矩阵 D
  - 链路容量 C

目标：
  - 最大化总吞吐量
  - 保证公平性
  - 优先满足高优先级流量

求解：
  使用线性规划求解最优路径分配

输出：
  - 每个流的路径
  - 每个链路的带宽分配
```

**形式化表达**：

$$
\max \sum_{f \in F} U_f(x_f)
$$

约束条件：

- $\forall e \in E: \sum_{f: e \in p_f} x_f \leq c_e$ （链路容量约束）
- $\forall f \in F: x_f \leq d_f$ （流量需求约束）
- $\forall f \in F: x_f \geq 0$ （非负约束）

其中 $U_f(x_f)$ 是效用函数，$x_f$ 是流 $f$ 的分配带宽，$d_f$ 是需求带宽。

## 8 跨领域洞察

### 8.1 SDN调度与网络调度

**SDN vs 传统网络**：

- **SDN**：集中控制，可编程，灵活
- **传统网络**：分布式控制，固定，稳定

**关键洞察**：**SDN调度是网络调度的集中化**，将分布式决策集中到控制器，提高网络可编程性和灵活性。

### 8.2 SDN调度与QoS调度

**SDN中的QoS**：

```text
SDN控制器
  ↓
根据QoS要求计算路由
  ↓
下发QoS流表项
  ↓
交换机执行QoS策略
```

**QoS流表项**：

- **带宽限制**：设置流的最大带宽
- **优先级标记**：设置流的优先级
- **队列调度**：选择队列调度算法

**关键洞察**：**SDN为QoS调度提供了集中化的控制平面**，可以全局优化QoS策略。

### 8.3 SDN调度与网络虚拟化

**网络虚拟化调度**：

```text
物理网络
  ↓
虚拟化为多个逻辑网络
  ↓
每个虚拟网络独立调度
  ↓
支持多租户隔离
```

**虚拟网络调度模型**：

$$
\text{Schedule}(slice, flow) = \text{Isolate}(slice) \land \text{QoS}(flow, slice)
$$

**关键洞察**：**SDN为网络虚拟化提供了灵活的调度机制**，支持细粒度的资源隔离和QoS保证。

### OpenFlow控制器-交换机通信算法

```python
class OpenFlowController:
    """
    OpenFlow控制器核心实现

    处理OpenFlow消息流，维护流表状态
    参考：ONF OpenFlow Switch Specification 1.5
    """

    def __init__(self, switch_conn):
        self.switch = switch_conn
        self.flow_tables = {}  # 流表缓存
        self.packet_buffer = {}  # PacketIn缓冲
        self.xid_counter = 0     # 事务ID计数器

    def handle_message(self, msg):
        """
        OpenFlow消息处理器

        时间复杂度：O(1) 到 O(n)，取决于消息类型
        空间复杂度：O(1) 到 O(n)
        """
        msg_type = msg.header.type

        handlers = {
            OFPT_HELLO: self.handle_hello,
            OFPT_ERROR: self.handle_error,
            OFPT_ECHO_REQUEST: self.handle_echo,
            OFPT_FEATURES_REPLY: self.handle_features,
            OFPT_PACKET_IN: self.handle_packet_in,
            OFPT_FLOW_REMOVED: self.handle_flow_removed,
            OFPT_PORT_STATUS: self.handle_port_status,
            OFPT_STATISTICS_REPLY: self.handle_stats,
        }

        handler = handlers.get(msg_type, self.handle_unknown)
        return handler(msg)

    def handle_packet_in(self, msg):
        """
        处理PacketIn消息

        时间复杂度：O(m)，m为流表项数量
        空间复杂度：O(1)
        """
        # 1. 解析数据包
        packet = PacketParser.parse(msg.data)
        in_port = msg.in_port

        # 2. 决策路由
        flow_entry = self.decide_routing(packet, in_port)

        if flow_entry:
            # 3. 下发流表项
            self.send_flow_mod(flow_entry)

            # 4. 转发当前包
            self.send_packet_out(msg.buffer_id, in_port, flow_entry.actions)
        else:
            # 5. 无法处理，丢弃或泛洪
            self.send_packet_out(msg.buffer_id, in_port, [OFPAT_DROP])

    def decide_routing(self, packet, in_port):
        """
        路由决策算法

        时间复杂度：O(n + log m)
        n = 网络拓扑节点数
        m = 路由表条目数

        空间复杂度：O(n)
        """
        # 1. 确定目的位置
        dst_location = self.topology.lookup(packet.dst_ip)

        # 2. 计算最短路径（Dijkstra）
        path = self.dijkstra(self.switch.id, dst_location)

        # 3. 确定出端口
        out_port = path[0].out_port

        # 4. 构造流表项
        match = Match(
            in_port=in_port,
            eth_dst=packet.dst_mac,
            eth_type=packet.eth_type
        )

        actions = [Action(OFPAT_OUTPUT, port=out_port)]

        return FlowEntry(match=match, actions=actions, priority=100)

    def send_flow_mod(self, flow_entry):
        """
        发送FlowMod消息

        时间复杂度：O(k)，k为指令数量
        """
        msg = OFPFlowMod(
            xid=self.next_xid(),
            command=OFPFC_ADD,
            match=flow_entry.match,
            instructions=flow_entry.actions,
            priority=flow_entry.priority,
            idle_timeout=60,
            hard_timeout=300
        )

        self.switch.send(msg)

    @staticmethod
    def dijkstra(source, destinations, topology):
        """
        Dijkstra最短路径算法

        时间复杂度：O((V + E) log V)
        V = 顶点数（交换机数）
        E = 边数（链路数）

        空间复杂度：O(V)
        """
        import heapq

        dist = {v: float('inf') for v in topology.nodes}
        prev = {v: None for v in topology.nodes}
        dist[source] = 0

        pq = [(0, source)]

        while pq:
            d, u = heapq.heappop(pq)

            if d > dist[u]:
                continue

            for v, weight in topology.neighbors(u):
                alt = dist[u] + weight
                if alt < dist[v]:
                    dist[v] = alt
                    prev[v] = u
                    heapq.heappush(pq, (alt, v))

        # 重建路径
        paths = {}
        for dest in destinations:
            path = []
            u = dest
            while u is not None:
                path.append(u)
                u = prev[u]
            paths[dest] = list(reversed(path))

        return paths
```

## P4语言详细实现

### P4程序结构详解

```p4
/* P4_16程序完整示例：L3转发
 * 参考：P4 Language Specification v1.2.1
 */

#include <core.p4>
#include <v1model.p4>

/* ==================== 1. 头部定义 ==================== */

// 以太网头部
define ETHERNET_HEADER_SIZE 14

header ethernet_h {
    bit<48> dst_addr;    // 目的MAC
    bit<48> src_addr;    // 源MAC
    bit<16> ether_type;  // 以太网类型
}

// IPv4头部
define IPV4_HEADER_SIZE 20

header ipv4_h {
    bit<4>  version;        // 版本 (4)
    bit<4>  ihl;            // 头部长度
    bit<8>  diffserv;       // DSCP (6 bits) + ECN (2 bits)
    bit<16> total_len;      // 总长度
    bit<16> identification; // 标识
    bit<3>  flags;          // 标志
    bit<13> frag_offset;    // 分片偏移
    bit<8>  ttl;            // 生存时间
    bit<8>  protocol;       // 协议
    bit<16> hdr_checksum;   // 头部校验和
    bit<32> src_addr;       // 源IP
    bit<32> dst_addr;       // 目的IP
}

// 元数据（控制器与数据平面通信）
struct metadata_t {
    bit<32> ingress_port;
    bit<32> egress_port;
    bit<32> next_hop;
    bit<8>  qos_class;
}

// 头部集合
struct headers_t {
    ethernet_h ethernet;
    ipv4_h     ipv4;
}

/* ==================== 2. 解析器 ==================== */

parser ParserImpl(packet_in packet,
                  out headers_t hdr,
                  inout metadata_t meta,
                  inout standard_metadata_t standard_metadata)
{
    state start {
        // 解析以太网头部
        packet.extract(hdr.ethernet);

        // 根据ether_type决定后续解析
        transition select(hdr.ethernet.ether_type) {
            0x0800: parse_ipv4;  // IPv4
            0x86DD: parse_ipv6;  // IPv6
            default: accept;     // 其他，直接接受
        }
    }

    state parse_ipv4 {
        packet.extract(hdr.ipv4);
        transition accept;
    }

    state parse_ipv6 {
        // IPv6解析（简化）
        transition accept;
    }
}

/* ==================== 3. 控制流（Ingress） ==================== */

control IngressImpl(inout headers_t hdr,
                    inout metadata_t meta,
                    inout standard_metadata_t standard_metadata)
{
    // 动作定义

    action drop() {
        // 标记丢弃
        mark_to_drop(standard_metadata);
    }

    action forward(bit<48> dst_mac, bit<32> next_hop, bit<9> port) {
        // 更新以太网头部
        hdr.ethernet.dst_addr = dst_mac;
        hdr.ethernet.src_addr = hdr.ethernet.dst_addr;  // 简化处理

        // 设置出端口
        standard_metadata.egress_spec = port;

        // 更新元数据
        meta.next_hop = next_hop;

        // 递减TTL
        hdr.ipv4.ttl = hdr.ipv4.ttl - 1;
    }

    action set_qos(bit<6> dscp, bit<8> queue_id) {
        // 设置DSCP
        hdr.ipv4.diffserv[7:2] = dscp;

        // 设置队列（通过元数据传递给队列调度）
        meta.qos_class = queue_id;
    }

    // 转发表（匹配目的IP，执行forward动作）
    table ipv4_forward {
        key = {
            hdr.ipv4.dst_addr: lpm;  // 最长前缀匹配
        }
        actions = {
            forward;
            drop;
            NoAction;
        }
        size = 65536;  // 表大小
        default_action = drop();
    }

    // QoS表（根据DSCP分类）
    table qos_classifier {
        key = {
            hdr.ipv4.diffserv[7:2]: exact;  // DSCP字段
        }
        actions = {
            set_qos;
            NoAction;
        }
        size = 64;  // DSCP只有64种
        default_action = NoAction();
    }

    apply {
        // 1. 检查TTL
        if (hdr.ipv4.ttl == 0) {
            drop();
            return;
        }

        // 2. QoS分类
        qos_classifier.apply();

        // 3. 路由查找
        ipv4_forward.apply();

        // 4. 校验和更新（可选，硬件通常自动处理）
    }
}

/* ==================== 4. 控制流（Egress） ==================== */

control EgressImpl(inout headers_t hdr,
                   inout metadata_t meta,
                   inout standard_metadata_t standard_metadata)
```

```python
class SDNControllerBase:
    """
    SDN控制器基类

    展示控制器核心处理流程
    """

    def __init__(self):
        self.switches = {}           # 交换机连接
        self.topology = Topology()   # 网络拓扑
        self.flow_tables = {}        # 流表缓存
        self.packet_handlers = []    # 包处理器

    def run(self):
        """
        控制器主循环

        时间复杂度：O(m) per event，m为处理器数量
        """
        while True:
            # 1. 接受新连接
            conn = self.accept_connection()
            if conn:
                self.handle_new_switch(conn)

            # 2. 处理消息（事件驱动）
            for switch_id, conn in self.switches.items():
                msg = conn.receive(timeout=0)
                if msg:
                    self.process_message(switch_id, msg)

            # 3. 周期性任务
            self.periodic_tasks()

    def process_message(self, switch_id, msg):
        """
        消息处理

        时间复杂度：O(1) 到 O(n)
        """
        msg_type = msg.type

        if msg_type == 'PACKET_IN':
            # 数据平面请求处理
            self.handle_packet_in(switch_id, msg)

        elif msg_type == 'FLOW_REMOVED':
            # 流表过期通知
            self.handle_flow_removed(switch_id, msg)

        elif msg_type == 'PORT_STATUS':
            # 端口状态变化
            self.handle_port_status(switch_id, msg)

        elif msg_type == 'STATS_REPLY':
            # 统计信息
            self.handle_stats_reply(switch_id, msg)

    def handle_packet_in(self, switch_id, msg):
        """
        PacketIn处理

        时间复杂度：O(p + r)
        p = 处理器数量
        r = 路由计算复杂度
        """
        packet = Packet(msg.data)

        # 1. 运行所有包处理器
        for handler in self.packet_handlers:
            result = handler.process(packet, switch_id, msg.in_port)
            if result.action == 'DROP':
                self.drop_packet(switch_id, msg.buffer_id)
                return
            elif result.action == 'FORWARD':
                self.forward_packet(switch_id, msg.buffer_id, result.out_port)
                return

        # 2. 默认处理：泛洪或学习
        self.learn_and_forward(switch_id, msg)

class OpenDaylightController(SDNControllerBase):
    """
    OpenDaylight控制器特点

    - MD-SAL：模型驱动的服务适配层
    - YANG建模：统一数据建模
    - 插件架构：OSGi Bundle
    """

    def __init__(self):
        super().__init__()
        self.md_sal = MDSAL()          # 模型驱动服务适配层
        self.yang_context = YANGContext()  # YANG上下文
        self.osgi_framework = OSGi()   # OSGi框架

    def install_flow(self, switch_id, flow):
        """
        ODL流表安装流程

        时间复杂度：O(log n)，n为流表项数量
        """
        # 1. YANG数据建模
        yang_data = self.yang_context.serialize(flow)

        # 2. MD-SAL事务
        with self.md_sal.transaction() as tx:
            tx.put(flow.id, yang_data)

        # 3. 转换为OpenFlow消息
        of_msg = self.convert_to_openflow(flow)

        # 4. 下发到交换机
        self.switches[switch_id].send(of_msg)

class ONOSController(SDNControllerBase):
    """
    ONOS控制器特点

    - 分布式架构：多实例状态同步
    - Atomix：分布式一致性
    - 意图框架：高抽象级别编程
    """

    def __init__(self):
        super().__init__()
        self.atomix = AtomixCluster()  # 分布式一致性
        self.intent_framework = IntentFramework()
        self.distributed_store = ConsistentMap()

    def install_flow(self, switch_id, flow, cluster_scope=True):
        """
        ONOS分布式流表安装

        时间复杂度：O(log n) 本地，O(k log n) 分布式（k为副本数）
        """
        if cluster_scope:
            # 分布式存储（Raft一致性协议）
            self.distributed_store.put(flow.id, flow)

            # 广播到其他控制器实例
            self.atomix.broadcast('flow_added', flow)

        # 本地下发
        super().install_flow(switch_id, flow)

    def process_intent(self, intent):
        """
        意图框架处理

        意图 -> 编译 -> 流表
        """
        # 1. 意图编译
        compiled = self.intent_framework.compile(intent)

        # 2. 生成流表
        flows = compiled.generate_flows(self.topology)

        # 3. 安装
        for flow in flows:
            self.install_flow(flow.switch_id, flow, cluster_scope=True)

class RyuController(SDNControllerBase):

class SimpleSwitch(app_manager.RyuApp):

    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)
        self.mac_to_port = {}

    @set_ev_cls(ofp_event.EventOFPPacketIn)
    def packet_in_handler(self, ev):
        """PacketIn处理器"""
        msg = ev.msg
        datapath = msg.datapath
        ofproto = datapath.ofproto

        pkt = packet.Packet(msg.data)
        eth = pkt.get_protocol(ethernet.ethernet)

        dst = eth.dst
        src = eth.src
        dpid = datapath.id
        in_port = msg.match['in_port']

        # 学习MAC地址
        self.mac_to_port.setdefault(dpid, {})
        self.mac_to_port[dpid][src] = in_port

        # 决策
        if dst in self.mac_to_port[dpid]:
            out_port = self.mac_to_port[dpid][dst]
        else:
            out_port = ofproto.OFPP_FLOOD

        # 安装流表
        self.add_flow(datapath, in_port, dst, out_port)
```
