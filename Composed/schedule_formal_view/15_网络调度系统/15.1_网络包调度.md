#### 2.2 技术特性对比矩阵

| 技术 | 优势 | 劣势 | 适用场景 | 性能 |
|------|------|------|---------|------|
| **FIFO** | 实现简单、延迟最低 | 无公平性保证、无QoS | 简单网络、低延迟需求 | O(1)时间复杂度 |
| **WFQ** | 加权公平性最优、有界延迟 | 实现复杂、需维护虚拟时间 | QoS网络、多租户 | O(log n)时间复杂度 |
| **DRR** | 公平性好、O(1)复杂度 | 延迟略高、需维护赤字计数器 | 高速路由器、大规模网络 | O(1)时间复杂度 |
| **FQ-CoDel** | 流公平性、延迟控制好 | 实现复杂、需流状态 | 现代Linux网络栈、云网络 | O(log n)时间复杂度 |
| **令牌桶** | 允许突发、长期速率保证 | 瞬时速率可能超过限制 | 流量整形、带宽限制 | O(1)时间复杂度 |
| **漏桶** | 输出平滑、严格限速 | 不允许突发、延迟增加 | 严格速率控制、平滑输出 | O(1)时间复杂度 |

#### 2.3 实现方式对比矩阵

| 实现方式 | 复杂度 | 性能 | 可维护性 | 扩展性 |
|---------|-------|------|---------|-------|

| **用户态实现(DPDK)** | 高 | 极高性能(<3μs延迟) | 高(用户态开发) | 高(独立升级) |
| **硬件实现(TCAM)** | 低 | 极高性能(<1μs延迟) | 低(需硬件支持) | 低(硬件固定) |
| **eBPF/XDP实现** | 中等 | 高性能(<5μs延迟) | 高(可动态加载) | 高(无需重启) |

## 📚 理论体系

### 理论基础

#### 调度系统/硬件/软件基础

15.1 网络包调度的理论基础：

**1. 调度系统基础**：

- **队列调度理论**：FIFO、WFQ、DRR、FQ-CoDel等算法的理论基础
- **公平性理论**：Max-Min Fairness、Proportional Fairness等公平性定义
- **延迟控制理论**：队列延迟、传输延迟、端到端延迟的分析模型
- **流量整形理论**：令牌桶、漏桶算法的数学建模

**2. 硬件基础**：

- **网络接口卡(NIC)**：硬件队列、RSS(Receive Side Scaling)、硬件加速
- **CPU架构**：多核处理、NUMA架构对网络包处理的影响
- **内存系统**：DMA(Direct Memory Access)、零拷贝技术
- **专用硬件**：TCAM(三态内容寻址存储器)用于快速包分类

**3. 软件基础**：

- **操作系统网络栈**：Linux内核网络栈架构、协议栈层次
- **用户态网络框架**：DPDK、XDP、eBPF等高性能网络框架
- **网络协议**：TCP/IP协议栈、套接字接口、网络缓冲区管理

#### 历史发展

**关键时间节点**：

- **1960-1970年代**：网络包调度理论建立
  - 1970年代：早期队列调度算法研究（FIFO、Priority Queuing）
  - 1975年：Jackson网络模型为队列调度提供理论基础
  - 理论基础：排队论（Queueing Theory）和调度理论

- **1980-1990年代**：公平队列调度算法发展
  - 1985年：Demers、Keshav、Shenker提出Fair Queuing算法
  - 1990年：Parekh & Gallager提出Generalized Processor Sharing (GPS)
  - 1995年：Shreedhar & Varghese提出Deficit Round Robin (DRR)
  - 1996年：Weighted Fair Queuing (WFQ)在路由器中广泛应用

- **2000年代至今**：现代网络包调度技术演进
  - 2000年代：Linux内核引入HTB (Hierarchical Token Bucket)和CBQ
  - 2010年：CoDel (Controlled Delay) AQM算法提出
  - 2012年：FQ-CoDel算法结合公平队列和延迟控制
  - 2013年：DPDK框架成熟，用户态网络包处理成为主流
  - 2015年：XDP (eXpress Data Path)和eBPF引入内核
  - 2020年代：AI驱动的智能网络包调度开始应用

### 理论框架

#### 核心假设

**假设1：调度算法影响网络性能**

- **内容**：不同的包调度算法（FIFO、WFQ、DRR、FQ-CoDel）会产生不同的延迟、吞吐量和公平性特征
- **适用范围**：所有网络包调度场景
- **限制条件**：假设有足够的CPU和内存资源支持调度算法执行
- **支持证据**：大量实验数据和形式化分析（Parekh & Gallager 1993, Shreedhar & Varghese 1995）

**假设2：带宽资源的有限性**

- **内容**：网络链路带宽是有限资源，需要公平分配或优先级分配
- **适用范围**：共享链路上的多流竞争场景
- **限制条件**：假设链路容量已知且稳定
- **支持证据**：网络拥塞理论和实践验证

**假设3：延迟与吞吐量的权衡**

- **内容**：网络包调度需要在延迟和吞吐量之间做出权衡，公平性算法通常以更高的延迟为代价
- **适用范围**：QoS保证场景
- **限制条件**：受物理距离和光速限制
- **支持证据**：队列理论和实际测量数据

#### 主要定理/结论

**定理15.1：WFQ公平性保证（Parekh & Gallager, 1993）**

- **内容**：对于权重为$w_i$的流$i$，WFQ保证其获得的带宽满足：$bw_i \geq \frac{w_i}{\sum_j w_j} \cdot B_{total}$
- **证据**：基于虚拟时间函数的形式化证明
- **应用**：QoS网络中的带宽保证，多租户网络资源分配

**定理15.2：DRR公平性（Shreedhar & Varghese, 1995）**

- **内容**：DRR算法在$O(1)$时间内保证流之间的公平性，每个流获得的带宽与其权重成正比
- **证据**：基于赤字计数器的数学证明
- **应用**：高速路由器中的公平队列调度

**结论15.3：FQ-CoDel延迟控制（Nichols & Jacobson, 2012）**

- **内容**：FQ-CoDel算法结合流级公平性和队列延迟控制，保证队列延迟不超过5ms
- **证据**：基于CoDel AQM机制的实验验证
- **应用**：现代Linux网络栈的默认调度算法，云网络的低延迟保证

## 1 网络包调度概述

### 1.1 网络包调度的核心挑战

网络包调度的核心挑战在于**带宽共享**和**延迟控制**：

- **带宽竞争**：多个流共享有限带宽
- **延迟累积**：队列延迟随队列长度增长
- **公平性**：不同流之间的带宽公平分配
- **突发处理**：处理流量突发和拥塞

### 1.2 包调度器的目标

包调度器需要在以下目标之间权衡：

1. **公平性**：保证各流获得公平带宽
2. **延迟控制**：最小化包延迟和抖动
3. **吞吐量**：最大化链路利用率
4. **QoS保证**：满足不同流的服务质量要求

## 2 主要队列调度算法

### 2.1 FIFO调度

**FIFO（First In First Out）**：先进先出调度

#### 2.1.1 核心算法

```text
单一队列，按到达顺序调度
  ↓
无优先级，无公平性保证
  ↓
简单高效，延迟最低
```

#### 2.1.2 伪代码实现

```python
class FIFOScheduler:
    def __init__(self):
        self.queue = deque()  # 单一FIFO队列
        self.lock = threading.Lock()

    def enqueue(self, packet):
        """入队操作"""
        with self.lock:
            self.queue.append(packet)

    def dequeue(self):
        """出队操作 - O(1)"""
        with self.lock:
            if self.queue:
                return self.queue.popleft()
            return None

    def schedule(self):
        """调度一个包"""
        return self.dequeue()
```

#### 2.1.3 复杂度分析

| **指标** | **复杂度** | **说明** |
|---------|-----------|---------|
| **时间复杂度（入队）** | $O(1)$ | 直接追加到队列尾部 |
| **时间复杂度（出队）** | $O(1)$ | 从队列头部移除 |
| **空间复杂度** | $O(n)$ | 存储n个数据包，每个包需维护指针 |
| **缓存友好性** | 高 | 顺序访问，良好的局部性 |

#### 2.1.4 特点

- **延迟最低**：无调度开销
- **无公平性**：无法保证带宽公平
- **适用场景**：低延迟网络、简单场景

**时间复杂度**：$O(1)$

### 2.2 WFQ调度

**WFQ（Weighted Fair Queuing）**：加权公平队列调度

#### 2.2.1 核心算法

```text
为每个流维护虚拟完成时间
  ↓
按虚拟完成时间排序
  ↓
优先调度虚拟完成时间最小的包
```

**虚拟完成时间计算**：

$$
F_i^k = \max(F_i^{k-1}, V(t)) + \frac{L_i^k}{w_i}
$$

其中：

- $F_i^k$：流$i$的第$k$个包的虚拟完成时间
- $V(t)$：虚拟时间函数
- $L_i^k$：包大小
- $w_i$：流$i$的权重

#### 2.2.2 伪代码实现

```python
import heapq

class WFQScheduler:
    def __init__(self, num_flows):
        self.num_flows = num_flows
        self.flows = {}  # 流ID -> 流状态
        self.virtual_time = 0.0  # 全局虚拟时间
        self.total_weight = 0  # 总权重
        self.pq = []  # 优先队列 (virtual_finish_time, flow_id, packet)

    def register_flow(self, flow_id, weight):
        """注册一个流及其权重"""
        self.flows[flow_id] = {
            'weight': weight,
            'last_finish_time': 0.0
        }
        self.total_weight += weight

    def enqueue(self, flow_id, packet):
        """入队操作 - O(log n)"""
        flow = self.flows[flow_id]
        weight = flow['weight']
        packet_length = packet.size

        # 计算虚拟开始时间和完成时间
        virtual_start = max(flow['last_finish_time'], self.virtual_time)
        virtual_finish = virtual_start + (packet_length / weight)
        flow['last_finish_time'] = virtual_finish

        # 加入优先队列
        heapq.heappush(self.pq, (virtual_finish, flow_id, packet))

    def dequeue(self):
        """出队操作 - O(log n)"""
        if self.pq:
            virtual_finish, flow_id, packet = heapq.heappop(self.pq)
            # 更新虚拟时间
            self.virtual_time = virtual_finish
            return packet
        return None

    def schedule(self):
        """调度一个包"""
        return self.dequeue()
```

#### 2.2.3 复杂度分析

| **指标** | **复杂度** | **说明** |
|---------|-----------|---------|
| **时间复杂度（入队）** | $O(\log n)$ | 堆插入操作，n为总包数 |
| **时间复杂度（出队）** | $O(\log n)$ | 堆删除操作 |
| **空间复杂度** | $O(m + n)$ | m为流数，n为队列中的包数 |
| **每流状态** | $O(1)$ | 每个流需维护last_finish_time |
| **缓存友好性** | 低 | 堆结构随机访问，缓存不友好 |

#### 2.2.4 特点

- **公平性最优**：保证加权公平带宽分配
- **延迟保证**：有界延迟保证
- **适用场景**：QoS网络、多租户环境

**时间复杂度**：$O(\log n)$（堆排序）

### 2.3 DRR调度

**DRR（Deficit Round Robin）**：赤字轮询调度

#### 2.3.1 核心算法

```text
为每个流维护赤字计数器
  ↓
轮询各流，分配配额
  ↓
发送包直到配额用完
```

#### 2.3.2 伪代码实现

```python
class DRRScheduler:
    def __init__(self, num_flows, quantum=1500):
        self.num_flows = num_flows
        self.quantum = quantum  # 默认配额（通常设为MTU大小）
        self.flows = {}  # 流ID -> 流状态
        self.active_list = deque()  # 活跃流列表
        self.current_index = 0

    def register_flow(self, flow_id, weight=1):
        """注册一个流"""
        self.flows[flow_id] = {
            'queue': deque(),  # 该流的包队列
            'deficit_counter': 0,  # 赤字计数器
            'weight': weight,  # 权重
            'active': False
        }

    def enqueue(self, flow_id, packet):
        """入队操作 - O(1)"""
        flow = self.flows[flow_id]
        was_empty = len(flow['queue']) == 0
        flow['queue'].append(packet)

        # 如果流之前为空，加入活跃列表
        if was_empty and not flow['active']:
            flow['active'] = True
            self.active_list.append(flow_id)

    def dequeue(self):
        """出队操作 - O(1)"""
        while self.active_list:
            flow_id = self.active_list[0]
            flow = self.flows[flow_id]

            # 增加赤字计数器
            flow['deficit_counter'] += self.quantum * flow['weight']

            # 尝试发送该流的包
            sent = False
            while flow['queue']:
                packet = flow['queue'][0]
                packet_size = packet.size

                if packet_size <= flow['deficit_counter']:
                    # 可以发送
                    flow['deficit_counter'] -= packet_size
                    flow['queue'].popleft()
                    sent = True

                    # 如果队列还有包，继续留在活跃列表
                    if flow['queue']:
                        break
                else:
                    # 配额不足，跳出内层循环
                    break

            # 如果该流还有包，移到活跃列表末尾
            if flow['queue']:
                self.active_list.append(self.active_list.popleft())
            else:
                # 流为空，移出活跃列表
                flow['active'] = False
                flow['deficit_counter'] = 0
                self.active_list.popleft()

            if sent:
                return packet

        return None

    def schedule(self):
        """调度一个包"""
        return self.dequeue()
```

#### 2.3.3 复杂度分析

| **指标** | **复杂度** | **说明** |
|---------|-----------|---------|
| **时间复杂度（入队）** | $O(1)$ | 直接追加到对应流的队列 |
| **时间复杂度（出队/调度）** | $O(1)$ amortized | 均摊O(1)，每次调度至少发送一个包 |
| **最坏情况单次调度** | $O(k)$ | k为单个配额内可发送的包数 |
| **空间复杂度** | $O(m + n)$ | m为流数，n为总包数 |
| **每流状态** | $O(1)$ | 每个流需维护deficit_counter和active标志 |
| **缓存友好性** | 高 | 顺序访问活跃列表，良好局部性 |

#### 2.3.4 特点

- **公平性**：轮询保证公平性
- **低复杂度**：$O(1)$时间复杂度
- **适用场景**：高速网络、路由器

**时间复杂度**：$O(1)$

### 2.4 FQ-CoDel调度

**FQ-CoDel（Fair Queuing with Controlled Delay）**：公平队列与受控延迟

#### 2.4.1 核心算法

```text
流隔离：每个流独立队列
  ↓
公平调度：轮询各流
  ↓
CoDel AQM：控制队列延迟
  ↓
丢弃超时包（>5ms）
```

#### 2.4.2 伪代码实现

```python
import time
import hashlib

class FQCoDelScheduler:
    # CoDel参数
    TARGET = 5e-3  # 目标延迟 5ms
    INTERVAL = 100e-3  # 测量间隔 100ms

    def __init__(self, max_flows=1024, quantum=1500):
        self.max_flows = max_flows
        self.quantum = quantum
        self.flows = {}  # 流哈希 -> 流状态
        self.active_list = deque()
        self.flow_count = 0

    def get_flow_hash(self, packet):
        """计算流的哈希值（通常基于五元组）"""
        flow_key = f"{packet.src_ip}:{packet.src_port}-{packet.dst_ip}:{packet.dst_port}:{packet.proto}"
        return hashlib.md5(flow_key.encode()).hexdigest()[:8]

    def create_flow(self, flow_hash):
        """创建新流"""
        if self.flow_count >= self.max_flows:
            return None

        self.flows[flow_hash] = {
            'queue': deque(),
            'deficit': 0,
            'active': False,
            'first_above_time': None,  # CoDel: 首次超过目标延迟的时间
            'dropping': False,  # CoDel: 是否正在丢弃
            'drop_next': 0,  # CoDel: 下次丢弃时间
            'count': 0  # CoDel: 连续丢弃计数
        }
        self.flow_count += 1
        return self.flows[flow_hash]

    def codel_should_drop(self, flow, now):
        """CoDel AQM: 判断是否应该丢弃包"""
        queue = flow['queue']
        if len(queue) == 0:
            flow['first_above_time'] = None
            return False

        # 计算队列头包的驻留时间
        sojourn_time = now - queue[0].enqueue_time

        if sojourn_time < self.TARGET:
            # 延迟低于目标，重置状态
            flow['first_above_time'] = None
            return False

        if flow['first_above_time'] is None:
            # 首次超过目标延迟
            flow['first_above_time'] = now + self.INTERVAL
            return False

        if now >= flow['first_above_time']:
            # 超过目标延迟超过INTERVAL时间，开始/继续丢弃
            return True

        return False

    def codel_drop(self, flow, now):
        """CoDel: 执行丢弃逻辑"""
        queue = flow['queue']
        if len(queue) == 0:
            return None

        dropped = queue.popleft()

        if not flow['dropping']:
            # 开始新的丢弃周期
            flow['dropping'] = True
            flow['count'] = 1
            flow['drop_next'] = now + self.INTERVAL / 16
        else:
            # 继续丢弃周期
            flow['count'] += 1
            # 使用平方根律计算下次丢弃时间
            flow['drop_next'] = now + self.INTERVAL / (flow['count'] ** 0.5)

        return dropped

    def enqueue(self, packet):
        """入队操作"""
        flow_hash = self.get_flow_hash(packet)

        if flow_hash not in self.flows:
            flow = self.create_flow(flow_hash)
            if flow is None:
                return False  # 流表已满
        else:
            flow = self.flows[flow_hash]

        packet.enqueue_time = time.time()
        was_empty = len(flow['queue']) == 0
        flow['queue'].append(packet)

        if was_empty and not flow['active']:
            flow['active'] = True
            flow['deficit'] = 0
            self.active_list.append(flow_hash)

        return True

    def dequeue(self):
        """出队操作"""
        rounds = 0
        max_rounds = len(self.active_list) * 2

        while self.active_list and rounds < max_rounds:
            rounds += 1
            flow_hash = self.active_list[0]
            flow = self.flows[flow_hash]

            if len(flow['queue']) == 0:
                flow['active'] = False
                self.active_list.popleft()
                continue

            # CoDel AQM检查
            now = time.time()
            if self.codel_should_drop(flow, now):
                dropped = self.codel_drop(flow, now)
                if dropped:
                    continue  # 继续处理下一个

            # DRR调度
            flow['deficit'] += self.quantum

            while flow['queue']:
                packet = flow['queue'][0]
                if packet.size <= flow['deficit']:
                    flow['deficit'] -= packet.size
                    flow['queue'].popleft()

                    if flow['queue']:
                        self.active_list.append(self.active_list.popleft())
                    else:
                        flow['active'] = False
                        flow['deficit'] = 0
                        self.active_list.popleft()

                    return packet
                else:
                    break

            # 配额不足，移到队列末尾
            flow['deficit'] = 0
            self.active_list.append(self.active_list.popleft())

        return None

    def schedule(self):
        """调度一个包"""
        return self.dequeue()
```

#### 2.4.3 复杂度分析

| **指标** | **复杂度** | **说明** |
|---------|-----------|---------|
| **时间复杂度（入队）** | $O(1)$ amortized | 哈希表查找+队列追加，均摊O(1) |
| **时间复杂度（出队）** | $O(\log m)$ | m为活跃流数，需维护活跃流列表 |
| **CoDel处理开销** | $O(1)$ | 每次出队检查队列头包 |
| **空间复杂度** | $O(m + n)$ | m为最大流数，n为总包数 |
| **每流状态** | $O(1)$ | deficit + CoDel状态（约32字节/流） |
| **哈希表开销** | $O(m)$ | 流哈希表存储 |

#### 2.4.4 特点

- **流公平性**：流级别公平
- **延迟控制**：CoDel控制队列延迟
- **适用场景**：现代Linux网络栈（默认）

**时间复杂度**：$O(\log n)$

## 3 流量整形

### 3.1 令牌桶算法

**令牌桶（Token Bucket）**：流量整形算法

**核心机制**：

```text
令牌桶：容量C，速率r
  ↓
每1/r秒产生一个令牌
  ↓
包发送需要消耗令牌
  ↓
无令牌时包等待或丢弃
```

**数学模型**：

$$
\text{令牌数}(t) = \min(C, \text{令牌数}(t_0) + r \cdot (t - t_0))
$$

**特点**：

- **突发容忍**：允许短时突发
- **速率限制**：长期平均速率限制
- **适用场景**：流量整形、限速

### 3.2 漏桶算法

**核心机制**：

```text
漏桶：容量C，漏出速率r
  ↓
包到达时加入桶
  ↓
桶满时包丢弃
  ↓
包以速率r漏出
```

**特点**：

- **平滑输出**：输出速率恒定
- **无突发**：不允许突发
- **适用场景**：严格速率控制

## 4 形式化模型

### 4.1 网络包调度问题形式化定义

#### 4.1.1 问题定义

网络包调度问题可以形式化定义为一个五元组：

$$
\text{包调度问题} = (L, P, B, C, O)
$$

其中：

- $L = \{l_1, l_2, \ldots, l_n\}$：链路集合，每条链路 $l_i$ 有带宽容量 $C_i$
- $P = \{p_1, p_2, \ldots, p_m\}$：数据包队列
  - $p_i = (flow_i, size_i, arrival_i, deadline_i)$
  - $flow_i \in F$：包所属的流标识
  - $size_i \in \mathbb{N}^+$：包大小（字节）
  - $arrival_i \in \mathbb{R}^+$：到达时间戳
  - $deadline_i \in \mathbb{R}^+ \cup \{\infty\}$：截止时间（可选）
- $B$：带宽约束集合
  - $B_{total}$：总可用带宽
  - $B_{min}, B_{max}$：每流最小/最大带宽保证
- $C$：QoS约束集合
  - 延迟约束：$d_i \leq D_i$，其中 $d_i$ 为实际延迟，$D_i$ 为最大允许延迟
  - 抖动约束：$j_i \leq J_i$，其中 $j_i = \max|d_i - \bar{d}|$
  - 丢包率约束：$loss_i \leq L_i$
  - 带宽保证约束：$bw_i \geq R_i$
- $O$：优化目标函数
  - 公平性目标：$\min \max_i |bw_i - bw_{fair}|$
  - 延迟目标：$\min \sum_i d_i$
  - 吞吐量目标：$\max \sum_i throughput_i$

#### 4.1.2 调度策略定义

**定义 4.1（调度策略）**：调度策略是一个函数 $\pi: \mathcal{H} \times \mathcal{S} \rightarrow A$，其中：

- $\mathcal{H}$：历史信息空间（过去到达的包、发送的包、队列状态）
- $\mathcal{S}$：当前系统状态空间（队列长度、流状态、时间等）
- $A$：动作空间（选择发送哪个包）

**定义 4.2（可行调度）**：一个调度序列 $\{a_t\}_{t=1}^T$ 是可行的，当且仅当：

1. 因果性：$a_t$ 只能选择在时间 $t$ 之前已到达的包
2. 容量约束：$\sum_{i} size(a_t^i) \leq C \cdot \Delta t$，其中 $C$ 为链路容量
3. QoS约束：所有QoS约束 $C$ 被满足

#### 4.1.3 最优调度问题

给定上述定义，最优网络包调度问题为：

$$
\begin{aligned}
\min_{\pi} \quad & \sum_{i} \alpha_i \cdot \text{Delay}_i + \beta_i \cdot \text{Unfairness}_i + \gamma_i \cdot \text{Loss}_i \\
\text{s.t.} \quad & \sum_{i} bw_i \leq B_{total} \\
& d_i \leq D_i, \quad \forall i \\
& loss_i \leq L_i, \quad \forall i \\
& bw_i \geq R_i, \quad \forall i \in \text{guaranteed flows}
\end{aligned}
$$

其中 $\alpha_i, \beta_i, \gamma_i$ 为权重系数。

### 4.2 公平性形式化定义

#### 4.2.1 Max-Min Fairness（最大-最小公平性）

**定义 4.3（Max-Min Fairness）**：一个带宽分配 $\{bw_i\}_{i=1}^n$ 满足Max-Min Fairness，当且仅当：

1. **可行性**：$\sum_{i=1}^n bw_i \leq B_{total}$ 且 $bw_i \geq 0$
2. **最优性**：对于任意其他可行分配 $\{bw'_i\}$，如果存在流 $j$ 使得 $bw'_j > bw_j$，则必然存在流 $k$ 使得 $bw'_k < bw_k \leq bw_j$

**直观解释**：Max-Min Fairness 试图最大化最小流的带宽，然后在此基础上最大化次小流的带宽，依此类推。

**等价定义（基于瓶颈链路）**：

一个分配是Max-Min Fair的，当且仅当每个流 $i$ 要么：

- 达到其最大需求 $R_i^{max}$，或者
- 在其路径上某条链路 $l$ 达到饱和，且 $i$ 在 $l$ 上的带宽不低于 $l$ 上任何其他流的带宽

**Max-Min Fairness 计算公式**：

对于 $n$ 个流共享带宽 $B$，设各流需求为 $r_1 \leq r_2 \leq \cdots \leq r_n$：

$$
bw_i = \begin{cases}
r_i & \text{if } r_i \leq \frac{B - \sum_{j<i} r_j}{n - i + 1} \\
\frac{B - \sum_{j<i} r_j}{n - i + 1} & \text{otherwise}
\end{cases}
$$

#### 4.2.2 Proportional Fairness（比例公平性）

**定义 4.4（Proportional Fairness）**：一个带宽分配 $\{bw_i\}_{i=1}^n$ 满足Proportional Fairness，当且仅当对于任意其他可行分配 $\{bw'_i\}$：

$$
\sum_{i=1}^n \frac{bw'_i - bw_i}{bw_i} \leq 0
$$

**直观解释**：Proportional Fairness 在最大化总带宽和对数效用之间取得平衡，即：

$$
\max \sum_{i=1}^n \log(bw_i)
$$

**与Max-Min Fairness的区别**：

| **特性** | **Max-Min Fairness** | **Proportional Fairness** |
|---------|---------------------|--------------------------|
| **优化目标** | 最大化最小带宽 | 最大化对数效用和 |
| **对大流的处理** | 可能严格限制 | 相对宽松 |
| **对小流的保护** | 更强 | 较弱 |
| **数学性质** | 离散优化 | 连续凸优化 |
| **实现复杂度** | 较高（需迭代水填充） | 较低（可用梯度法） |

**Proportional Fairness 的求解**：

通过求解以下凸优化问题：

$$
\begin{aligned}
\max_{\{bw_i\}} \quad & \sum_{i=1}^n \log(bw_i) \\
\text{s.t.} \quad & \sum_{i \in S_l} bw_i \leq C_l, \quad \forall l \in L \\
& bw_i \geq 0, \quad \forall i
\end{aligned}
$$

其中 $S_l$ 为使用链路 $l$ 的流集合，$C_l$ 为链路 $l$ 的容量。

#### 4.2.3 Jain's Fairness Index（公平性指标）

**定义 4.5（Jain's Fairness Index）**：对于 $n$ 个流的带宽分配 $\{bw_i\}_{i=1}^n$，Jain's Fairness Index 定义为：

$$
J = \frac{\left(\sum_{i=1}^n bw_i\right)^2}{n \cdot \sum_{i=1}^n bw_i^2}
$$

**性质**：

- $J \in [\frac{1}{n}, 1]$
- $J = 1$：完全公平（所有流获得相同带宽）
- $J = \frac{1}{n}$：极度不公平（一个流获得全部带宽）

**计算示例**：

| **场景** | 带宽分配 | Jain's Index |
|---------|---------|-------------|
| 完全公平 | [100, 100, 100] | $J = \frac{300^2}{3 \times 30000} = 1.0$ |
| 较公平 | [80, 100, 120] | $J = \frac{300^2}{3 \times 30800} \approx 0.97$ |
| 较不公平 | [50, 100, 150] | $J = \frac{300^2}{3 \times 35000} \approx 0.86$ |
| 极度不公平 | [0, 0, 300] | $J = \frac{300^2}{3 \times 90000} \approx 0.33$ |

### 4.3 延迟保证形式化

#### 4.3.1 延迟组成分析

网络包的总延迟由以下部分组成：

$$
D_{total} = D_{queue} + D_{trans} + D_{prop} + D_{proc}
$$

其中：

- $D_{queue}$：队列延迟（调度器控制）
- $D_{trans} = \frac{L}{R}$：传输延迟（包长 $L$ / 链路速率 $R$）
- $D_{prop} = \frac{d}{c}$：传播延迟（距离 $d$ / 光速 $c$）
- $D_{proc}$：处理延迟（协议栈处理、路由查表等）

**调度器可控部分**：主要控制 $D_{queue}$

#### 4.3.2 延迟保证形式化定义

**定义 4.6（延迟保证）**：调度器提供 $(\bar{d}, \epsilon)$-延迟保证，如果：

$$
P(D_{queue} > \bar{d}) \leq \epsilon
$$

即队列延迟超过阈值 $\bar{d}$ 的概率不超过 $\epsilon$。

**定义 4.7（确定性延迟保证）**：调度器提供确定性延迟保证 $\bar{d}$，如果：

$$
D_{queue} \leq \bar{d}, \quad \text{for all packets}
$$

#### 4.3.3 WFQ延迟保证定理

**定理 4.1（WFQ延迟上界）**：对于使用WFQ调度的流 $i$，设其权重为 $w_i$，包 $p$ 的大小为 $L$，系统总带宽为 $B$，则包 $p$ 的队列延迟上界为：

$$
D_{queue} \leq \frac{L_{max}}{B} + \frac{L}{w_i \cdot B / W_{total}}
$$

其中：

- $L_{max}$：系统中最大包大小
- $W_{total} = \sum_j w_j$：总权重

**证明概要**：

1. 最坏情况下，包 $p$ 到达时刚好错过了调度时机
2. 需要等待当前正在传输的最大包完成：$\frac{L_{max}}{B}$
3. 然后在自己的配额内完成传输：$\frac{L}{w_i \cdot B / W_{total}}$

#### 4.3.4 CoDel延迟控制机制

**CoDel目标**：保持队列延迟在目标值 $target$（默认5ms）附近。

**控制律**：

设 $sojourn(t)$ 为时间 $t$ 时队列头包的驻留时间：

$$
\text{drop}(t) = \begin{cases}
\text{False} & \text{if } sojourn(t) < target \\
\text{False} & \text{if } sojourn(t) \geq target \text{ but duration} < interval \\
\text{True} & \text{if } sojourn(t) \geq target \text{ and duration} \geq interval
\end{cases}
$$

其中 $duration$ 为连续超过 $target$ 的时间。

**CoDel的数学模型**：

CoDel可以看作是一个带滞后的比例控制器，控制律为：

$$
\text{drop\_probability} = f(\int_{t-interval}^t \mathbb{1}_{[sojourn > target]}(s) ds)
$$

其中 $f(\cdot)$ 是一个非线性函数，实现平方根律的丢弃间隔。

### 4.4 调度算法复杂度对比

### 4.2 调度算法复杂度

| **算法** | **时间复杂度** | **公平性** | **延迟保证** | **适用场景** |
|---------|--------------|-----------|------------|------------|
| **FIFO** | $O(1)$ | 无 | 无 | 简单场景 |
| **WFQ** | $O(\log n)$ | 加权公平 | 有界延迟 | QoS网络 |
| **DRR** | $O(1)$ | 轮询公平 | 无 | 高速网络 |
| **FQ-CoDel** | $O(\log n)$ | 流公平 | 有界延迟 | 现代网络栈 |

### 4.3 定理：WFQ公平性保证

**定理15.1（WFQ公平性）**：

对于权重为$w_i$的流$i$，WFQ保证其获得的带宽满足：

$$
bw_i \geq \frac{w_i}{\sum_j w_j} \cdot B_{total}
$$

**证明**：由虚拟完成时间的定义，权重大的流虚拟完成时间增长慢，获得更多调度机会。∎

#### 5.1.2 pfifo_fast实现

**源码位置**：`net/sched/sch_fifo.c`

**核心实现**：

```c
// 简化版核心逻辑
struct pfifo_fast_priv {
    struct qdisc_skb_head q[3];  // 3个优先级队列
};

static int pfifo_fast_enqueue(struct sk_buff *skb, struct Qdisc *qdisc) {
    // 根据skb优先级选择队列（0-2）
    int band = skb_get_tc_band(skb);
    return __qdisc_enqueue_tail(skb, qdisc, &priv->q[band]);
}

static struct sk_buff *pfifo_fast_dequeue(struct Qdisc *qdisc) {
    // 从高优先级到低优先级轮询
    for (int i = 0; i < 3; i++) {
        skb = __qdisc_dequeue_head(&priv->q[i]);
        if (skb) return skb;
    }
    return NULL;
}
```

**性能特点**：

- 3个严格优先级队列（Band 0-2）
- $O(1)$ 入队/出队
- 无公平性保证，高优先级可能饿死低优先级

**五级调度层次**：

| **层级** | **名称** | **功能** | **调度算法** |
|---------|---------|---------|-------------|
| Level 1 | Port | 物理端口级别 | WFQ |
| Level 2 | Subport | 逻辑子端口（租户/业务） | WFQ |
| Level 3 | Pipe | 流管道（用户/连接） | WFQ |
| Level 4 | Traffic Class | 流量类别（QoS等级） | Strict Priority |
| Level 5 | Queue | 队列（同类别内的流） | WFQ |

**核心数据结构**：

```c

    uint32_t n_pipes_per_subport_log2;
    uint32_t n_subport_profiles;  // 子端口配置文件数

    // 运行时状态
    struct rte_sched_subport *subports;
    struct rte_sched_pipe *pipes;
    struct rte_sched_queue *queues;
};

struct rte_sched_subport {
    uint64_t tb_time;           // 令牌桶时间戳
```

## 7 跨领域洞察

### 7.1 网络调度与CPU调度的类比

| **维度** | **CPU调度** | **网络包调度** |
|---------|------------|--------------|
| **调度单元** | 进程/线程 | 数据包/流 |
| **资源** | CPU时间片 | 带宽 |
| **公平性** | CFS红黑树 | WFQ虚拟时间 |
| **优先级** | 进程优先级 | 流权重 |
| **延迟** | 响应时间 | 包延迟 |

**关键洞察**：网络包调度可以视为**带宽资源的进程调度**，流类似于进程。

### 7.2 物理约束：光速限制

**光速限制**：$c = 3 \times 10^8$ m/s

**跨地域延迟**：

- 本地网络：$< 1$ms（< 300km）
- 跨城市：$10-50$ms（1000-15000km）
- 跨洲：$100-300$ms（> 10000km）

**关键洞察**：**物理距离决定延迟下界**，调度算法无法突破光速限制。

### 7.3 队列延迟的级联放大

**单跳延迟**：$d_1 = 1$ms
**多跳延迟**：$d_n = n \cdot d_1 = n$ms（无拥塞）

**拥塞时延迟**：
$$
d_n = n \cdot d_1 + \sum_{i=1}^{n} \frac{Q_i}{B_i}
$$

其中$Q_i$是队列长度，$B_i$是带宽。

**关键洞察**：**队列延迟在多跳网络中累积**，需要端到端拥塞控制。
