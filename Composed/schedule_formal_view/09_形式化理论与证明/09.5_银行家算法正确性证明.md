## 1. 引言

### 1.1 银行家算法概述

银行家算法由Edsger Dijkstra于1965年提出，是一种经典的死锁避免算法。它通过模拟资源分配来确保系统始终处于**安全状态**，从而避免死锁的发生。

### 1.2 核心思想

银行家算法类比银行贷款：

- 银行家（操作系统）拥有固定资金（资源）
- 客户（进程）申请贷款（资源）
- 银行家只在确定能收回所有贷款时才批准申请

## 2. 形式化定义

### 2.1 系统状态定义

**定义9.5.1（资源分配系统）**：

资源分配系统是一个六元组 $\mathcal{S} = (P, R, E, A, C, N)$，其中：

- $P = \{P_1, P_2, \ldots, P_n\}$：进程集合，$|P| = n$
- $R = \{R_1, R_2, \ldots, R_m\}$：资源类型集合，$|R| = m$
- $E = (E_1, E_2, \ldots, E_m)$：资源总量向量，$E_j$ 表示 $R_j$ 的实例总数
- $A = (A_1, A_2, \ldots, A_m)$：可用资源向量，$A_j$ 表示 $R_j$ 的可用实例数
- $C = [c_{ij}]_{n \times m}$：当前分配矩阵，$c_{ij}$ 表示进程 $P_i$ 持有的 $R_j$ 实例数
- $N = [n_{ij}]_{n \times m}$：需求矩阵，$n_{ij}$ 表示进程 $P_i$ 还需要的 $R_j$ 实例数

**定义9.5.2（最大需求矩阵）**：

最大需求矩阵 $M = [m_{ij}]_{n \times m}$，其中：

$$
m_{ij} = c_{ij} + n_{ij}
$$

**系统状态约束**：

$$
\forall j. \sum_{i=1}^{n} c_{ij} + A_j = E_j
$$

### 2.2 安全状态定义

**定义9.5.3（安全序列）**：

进程序列 $\langle P_{s_1}, P_{s_2}, \ldots, P_{s_n} \rangle$ 是安全序列，当且仅当对每个 $P_{s_i}$：

$$
Need_{s_i} \leq Available_i
$$

其中：

$$
Available_i = A + \sum_{k=1}^{i-1} C_{s_k}
$$

**定义9.5.4（安全状态）**：

系统状态是安全的，当且仅当存在至少一个安全序列。

**定义9.5.5（不安全状态）**：

系统状态是不安全的，当且仅当不存在任何安全序列。

## 3. 银行家算法

### 3.1 安全状态检测算法

```python
def is_safe_state(n, m, Available, Allocation, Need):
    """
    检测系统是否处于安全状态

    参数:
        n: 进程数
        m: 资源类型数
        Available: 可用资源向量
        Allocation: 当前分配矩阵
        Need: 需求矩阵

    返回:
        (is_safe, safe_sequence)
    """
    Work = Available.copy()
    Finish = [False] * n
    safe_sequence = []

    while len(safe_sequence) < n:
        found = False

        for i in range(n):
            if not Finish[i] and Need[i] <= Work:
                # 进程i可以完成
                Work = Work + Allocation[i]
                Finish[i] = True
                safe_sequence.append(i)
                found = True
                break

        if not found:
            # 没有进程可以完成，系统不安全
            return False, []

    return True, safe_sequence
```

**时间复杂度**：$O(n^2 \cdot m)$

### 3.2 资源请求算法

```python
def request_resources(P_i, Request, Available, Allocation, Need):
    """
    处理进程P_i的资源请求

    返回:
        True: 请求被批准
        False: 请求被拒绝
    """
    # 步骤1: 检查Request <= Need
    if Request > Need[i]:
        raise Error("请求超过最大需求")

    # 步骤2: 检查Request <= Available
    if Request > Available:
        return False  # 资源不足，等待

    # 步骤3: 试分配
    Available_temp = Available - Request
    Allocation_temp = Allocation.copy()
    Allocation_temp[i] = Allocation[i] + Request
    Need_temp = Need.copy()
    Need_temp[i] = Need[i] - Request

    # 步骤4: 检查安全性
    is_safe, _ = is_safe_state(n, m, Available_temp,
                                Allocation_temp, Need_temp)

    if is_safe:
        # 正式分配
        Available = Available_temp
        Allocation = Allocation_temp
        Need = Need_temp
        return True
    else:
        return False  # 不安全，拒绝请求
```

## 4. 正确性证明

### 4.1 安全性定理

**定理9.5.1（银行家算法安全性）**：

如果银行家算法批准资源请求，系统仍处于安全状态。

**证明**：

**前提**：请求批准当且仅当试分配后系统处于安全状态。

**归纳基础**：

初始状态假设为安全状态（由系统初始化保证）。

**归纳步骤**：

假设当前状态安全，存在安全序列 $\langle P_{s_1}, \ldots, P_{s_n} \rangle$。

进程 $P_i$ 发出请求 $Request_i$，算法检查：

1. **可行性检查**：$Request_i \leq Available$（资源足够）
2. **合法性检查**：$Request_i \leq Need_i$（不超过最大需求）
3. **安全性检查**：试分配后存在安全序列

设试分配后状态为 $S'$，算法验证 $S'$ 的安全序列存在。

根据安全序列定义，$S'$ 中每个进程都能完成，释放资源给后续进程。

因此 $S'$ 是安全状态。$\square$

### 4.2 无死锁定理

**定理9.5.2（无死锁保证）**：

银行家算法保证系统永远不会进入死锁状态。

**证明**：

**引理**：安全状态 $\implies$ 无死锁

假设系统处于安全状态，存在安全序列 $\langle P_{s_1}, \ldots, P_{s_n} \rangle$。

对于安全序列中的第一个进程 $P_{s_1}$：

$$
Need_{s_1} \leq Available = A
$$

因此 $P_{s_1}$ 可以获得所需资源完成执行，释放其占用的资源 $C_{s_1}$。

完成后，可用资源变为 $A + C_{s_1}$。

对于 $P_{s_2}$：

$$
Need_{s_2} \leq A + C_{s_1}
$$

同理，$P_{s_2}$ 可以完成并释放资源。

依此类推，安全序列中所有进程都可以按顺序完成。

因此，安全状态下不存在循环等待，无死锁。$\square$

**主定理证明**：

由定理9.5.1，银行家算法保证系统始终处于安全状态。

由引理，安全状态意味着无死锁。

因此，银行家算法保证无死锁。$\square$

### 4.3 活性定理

**定理9.5.3（有限等待/活性）**：

在银行家算法中，任何资源请求最终都会被满足（假设进程最终释放资源）。

**证明**：

考虑进程 $P_i$ 请求资源 $Request_i$ 被拒绝的情况：

请求被拒绝仅当试分配后系统不安全，即：

$$
\exists j. Available_j - Request_{ij} < \min_{k} Need_{kj}
$$

这意味着批准请求可能导致某些进程无法完成。

**情况1**：$P_i$ 已持有足够资源完成

$P_i$ 可以继续执行，完成后释放资源，$Available$ 增加。

**情况2**：其他进程可以完成

安全序列中的进程可以依次完成，释放资源。

由于进程最终释放资源，$Available$ 会不断增加。

最终，$Available \geq Request_i$，且批准请求后系统安全，请求被满足。

因此，任何请求最终都会被满足。$\square$

### 4.4 最优性定理

**定理9.5.4（最大资源利用率）**：

在满足安全性的前提下，银行家算法允许最大可能的资源分配。

**形式化**：

设 $Safe$ 是所有安全状态的集合。对于任何安全状态 $S \in Safe$，银行家算法允许的资源分配不会比任何其他安全算法更保守。

**证明**：

银行家算法的安全检测是**充分必要**的：

- **充分性**：通过安全序列证明（定理9.5.1）
- **必要性**：存在安全序列 $\implies$ 状态安全

任何其他安全算法必须拒绝至少与银行家算法相同的请求，否则将导致不安全状态。

因此，银行家算法在安全算法中是最优的（最不保守）。$\square$

|---------|--------------|---------|
| 安全状态检测 | $O(n^2 \cdot m)$ | n个进程，m种资源 |
| 资源请求处理 | $O(n^2 \cdot m)$ | 包含安全检测 |

$$
Space = O(n \cdot m) \text{（存储分配矩阵和需求矩阵）}
$$

## 6. 形式化验证（TLA+风格）

### 6.1 状态不变式

**安全不变式**：

$$
\Box \left( \exists \text{ safe sequence } \iff \text{system is safe} \right)
$$

**资源守恒不变式**：

$$
\Box \left( \sum_{i=1}^{n} C_{ij} + A_j = E_j, \quad \forall j \right)
$$

### 6.2 活性性质

$$
\Diamond \left( Request_i \text{ granted} \right), \quad \forall i
$$

（假设公平性和进程最终释放资源）

## 7. 批判性分析

### 7.1 银行家算法的局限

1. **资源利用率**：保守策略导致资源利用率降低
2. **进程数量**：假设进程数量固定，不支持动态创建
3. **最大需求**：需要预先知道进程的最大资源需求
4. **算法开销**：$O(n^2 \cdot m)$ 的复杂度在高负载时成为瓶颈

### 7.2 现代替代方案

| **方案** | **优势** | **劣势** |
|---------|---------|---------|
| **鸵鸟算法** | 无开销 | 可能死锁 |
| **死锁检测+恢复** | 更高利用率 | 恢复代价高 |
| **资源有序分配** | 简单 | 限制编程模型 |
| **事务内存** | 自动回滚 | 硬件支持要求高 |

## 8. 扩展：多维资源形式化

### 8.1 向量比较定义

对于资源向量 $X = (x_1, \ldots, x_m)$ 和 $Y = (y_1, \ldots, y_m)$：

$$
X \leq Y \iff \forall j. x_j \leq y_j
$$

$$
X < Y \iff X \leq Y \land \exists j. x_j < y_j
$$

### 8.2 多维安全条件

系统安全当且仅当：

$$
\exists \pi. \forall i. Need_{\pi(i)} \leq A + \sum_{k=1}^{i-1} C_{\pi(k)}
$$

其中 $\pi$ 是 $\{1, \ldots, n\}$ 的一个排列。
